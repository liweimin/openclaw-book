# OpenClaw 工作区根文件：AGENTS、SOUL、USER、IDENTITY、TOOLS、HEARTBEAT、BOOTSTRAP、MEMORY

副标题：把最关键、最能调教 agent 的这组文件一次讲清楚

## 1. 为什么这一章很重要

OpenClaw 真正好不好用，很多时候不取决于你临时在聊天里说了什么，而取决于工作区根目录这组文件写得是否清楚。

这组文件里，最关键的是：

- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `IDENTITY.md`
- `TOOLS.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md`
- `MEMORY.md`

如果你把它们分工搞清楚，后面很多问题都会顺：

- 为什么这个 agent 老是答非所问
- 为什么它太像客服
- 为什么它不记得你的长期偏好
- 为什么它总是不先读对文件
- 为什么 heartbeat / cron 行为不稳定

这一章就是专门讲“这些文件各自干什么、该放什么、怎么改”。

---

## 2. 先记住一个总原则：它们不是一类文件

这组文件虽然都在工作区根目录，但职责完全不同。

最稳的理解是：

- `AGENTS.md`
  - 做事规则
- `SOUL.md`
  - 语气、边界、人格风格
- `USER.md`
  - 用户画像
- `IDENTITY.md`
  - agent 自己的名字、气质、头像等身份信息
- `TOOLS.md`
  - 你这台机器 / 这套环境的本地说明
- `HEARTBEAT.md`
  - heartbeat 轮询时看的短清单
- `BOOTSTRAP.md`
  - 只在 brand-new workspace 第一次启动时用的一次性引导
- `MEMORY.md`
  - 长期、稳定、值得持续记住的信息

一句话：

**规则写进规则文件，身份写进身份文件，长期记忆写进记忆文件，不要什么都往 `AGENTS.md` 里塞。**

---

## 3. 这些文件什么时候会进入上下文

官方文档和源码都非常明确：

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md`
- `MEMORY.md` / `memory.md`

都属于工作区 bootstrap 文件，会进入 prompt 注入链路。

这意味着两件事：

1. 它们真的很重要
2. 它们真的会消耗 token

所以这组文件有一个共同要求：

- 要短
- 要稳
- 要长期有效

而像 `memory/YYYY-MM-DD.md` 这种 daily 文件，**不会自动注入**。  
它们通常是 agent 在需要时主动读取，或者通过 memory 工具按需访问。

另外还有一个非常重要的边界：

- 正常主会话：会注入这整组根文件
- sub-agent 会话：通常只注入 `AGENTS.md` 和 `TOOLS.md`

所以：

- `AGENTS.md` 和 `TOOLS.md` 更像“基础规则层”
- `SOUL.md`、`USER.md`、`IDENTITY.md` 更像“主会话人格层”

---

## 4. `AGENTS.md`：最重要，但也最容易被写坏

### 它是干什么的

`AGENTS.md` 的职责是：

- 规定这个 agent 怎么做事
- 规定优先级
- 规定先读什么、后做什么
- 规定哪些问题先读文件，哪些问题再搜索
- 规定哪些动作必须写回文件

它最像：

- 操作手册
- 工作守则
- 执行顺序说明

### 里面适合放什么

适合放：

- Session 启动顺序
- 文件读取顺序
- 当前状态问题 vs 模糊历史问题的分流规则
- 写 daily / 更新 current-week 的规则
- 安全边界
- 输出风格的硬规则

### 不适合放什么

不适合放：

- 很多重复背景
- 大段人生哲学
- 具体项目资料
- 变化很频繁的今天/本周任务
- 一堆“如果有空可以顺便看看”的泛建议

### 默认模板大意是什么意思

官方默认 `AGENTS.default.md` 和工作区模板，核心意思大概是：

- 每次 session 开始先读关键文件
- 把记忆写进文件，而不是只靠聊天
- 对群聊和对外动作要谨慎
- 工具规则和环境说明分开管理

这套默认模板更像一个“通用个人助手起点”，不是你最终版。

### 自己优化时怎么改

最稳的做法是：

1. 保留启动顺序
2. 保留关键安全边界
3. 保留文件分工
4. 删掉与你当前场景无关的泛用说明
5. 把规则改成直接执行式

这里最重要的一条经验是：

**把规则写成“你现在要怎么做”，不要写成“这个文件负责什么”。**

坏写法：

- `AGENTS.md 负责维护 daily 文件`

更好的写法：

- `下午 5 点后，先问 3 个问题；用户回复后，把结果写回今天的 daily 文件`

### 哪些建议保留

- Session 启动顺序
- 先读文件还是先搜索
- daily / weekly 更新规则
- 安全边界

### 哪些可以按自己情况删减

- 群聊规则
- reaction 规则
- 与当前场景无关的 heartbeat 巡检清单
- 不再使用的历史实验说明

---

## 5. `SOUL.md`：它不是“规则文件”，而是人格与边界文件

### 它是干什么的

`SOUL.md` 更接近：

- 说话风格
- 人格边界
- 语气
- 整体气质

官方模板的核心意思大概是：

- 别像客服
- 别满嘴套话
- 要真帮忙，不要表演式帮助
- 有边界感
- 尊重隐私

### 里面适合放什么

适合放：

- 你希望它更像什么样的人
- 回答是更直接还是更温和
- 对外动作要不要更谨慎
- 是否允许表达偏好和判断

### 不适合放什么

不适合放：

- 具体业务流程
- 每天要做的任务
- 路径、端口、命令
- 长期项目知识

这些更适合去：

- `AGENTS.md`
- `TOOLS.md`
- `MEMORY.md`

### 怎么优化最有效

如果你发现它：

- 太像客服
- 太爱夸人
- 太虚
- 太像套模板

那优先改 `SOUL.md`。

如果你发现它：

- 不先读文件
- 不按顺序做事
- 不更新 daily

那优先改 `AGENTS.md`，不是改 `SOUL.md`。

---

## 6. `USER.md`：写“你是谁”，不是写项目文档

### 它是干什么的

`USER.md` 用来描述这个 agent 正在长期服务的人。

你当前工作区里的默认模板很典型，核心字段是：

- Name
- What to call them
- Pronouns
- Timezone
- Notes
- Context

### 里面适合放什么

适合放：

- 你希望它怎么称呼你
- 时区
- 你现在主要在做什么类型的事
- 你典型的工作方式
- 你讨厌什么样的回答方式

### 不适合放什么

不适合放：

- 很细的每日任务
- 私密程度过高、又没必要长期注入的内容
- 过度详细的人生履历

### 最稳的写法

写成“长期稳定、会影响协作方式”的信息。

例如：

- 默认用中文
- 更喜欢直接进入结论
- 更看重可执行建议
- 主要通过飞书和本地工作区使用 OpenClaw

### 哪些内容后面可以慢慢补

- 你的项目背景
- 你的工作习惯
- 你对这类助手的期待

它不要求一次写满，可以边用边补。

---

## 7. `IDENTITY.md`：让 agent 有身份，但不要过度戏剧化

### 它是干什么的

`IDENTITY.md` 用来描述 agent 自己的身份信息。

从官方 CLI 文档和源码测试可以确定，常见字段包括：

- `Name`
- `Creature`
- `Vibe`
- `Emoji`
- `Avatar`

你当前工作区里的默认模板也是按这几个字段来的。

### 这些字段分别有什么用

- `Name`
  - 这个 agent 叫什么
- `Creature`
  - 更像什么存在：机器人、AI、伙伴等
- `Vibe`
  - 给人什么感觉
- `Emoji`
  - 签名 emoji
- `Avatar`
  - 头像路径或 URL

### 里面适合放什么

适合放：

- 一个稳定名字
- 一个不让人出戏的 vibe
- 简短、好记的身份设定

### 不适合放什么

不适合放：

- 冗长人设小作文
- 复杂世界观
- 具体执行规则
- 业务知识

### 小白建议

第一版别太花哨。  
够用就行。

例如：

- Name: Weekly Assistant
- Creature: AI assistant
- Vibe: calm and practical
- Emoji: 🗂️

这样就足够了。

---

## 8. `TOOLS.md`：写环境说明，不是开工具权限

### 它是干什么的

这是最容易被误解的文件之一。

`TOOLS.md` 的作用不是：

- 开权限
- 关权限
- 决定工具是否存在

官方文档说得很明确：  
它只是“用户自己维护的本地工具说明”。

### 里面适合放什么

适合放：

- 本地路径习惯
- SSH 别名
- 浏览器使用约定
- 你机器上的命令习惯
- 某些本地工具如何使用

例如：

- 这个项目常用目录在哪
- 哪个命令启动哪个服务
- 哪个浏览器 profile 用于测试

### 不适合放什么

不适合放：

- “请启用 read / write / edit”
- “请允许 exec”

这些是配置文件和工具策略的事，不是 `TOOLS.md` 的事。

### 什么时候最值得改

如果你发现 agent：

- 老是找错路径
- 老是忘记本地习惯
- 老是不会用你机器上的某个常用工具

就优先补 `TOOLS.md`。

---

## 9. `HEARTBEAT.md`：小、短、明确

### 它是干什么的

`HEARTBEAT.md` 是 heartbeat 轮询时看的短清单。

官方文档很强调两点：

1. 它是可选的
2. 要尽量短

### 为什么要短

因为 heartbeat 会反复跑。  
它每次都把这个文件带进上下文时，token 会持续消耗。

所以 `HEARTBEAT.md` 最适合：

- 一小段检查清单
- 一两个提醒规则
- 非常明确的触发标准

### 不适合放什么

不适合放：

- 很长的操作说明
- 一整套工作流
- 大量背景资料

这些更适合：

- `AGENTS.md`
- cron 提示词

### 和 cron 的区别

- `cron`
  - 更适合精确时间点
- `HEARTBEAT.md`
  - 更适合“每隔一段时间检查一下有没有必要提醒”

对你当前这条主线来说：

- 第一阶段先用 cron
- 第二阶段再考虑 heartbeat

是更稳的顺序。

---

## 10. `BOOTSTRAP.md`：一次性，不是长期规则文件

### 它是干什么的

`BOOTSTRAP.md` 是 brand-new workspace 第一次启动时的一次性引导文件。

官方文档明确说：

- brand-new workspace 才会创建它
- bootstrap 完成后会删除
- 之后不应该再反复回来

### 它的作用

主要是第一次引导 agent：

- 明白自己是谁
- 收集一些初始身份信息
- 把这些信息写进 `IDENTITY.md`、`USER.md`、`SOUL.md`

### 你平时要不要改它

通常不用。

大多数人真正长期会改的是：

- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `TOOLS.md`
- `MEMORY.md`

`BOOTSTRAP.md` 更像第一次开机的向导，不是日常调教主战场。

---

## 11. `MEMORY.md`：长期记忆，不是杂物间

### 它是干什么的

`MEMORY.md` 用来放长期、稳定、值得持续注入的内容。

官方文档和 memory 章节都非常一致：

- 持久偏好
- 稳定决策
- 长期事实

更适合放进 `MEMORY.md`。

### 里面适合放什么

适合放：

- 长期偏好
- 长期目标
- 稳定约束
- 经常反复用到的协作规则

### 不适合放什么

不适合放：

- 今天的临时任务
- 高频变化的周计划
- 大量已完成事项
- 冗长聊天摘录

这些更适合：

- `memory/YYYY-MM-DD.md`
- `memory/weekly/current-week.md`

### 一个很重要的边界

`MEMORY.md` 会参与注入，所以它不是无限大的垃圾桶。

写太长会带来两个问题：

1. token 消耗变高
2. 真正重要的信息被淹没

所以最稳的原则是：

**只留长期、稳定、真的值得持续带入上下文的信息。**

## 12. 这些文件的标准初始化模板，到底在表达什么

把官方模板翻成人话，大概是下面这层意思。

### `AGENTS.md`

默认是在说：

- 每次 session 开始先读关键文件
- 记忆要写进文件
- 群聊和外发动作要谨慎

### `SOUL.md`

默认是在说：

- 别像客服
- 别演示式帮助
- 有边界，有风格

### `USER.md`

默认是在说：

- 先认识你在服务谁
- 学会怎么称呼对方
- 逐步形成更贴合的协作方式

### `IDENTITY.md`

默认是在说：

- 给这个 agent 一个可识别的身份
- 但先不要写得太复杂

### `TOOLS.md`

默认是在说：

- 把这台机器和这套环境特有的说明放这里
- 不要把环境细节硬塞进技能或人格文件

### `HEARTBEAT.md`

默认是在说：

- 如果要做 heartbeat，就给一个很短的清单

### `BOOTSTRAP.md`

默认是在说：

- 这是第一次启动用的，不是长期规则

### `MEMORY.md`

默认是在说：

- 长期有效的东西才值得写进来

## 13. 自己优化时，最容易踩的坑

### 坑 1：把所有东西都塞进 `AGENTS.md`

结果会是：

- 太长
- 太散
- 规则和背景混在一起

### 坑 2：把 `SOUL.md` 当规则文件用

结果会是：

- 它会说话有风格
- 但做事顺序还是乱

### 坑 3：把 `TOOLS.md` 当权限开关

结果会是：

- 你以为写了它就有工具
- 实际上工具可用性根本不由它决定

### 坑 4：把 `MEMORY.md` 写成流水账

结果会是：

- 既浪费 token
- 也让长期重点越来越模糊

### 坑 5：把 `HEARTBEAT.md` 写太大

结果会是：

- 每次 heartbeat 都在烧 token
- 而且效果还未必更好

### 坑 6：忘了 sub-agent 只看一部分文件

如果你后面开始用 sub-agent，要记住：

- 它通常只注入 `AGENTS.md` 和 `TOOLS.md`

所以别把关键执行规则只写在别的地方，然后指望 sub-agent 自动继承。

---

## 14. 我给小白的最稳建议

如果你现在刚开始调这套文件，我建议按这个顺序：

1. 先把 `AGENTS.md` 写清楚
2. 再把 `SOUL.md` 收成你喜欢的风格
3. 再补 `USER.md`
4. 再补 `TOOLS.md`
5. 再慢慢积累 `MEMORY.md`
6. 需要 heartbeat 时，再单独写 `HEARTBEAT.md`

`BOOTSTRAP.md` 不用当日常主战场。  
`IDENTITY.md` 则够用就行，不要过度设计。

---

## 15. 一句话总结

这组文件不是“都很重要，所以都写很多”。

更好的理解是：

- `AGENTS.md` 管怎么做事
- `SOUL.md` 管怎么说话
- `USER.md` 管你在服务谁
- `IDENTITY.md` 管你是谁
- `TOOLS.md` 管你这台机器有什么习惯
- `HEARTBEAT.md` 管 heartbeat 小清单
- `BOOTSTRAP.md` 管第一次启动
- `MEMORY.md` 管长期稳定记忆

真正好用的 OpenClaw，不是聊天里一次次重新教，而是把这些文件分工写对。
