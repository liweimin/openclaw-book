# OpenClaw 记忆与检索：从工作区文件到 QMD

副标题：把 `MEMORY.md`、`memory/*.md`、`sessions/*.jsonl`、`memory_search` 和 `QMD` 一次讲清楚

## 1. 为什么这一章很重要

很多人第一次用 OpenClaw，都会混淆下面这些东西：

- 聊天记录
- 工作区文件
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- `sessions/*.jsonl`
- `memory_search`
- `QMD`

一旦混在一起，就很容易出现这些疑问：

- 我明明 `/new` 了，它为什么还知道我今天干了什么？
- `memory_search` 明明返回空，为什么它回答得又大体是对的？
- daily 文件是不是系统自己会自动维护？
- `sessions` 已经记录聊天了，为什么还要 `memory/`？
- 内置后端和 QMD 到底差在哪？

这一章就是专门解决这些问题的。

---

## 2. 先分清 4 层：不要把“记忆”当成一件事

对 OpenClaw 来说，所谓“记忆”，至少要拆成 4 层来看。

### 第一层：当前会话层

这一层是：

- 当前 session 的聊天历史
- 当前轮次的工具调用结果
- compaction 之后留下的摘要

它的特点是：

- 适合继续当前对话
- 适合短期连续推进
- 不等于长期整理过的知识

### 第二层：自动注入的根文件

这一层是：

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md`
- `MEMORY.md`

这类文件会进入 prompt 注入链路，所以：

- 它们要短
- 要稳
- 要长期有效

### 第三层：工作区资料层

这一层主要是：

- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- `memory/weekly/current-week.md`
- `memory/weekly/archive/*.md`

这是你主动维护的“长期资料层”。

### 第四层：检索层

这一层主要是：

- `memory_search`
- `memory_get`
- 内置 memory backend
- `QMD`

它解决的是：

- 文件多了以后怎么找
- 跨会话怎么回忆
- 搜到后怎么把需要的内容再拉回来

## 3. 哪些内容会自动进入上下文，哪些不会

这个问题一定要讲准。

### 会自动进入上下文的

默认更接近这些：

- 当前 session 的历史
- 自动注入的根文件

也就是说，像 `AGENTS.md`、`SOUL.md`、`USER.md`、`MEMORY.md` 这类文件，本来就在系统提示词链路里。

### 不会默认自动注入的

这类文件不要想当然：

- `memory/2026-03-08.md`
- `memory/weekly/current-week.md`
- 普通工作区里的其他 `.md`

它们不是天然每次都被自动塞进 prompt。

### 那为什么 `/new` 后它还是可能知道今天发生了什么

因为 `/new` 清掉的是旧聊天历史，不是把 agent 变成一张白纸。

`/new` 后至少还会有两层东西：

1. 自动注入的根文件
2. Session Startup sequence

如果你的 `AGENTS.md` 明确写了：

- 先读今天的 daily
- 再读昨天的 daily
- 再读 `current-week.md`

那 agent 就会在新会话启动时主动把这些文件读进来。

这也是为什么：

- 你明明 `/new` 了
- 它却仍然能总结“今天做过什么”

根本原因不是它“神奇地一直记得”，而是它**新会话一开始又把相关文件重新读了一遍**。

## 4. `MEMORY.md`、daily、current-week、sessions 到底各管什么

### `MEMORY.md`

放长期稳定信息：

- 长期偏好
- 固定规则
- 长期目标
- 持续有效的边界

不适合：

- 高频变化的周计划
- 大量已完成事项
- 冗长流水账

### `memory/YYYY-MM-DD.md`

放当天推进记录：

- 今天准备做什么
- 今天完成了什么
- 当前卡点
- 明天第一步

它更像当天日志，不是长期总表。

### `memory/weekly/current-week.md`

放当前周仍在推进的内容：

- 本周目标
- 当前优先级
- 本周待办
- 延后事项
- 还没完成、但还要继续滚动的任务

### `sessions/*.jsonl`

这是原始会话转录，不是整理过的知识库。

它的价值在于：

- 审计
- 排错
- 看一轮到底有没有调工具
- 追查“为什么这次没回复”

但它不适合直接替代 daily 和记忆文件，因为它太原始、太长、太杂。

一句话记忆：

- `sessions` 像聊天录像
- `memory/*.md` 像整理过的工作笔记

两者应该并存，不是二选一。

## 5. 当前状态问题，为什么应该“先读文件，不先搜索”

这是这次优化里最重要的一条规则。

### 哪些属于“当前状态问题”

例如：

- 今天我做了什么
- 今天咱们都聊了什么
- 今天先做什么
- 本周重点是什么
- 当前周计划是什么
- 明天第一步是什么

这类问题的特点是：

- 范围很近
- 文件位置很明确
- 不是在问“很久以前的模糊历史”

所以更稳的做法是：

1. 先读今天的 daily
2. 再读 `current-week.md`
3. 必要时再读昨天的 daily
4. 必要时再读 `MEMORY.md`

### 为什么这样更稳

因为这类问题最怕两件事：

- 搜索 query 发散
- 搜空了反而以为“记忆坏了”

而直接读当前文件：

- 可解释
- 可验证
- 更容易稳定复现

---

## 6. 模糊历史问题，什么时候再用 `memory_search`

### 哪些属于“模糊历史问题”

例如：

- 之前是不是讨论过
- 上次怎么说的
- 哪天提过这个方案
- 以前聊过那个点吗

这类问题才更适合优先走：

- `memory_search`
- 必要时结合 session transcript 搜索

### 这条规则为什么重要

因为搜索不是越早用越好。

如果本来就知道答案大概率在：

- 今天的 daily
- 当前周计划

那先搜索反而容易把简单问题复杂化。

---

## 7. `memory_search` 真正的工作方式是什么

这一点非常容易误解。

### 不是检索引擎自己帮你“拆问题”

当 agent 调用 `memory_search` 时，真正传进去的是一个 `query` 字符串。

这条 `query` 不是 QMD 自己从用户原话里拆出来的，而是：

- 模型先理解你的问题
- 模型再自己决定要搜什么
- 然后把那条 query 交给 `memory_search`

### 所以“搜成什么样”，很大程度上是 agent 行为问题

比如用户问：

```text
今天咱们都聊了啥呀，我干了啥呀，给我总结简单的列点形式，让我有个整体的认知。
```

如果模型先读了 today daily，再脑补出一串很散的词，例如：

- `今天`
- `完成`
- `weekly-assistant`
- `飞书`
- `绑定`
- `权限`
- `周计划`

那问题不在 QMD，而在：

- 这条 query 本身太散
- 太多泛词
- 太多不是用户当前问题里的关键词

### 这次优化后的建议规则

`memory_search` 的 query 应该优先来自：

- 用户当前这句话里的时间范围
- 用户当前这句话里的实体关键词
- 用户当前这句话里的任务主题

不要优先从历史上下文里脑补一长串相关词。

---

## 8. 时间词为什么不能直接拿去搜

用户常说这些词：

- 今天
- 昨天
- 本周
- 上周
- 上次
- 之前

这些词不能简单粗暴地直接原样丢进检索。

### 更稳的做法

- `今天`
  - 优先读今天的 daily
  - 必要时把它转换成具体日期

- `昨天`
  - 优先读昨天的 daily

- `本周`
  - 优先读 `current-week.md`
  - 再补读最近几天 daily

- `上周`
  - 优先看 `weekly/archive/`
  - 必要时再做搜索

- `之前 / 上次`
  - 才更适合进模糊历史搜索
  - 但必须再结合实体词

一句话：

**时间词本身不是好的检索词，应该先被转换成具体文件范围或具体日期范围。**

## 9. 为什么 `memory_search` 为空，但回答仍然可能是对的

这是很多人第一次排查时最困惑的点。

答案通常不是“记忆坏了”，而是：

### 情况 1：当前会话里已经读过相关文件

例如：

- `/new` 后先读了今天的 daily
- 再回答“今天我做了什么”

这时就算某次 `memory_search` 返回空：

- 模型仍然可以根据刚读过的文件回答

### 情况 2：当前 session 里已经有足够上下文

哪怕搜索没有命中，当前会话里可能已经有：

- 相关工具结果
- 相关 daily 内容
- 当前周计划摘要

所以不要把：

- `memory_search` 某次没命中

直接等同于：

- 这个 agent 什么都不知道

## 10. 内置 backend 和 QMD 到底差在哪

### 10.1 先说结论

它们都不是“只会纯关键词”或“只会纯语义”的极端方案。

两边都支持混合检索思路，只是：

- 内置 backend 更透明、参数更好理解
- QMD 更像高性能、可扩展、偏进阶的方案

### 10.2 内置 backend

优点：

- 零依赖
- 更稳定
- 更适合先跑通
- 一些参数暴露得更清楚

如果没额外配 embedding provider，内置 backend 仍然能工作，只是更偏：

- FTS / BM25 / 关键词检索

如果配了 embedding，再走 hybrid：

- 关键词
- 向量
- 加权合并

### 10.3 `extractKeywords(cleaned)` 是在干嘛

这一步不是 LLM 再帮你重写问题，而是一个更确定性的关键词清洗 / 提取动作。

它的作用可以理解成：

- 把一整句偏口语、偏聊天的话
- 提炼成更适合文本检索的关键词

它的好处是：

- 会去掉一些废词
- 对 FTS / BM25 更友好

它的局限是：

- 不是智能理解器
- 如果模型一开始构造的 query 就很差，它也救不了太多

### 10.4 QMD

QMD 更适合：

- 数据量更大
- 会话更多
- 希望把 sessions 也纳入检索

它的实际能力不是“只有语义”，而是更接近：

- BM25
- vectors
- reranking

对当前这条主线案例来说，QMD 的好处是：

- 可以把 `sessions/*.jsonl` 也纳入检索源
- 对长期积累的个人助手更有扩展性

但它也更适合：

- 已经愿意多装一点依赖的人
- 准备认真调长期助手的人

### 10.5 什么时候优先用哪一个

如果你现在最在意的是：

- 小白先跑通
- 更好排错
- 更容易理解

内置 backend 往往更合适。

如果你现在更在意的是：

- session 也要纳入检索
- 数据会越来越多
- 想在 WSL2 / Linux 下长期维护

QMD 会更有价值。

## 11. 为什么我们当前实验里保留了 daily 和记忆文件，而不是只靠 session 搜索

因为 session 搜索再强，也不应该替代结构化 daily。

理由很简单：

- session 是原始转录
- daily 是整理后的当天推进记录

对于这个“个人周计划与每日待办助手”来说：

- 早上问今天做什么
- 下午补进度
- 周五复盘

最稳的基础都还是：

- `memory/YYYY-MM-DD.md`
- `memory/weekly/current-week.md`

而不是把所有希望都压在“搜索 session transcript”上。

这也是为什么：

- 我们把 Cron 的设计重点放在“让 daily 可靠落盘”
- 而不是只追求“搜一下就知道”

---

## 12. 这个案例里，推荐的实际策略

对 `weekly-assistant`，最稳的顺序是：

1. 先把 `MEMORY.md`、daily、current-week 分工理顺
2. 让 Cron 把 daily 更新工作流跑稳
3. “今天 / 本周”类问题先读文件
4. “之前 / 上次 / 哪天提过”类问题再搜索
5. 搜索 query 优先从用户当前提问提取，不优先依赖历史脑补
6. session 搜索作为增强，不作为唯一依赖

一句话总结：

**当前状态靠读文件，模糊历史靠搜索，想长期记住的内容必须写回文件。**

## 13. 小白怎么自己验证这套机制有没有跑对

### 验证 1：看当前后端状态

```bash
openclaw memory status --agent weekly-assistant
```

你重点看：

- 当前 provider 是什么
- sources 里有没有 `memory`
- 如果你开了 session 索引，有没有 `sessions`

### 验证 2：看“今天 / 本周”类问题是不是先读文件

在飞书里开：

```text
/new
/think low
/reasoning on
/verbose full
```

然后问：

```text
今天我做了什么？
```

重点看：

- 有没有优先读今天 daily
- 有没有优先读 `current-week.md`
- 有没有一上来就跑发散搜索

### 验证 3：看“模糊历史”类问题是不是再搜索

继续问：

```text
之前我们讨论飞书绑定的时候，最后怎么定的？
```

重点看：

- 有没有触发 `memory_search`
- query 是否短、是否来自你当前这句话

### 验证 4：看下午追问后，daily 是否真更新了

等 17:00 的 Cron，或手动跑对应 job。  
你回复后，再检查今天的 `memory/YYYY-MM-DD.md`。

如果只聊了，但没写进去，这条链路就还不算稳定。

---

## 14. 一句话总结

OpenClaw 的“记忆”不是一团神秘黑盒。

更准确地说，它是：

- 当前 session
- 自动注入的根文件
- 工作区里的长期资料
- `memory_search` / `memory_get`
- 内置 backend 或 QMD

共同组成的一套系统。

真正稳定的个人助手，不是“让它自己神奇记住一切”，而是：

**把该长期保留的内容写到正确的文件里，把当前状态问题先交给文件，把模糊历史问题再交给检索。**
