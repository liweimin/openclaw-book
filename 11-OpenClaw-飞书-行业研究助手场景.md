# 第十一章：OpenClaw 的飞书行业研究助手场景

## 1. 这个场景解决什么问题

很多人要的不是“泛泛聊天”，而是一个真的能用的行业研究助手。

比如你在飞书里直接说：

```text
帮我持续研究食品行业，重点看新品、渠道和投融资。
```

一个成熟的 Agent 不应该立刻开始瞎搜。  
它应该先补问关键约束，例如：

1. 你说的“食品”具体是零食、饮料、乳制品，还是整个大食品赛道？
2. 重点看中国还是全球？
3. 你更关注政策、品牌、新品、渠道、价格、供应链，还是投融资？
4. 有没有必须优先的网站来源？
5. 你要一次性研究，还是每天/每周持续跟踪？
6. 结果是直接发回当前飞书会话，还是顺手写成飞书文档？

等这些信息确认完，再开始产出。  
这才是一个真正可落地的“行业研究应用场景”。

## 2. 适合写进书里的场景定义

### 场景名

飞书行业研究助手

### 典型用户

- 消费投资人
- 行业分析师
- 咨询顾问
- 品牌市场团队
- 战略研究团队

### 典型输入

```text
帮我做食品行业研究。
```

或者更具体一点：

```text
帮我跟踪中国食品饮料行业，重点看新品、渠道和投融资，每周一早上发我一版周报。
```

### Agent 的目标行为

1. 如果信息不全，先追问，不急着开始搜。
2. 信息补全后，先复述研究任务，避免理解偏差。
3. 先产出一版即时研究结果。
4. 如果用户要求日更/周更，再把这套研究偏好沉淀下来。
5. 如需定期跟踪，再建立 cron 定时任务。

## 3. 这个场景为什么适合用 Skill 来做

这个需求的核心不是某个单一工具，而是稳定流程：

1. 先收集研究简报
2. 再确认边界
3. 再组织来源
4. 再输出固定结构
5. 如果用户要持续跟踪，再记忆化并定时化

这种“先问清楚，再开始做”的流程，最适合用 Skill 固定下来。  
因为 Skill 本质上就是给 Agent 一份可复用的工作说明书。

但这里一定要讲清楚：

> Skill 解决的是流程约束，不是全部执行能力。  
> 真正把行业研究跑起来，还需要飞书渠道、browser 检索能力、memory 和记忆检索、以及 cron 定时能力配合。

## 4. 建议的交互流程

## 第一步：用户在飞书里提出模糊需求

```text
帮我研究食品行业。
```

## 第二步：Agent 自动追问

理想回答应该像这样：

```text
可以。我先补齐研究简报，确认后再开始：

1. 你关注的是整个食品行业，还是某个细分赛道，比如零食、饮料、乳制品、预制菜？
2. 重点看哪些维度：政策、市场规模、新品、品牌、渠道、价格、供应链、投融资、舆情？
3. 地域范围看中国、全球，还是指定地区？
4. 有没有必须优先的网站或机构来源？
5. 你要一次性研究，还是每天/每周持续跟踪？
6. 结果直接回当前飞书会话，还是写成飞书文档？
```

## 第三步：用户补充约束

```text
中国市场，重点看饮料和零食。
关注新品、渠道和投融资。
优先看 Foodaily、马上赢、尼尔森IQ、36氪消费、国家统计局。
每周一上午 9 点给我一版周报，先发当前飞书私聊。
```

## 第四步：Agent 复述并确认

```text
我理解的任务是：

1. 研究对象：中国食品行业中的饮料和零食
2. 核心维度：新品、渠道、投融资
3. 优先来源：Foodaily、马上赢、尼尔森IQ、36氪消费、国家统计局
4. 输出频率：每周一上午 9 点
5. 输出位置：当前飞书私聊

如果没问题，我先给你产出第一版研究简报，再按这个配置做周更。
```

## 第五步：先产出首版，再决定要不要定时

这一步很重要。  
不要一上来就定时，因为用户经常会在第一版出来后修改方向。

## 5.2 建议一起打开的能力

```bash
openclaw hooks enable session-memory
openclaw gateway restart
openclaw skills list
```

这里 3 个能力分别解决：

1. `session-memory`：用户确认过的研究偏好，后面更容易沉淀到记忆。
2. `skills`：让 Agent 知道它要先补问，不要直接开搜。
3. 飞书渠道：让整个流程在飞书里完成。

## 5.3 为什么建议开 `memorySearch`

因为行业研究这种需求往往不是一次性的。  
用户会反复告诉你：

- 我更关注哪个细分赛道
- 我更信哪些来源
- 我想要日报还是周报
- 我喜欢什么输出结构

这些内容非常适合沉淀为长期偏好。

## 7. 怎么把这个 Skill 用起来

OpenClaw 识别 Skill 的标准位置有两个：

1. `~/.openclaw/skills`
2. `<workspace>/skills`

你现在看到的这个示例 Skill 在书稿目录里。  
真正使用时，把整个文件夹复制到下面任一位置：

```text
~/.openclaw/skills/industry-research-assistant
```

或者：

```text
<你的 workspace>/skills/industry-research-assistant
```

然后执行：

```bash
openclaw skills list
openclaw skills info industry-research-assistant
openclaw gateway restart
```

## 8. 在飞书里怎么真正跑起来

## 8.1 一次性研究

在飞书私聊里直接发：

```text
帮我研究食品行业。
```

如果 Skill 生效，Agent 就会先追问。  
当你补全信息后，它再给你第一版研究结果。

## 8.2 持续研究

当第一版结果方向没问题，再在飞书里发：

```text
确认，按这个口径每周跟踪。
```

这时建议做两件事：

1. 让 Agent 把偏好写进 Memory
2. 给它加一个 cron 任务

## 9. 推荐的输出结构

行业研究类输出如果不固定结构，很容易越来越散。  
建议在书里固定成下面这套结构：

1. 本期核心结论
2. 关键动态
3. 重点数据与事实
4. 机会与风险
5. 值得继续跟踪的公司、品牌、渠道或事件
6. 参考来源

如果是周报，可以额外加一节：

7. 与上周相比的新变化

## 10. 如何给它配“每天/每周”的持续跟踪

这个需求最适合走 OpenClaw 的 cron，而不是让用户每次手动催。

### 每周一 9 点的周报示例

```bash
openclaw cron add ^
  --name "食品行业周报" ^
  --cron "0 9 * * 1" ^
  --tz "Asia/Shanghai" ^
  --session isolated ^
  --message "根据用户确认过的行业研究偏好，持续跟踪中国食品行业中的饮料和零食赛道，重点看新品、渠道和投融资。优先使用用户指定来源。输出固定为：核心结论、关键动态、重点数据、机会与风险、下周观察清单。结果按 announce 投递到上次飞书会话。" ^
  --announce
```

### 每天 8 点的日报示例

```bash
openclaw cron add ^
  --name "食品行业日报" ^
  --cron "0 8 * * 1-5" ^
  --tz "Asia/Shanghai" ^
  --session isolated ^
  --message "根据用户确认过的行业研究偏好，跟踪中国食品行业中的饮料和零食赛道，重点看新品、渠道和投融资。优先使用用户指定来源。结果按简洁日报格式输出，并投递到上次飞书会话。" ^
  --announce
```

### 为什么这里建议用 `isolated`

因为行业研究的定时任务很容易越跑越长。  
用隔离式 cron 更稳，有 3 个好处：

1. 不污染主会话
2. 可以单独看运行历史
3. 后面更容易换成更便宜的模型做定时任务

## 11. 这个场景最容易踩的坑

## 坑 1：没有 Skill，Agent 可能直接开始搜

所以这类需求最好不要只靠一句自然语言愿望。  
你要给它一个明确的工作流程约束。

## 坑 2：第一次就上定时任务

正确顺序应该是：

1. 先跑一次
2. 再修口径
3. 再定时

## 坑 3：没有固定来源优先级

行业研究里最怕“这一周看媒体，下一周看自媒体，第三周又换口径”。  
所以来源优先级必须在追问阶段定下来。

## 坑 4：没有沉淀 Memory

如果不记忆化，用户每次都得重复解释研究范围和输出格式。

## 坑 5：把“行业研究”写得过大

“研究食品行业”太泛。  
真正能长期稳定执行的，一定是：

- 有细分赛道
- 有关注维度
- 有来源优先级
- 有产出频率
- 有交付位置

## 12. 这段话可以直接写进书里

> 行业研究不是“给 Agent 一个行业词，让它随便搜”，而是先把研究简报补齐。对 OpenClaw 来说，最稳的做法是：通过飞书先收集行业、细分赛道、关注维度、来源优先级、地域范围、更新频率和交付位置，再由 Skill 约束 Agent 先确认后执行。对于一次性研究，先产出首版；对于持续跟踪，再把偏好写入 Memory 并通过 cron 建立每周或每日任务。

## 13. 本章对应的最小落地组合

如果你只想先跑一个最小可用版，建议组合是：

1. 飞书私聊接通
2. 行业研究 Skill 生效
3. `memorySearch` 打开
4. `session-memory` 打开
5. 第一版人工确认通过后，再补 cron

这样最稳，也最适合写书。
