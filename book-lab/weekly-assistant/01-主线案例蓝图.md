# 主线案例：个人周计划与每日待办助手蓝图

## 1. 为什么要先设计这个主线案例

这本书后面会讲很多能力：

- Dashboard
- Gateway
- 工作区
- Memory
- Heartbeat
- Cron
- 飞书

如果这些能力各讲各的，读者很容易学完一圈，还是不知道它们组合起来到底能做什么。

所以本书后续会用一个持续贯穿的案例，把这些能力逐步拼成一个真的可用的小助手：

**个人周计划与每日待办助手。**

它不是一次性 Demo，而是一个可以每周循环起来的使用场景。

---

## 2. 这个小助手到底要解决什么问题

很多人真正需要的，不是“一个会聊天的 AI”，而是一个能帮自己把事情理顺、推进、复盘的助手。

这个案例要解决的，是下面这类现实问题：

- 脑子里总有很多零散想法，但没有整理成清晰计划
- 每周想做很多事，最后经常抓不住重点
- 每天打开电脑时，不知道今天最该先做什么
- 周末想复盘时，又很难回想这一周到底推进了什么
- 想把 AI 接进自己的工作和生活，但不想只停留在“问一句答一句”

这个小助手的目标，不是替你决定人生，而是帮你完成下面这个循环：

1. 收集想法
2. 整理计划
3. 每日推进
4. 晚间/事后更新
5. 周末复盘
6. 滚动形成下周计划

只要这个循环成立，OpenClaw 就不再只是聊天页，而是一个开始真正参与日常工作流的个人助手。

---

## 3. 这个案例的完整闭环

### 阶段 1：先把“脑子里的事”倒出来

用户先不追求规范，不需要一开始就写成正式待办系统。

只要把下面这些东西记录下来就行：

- 这周想做什么
- 哪些事情最重要
- 哪些只是半成型想法
- 哪些任务有时间压力
- 哪些事情一直拖着没做

这一阶段的重点不是管理，而是**收集真实输入**。

### 阶段 2：整理成下周行动清单

OpenClaw 跑通之后，用户通过飞书私聊把这些原始想法交给它。

OpenClaw 帮用户做第一次结构化整理，产出：

- 下周推荐清单
- 任务优先级
- 今天先做什么
- 哪些信息还不完整

这一阶段的重点是：**从混乱输入到结构化输出**。

### 阶段 3：每天推进

用户在每天开始工作时，可以回来问：

- 我今天最应该先做什么
- 如果我只有 1 小时，先做哪一件
- 哪些任务应该继续推进，哪些可以延后

用户在当天结束时，也可以继续补充：

- 今天完成了什么
- 哪个任务卡住了
- 明天第一步应该接着做什么

这一阶段的重点是：**从周计划变成每日推进**。

### 阶段 4：让计划留在系统里，而不只是留在聊天记录里

如果只靠聊天历史，这个助手最终会变得不稳定。

要让它真的能长期使用，后面需要把这套信息逐步落到这个独立 agent 的工作区中：

- `MEMORY.md`
  - 放长期目标、习惯、固定偏好、长期项目
- `memory/YYYY-MM-DD.md`
  - 放当天推进、阻塞点、已完成事项
- `HEARTBEAT.md`
  - 放提醒规则，例如“早上提醒今天重点，晚上提醒复盘”

这一阶段的重点是：**从会话连续性升级到可持续记忆**。

### 阶段 5：周期性提醒与复盘

当 `HEARTBEAT.md`、Heartbeat、后续的 Cron 和飞书入口逐步加上后，这个助手就可以进入更完整的循环：

- 早上提醒：今天先做什么
- 晚上提醒：今天完成了吗
- 周末提醒：该复盘本周并生成下周计划了

这一阶段的重点是：**从被动问答升级到半主动陪跑**。

### 阶段 6：周末滚动

这个案例最关键的一步，不是“计划生成”，而是“每周滚动”。

到周末时，用户应该能和 OpenClaw 一起完成：

1. 回顾上周完成了什么
2. 看哪些任务没完成
3. 判断没完成的原因
4. 决定哪些任务延续到下周
5. 重新生成下周建议清单

到这里，这个助手才真正形成一个持续循环。

## 4. 这个案例能体现 OpenClaw 的哪些能力

这个主线案例之所以适合作为整本书的核心，是因为它不是只用到一两个功能，而是会逐步覆盖 OpenClaw 的核心能力。

### 第一层：入口能力

- 飞书 DM 主入口
- Dashboard / WebUI 辅助入口

解决的是：**从哪里和助手交互**。

### 第二层：会话能力

- 持续会话
- 会话历史
- 多次修订同一份计划

解决的是：**上次说过的话，这次还能继续接着做**。

### 第三层：工作区能力

- 工作区文件
- 工作区上下文
- 把结果落回工作区

解决的是：**不只是聊天，还能把资料组织起来**。

### 第四层：Memory 能力

- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- 后续 `memory_search`

解决的是：**长期目标和短期推进怎么区分保存**。

### 第五层：自动化能力

- Heartbeat
- 后续的 Cron

解决的是：**提醒、复盘、周期性检查怎么自动跑**。

### 第六层：多入口能力

- 飞书

解决的是：**这个助手不一定只在本机浏览器里使用**。

## 5. 哪些部分现在可以稳写，哪些要分阶段实现

为了保证这本书里的内容是真能落地的，需要先把边界说清楚。

### 现在可以稳写的

- 用独立 agent 承载这套长期助手
- 通过飞书 DM 做第一次周计划整理
- 在同一个 agent 的主 DM 会话里持续修订计划
- 使用工作区固定文件承接长期信息
- 后面通过 Heartbeat 做轻提醒
- 让飞书持续作为这个助手的主要入口

### 不能在前面章节里写得过满的

- 第二章默认就自动写任意工作区文件
- 第二章默认就自动从普通文件完整回读所有历史
- 第二章默认就每天主动追问你任务有没有完成

这些不是完全不能做，而是需要随着后面章节逐步补齐：

- 合适的工具权限
- 工作区文件约定
- Memory 习惯
- Heartbeat / Cron 配置
- 飞书入口

这本书后面会按这个顺序逐步把它做完整。

## 6. 这套小助手的最小可用版本

如果只做一个“今天就能开始用”的最小版本，它应该长这样：

### 输入

用户在飞书里告诉 `weekly-assistant`：

- 我这周在想什么
- 我下周大概要做什么
- 哪些事最着急
- 哪些事只是备选

### 输出

OpenClaw 帮用户整理成：

- 下周推荐清单
- 今天最先做什么
- 哪些信息还不够完整

### 使用方式

- 先给这个案例单独建一个 agent
- 先在飞书私聊里使用
- 以该 agent 的主 DM 会话连续推进
- 暂时不要求自动提醒

这是这本书前面几章最稳的起点。

---

## 7. 这套小助手的完整版本

当整本书的前半部分都走完后，它应该能进化成下面这样：

### 周内

- 用户每天来问“今天先做什么”
- 用户补充“这件事我做完了”
- OpenClaw 结合当前计划继续推进

### 晚间

- OpenClaw 提醒用户是否要做简短复盘
- 用户把完成情况记录下来

### 周末

- OpenClaw 帮用户回顾上周
- 识别延续任务、废弃任务、新出现的任务
- 重新整理成下周清单

### 入口

- 本地 Dashboard 可用
- 飞书 DM 可用

### 持续化

- 长期信息放入 `MEMORY.md`
- 每日推进放入 `memory/YYYY-MM-DD.md`
- 提醒规则放入 `HEARTBEAT.md`

## 8. 这套主线案例在本书中的章节分布

这套案例不会在一章里一次做完，而是会随着全书推进逐步补齐。

### 第一章：先把原始想法记下来

读者要做的事情：

- 把“这周脑子里的事情”先倒出来

本章产物：

- 一份原始想法记录

### 第二章：第一次整理成下周计划

读者要做的事情：

- 装好 OpenClaw
- 建一个独立 agent
- 把飞书入口接到这个 agent
- 把原始想法交给 OpenClaw
- 在同一会话里把计划整理清楚

本章产物：

- 第一版下周建议清单

### 第三章：让这套东西开始可持续

读者要做的事情：

- 理解工作区
- 准备长期使用的记录位置
- 区分“聊天历史”和“长期资料”

本章产物：

- 一套用于持续使用的小助手资料结构

### 第四章：接入飞书，让入口更贴近日常

读者要做的事情：

- 把小助手稳定接到飞书
- 理清单独 agent、渠道账号、bindings 三者关系

本章产物：

- 一个能在飞书里继续对话的待办助手入口

### 第五章：把它真正用在每周和每天

读者要做的事情：

- 跑完整的计划整理、今日行动、进度补充

本章产物：

- 第一版可重复使用的周计划 / 日待办工作流

### 第六章：补上提醒、排错和日常稳定性

读者要做的事情：

- 配置 Heartbeat
- 学会看日志、重启 Gateway、稳定使用

本章产物：

- 一个能做轻提醒和日常检查的小助手

### 第七章：再回头理解为什么它能工作

读者要做的事情：

- 把前面用过的能力和源码结构对应起来

本章产物：

- 对整个小助手运行机制的系统理解

这本书后面要做的，不是把 OpenClaw 讲成一个个孤立功能，而是逐步把它拼成一个真的能循环起来的个人待办助手：

**平时记事，周末复盘，下周规划，每日推进，后续再接飞书和提醒。**
