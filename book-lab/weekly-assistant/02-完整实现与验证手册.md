# 主线案例：个人周计划与每日待办助手

副标题：先做一套能在本机验证的完整实现，再拆回各章节

## 1. 这份文档是干什么的

这不是分章节教学稿，而是一份**整套场景的验证手册**。

目的只有一个：

先把“个人周计划与每日待办助手”完整跑通，确认它在本机上确实可用；确认可行之后，再把它拆回第一章到第七章。

所以这份文档会直接回答下面这些最关键的问题：

- 这个助手到底依赖 OpenClaw 的哪些能力
- 哪些能力现在就能稳定落地
- 每周计划应该存在哪里
- `MEMORY.md` 到底该放什么，不该放什么
- 每天怎么问、每天怎么更新
- 周末怎么复盘并滚动到下周
- 提醒是怎么做的
- 飞书接入后怎样继续使用

---

## 2. 先给结论：这套助手可以做成，但要分清“会话”与“文件”

这个案例能稳定落地，关键不在于“聊天本身”，而在于把两层东西分开：

### 第一层：会话层

用来做：

- 临时沟通
- 当下整理
- 追问和修订

这层依赖：

- 飞书 DM
- 可选的 Dashboard / WebUI（主要用于调试和排错）
- 会话历史

### 第二层：文件层

用来做：

- 长期保存
- 周计划滚动
- 每日推进记录
- 提醒规则

这层依赖：

- 工作区文件
- `MEMORY.md`
- `memory/*.md`
- `HEARTBEAT.md`

**这套小助手要稳定，就不能只靠聊天历史。**

---

## 3. 不建议把“滚动中的周计划”全放进 `MEMORY.md`

这个问题很关键，先说结论：

**不建议把每周滚动计划全部塞进 `MEMORY.md`。**

原因有 3 个：

### 原因 1：`MEMORY.md` 更适合放长期稳定信息

官方文档里，`MEMORY.md` 的定位更接近：

- 长期目标
- 稳定偏好
- 重要背景
- 持续有效的约束

例如：

- 我平时更重视工作任务而不是琐碎杂事
- 我希望助手默认用中文输出
- 我每周会做一次周复盘
- 我习惯先做高价值、低阻力的任务

这些适合放 `MEMORY.md`。

### 原因 2：每周计划是高频变化内容

每周计划的特点是：

- 本周有效
- 下周会改
- 有些任务完成后就应该归档
- 有些任务会延期到下一周

这类内容如果一直直接往 `MEMORY.md` 里塞，最后会变得越来越乱。

### 原因 3：`MEMORY.md` 会成为长期上下文的一部分

它不是无限大的垃圾桶。

如果把大量周计划、历史任务、已完成事项都堆进去，后面会有两个问题：

- 读起来越来越乱
- 真正重要的长期规则被淹没

所以更合理的做法是：

- `MEMORY.md` 放长期不易变化的内容
- 周计划放到 `memory/weekly/` 里
- 每日推进放到 `memory/YYYY-MM-DD.md`

---

## 4. 这套助手推荐的文件结构

下面这套结构，是我认为最适合这个案例、也最容易后续扩展的版本。

```text
~/.openclaw/workspace-weekly-assistant/
├─ MEMORY.md
├─ HEARTBEAT.md
├─ memory/
│  ├─ YYYY-MM-DD.md
│  └─ weekly/
│     ├─ current-week.md
│     └─ archive/
│        ├─ 2026-W10.md
│        └─ 2026-W11.md
```

### 每个文件分别干什么

#### `MEMORY.md`

放长期有效的信息，例如：

- 你的长期目标
- 你的任务偏好
- 你希望助手怎么协助你
- 周复盘和下周规划的固定规则

#### `memory/weekly/current-week.md`

放当前这一周仍在推进的内容，例如：

- 本周目标
- 本周待办
- 当前优先级
- 还没完成的任务
- 下周候选任务

这是这套小助手最核心的“当前周计划文件”。

#### `memory/weekly/archive/YYYY-Www.md`

放历史周计划归档，例如：

- 上周完成了什么
- 没完成什么
- 哪些移到下周

#### `memory/YYYY-MM-DD.md`

放当天推进记录，例如：

- 今天准备做什么
- 今天完成了什么
- 卡住的点是什么
- 明天第一步做什么

#### `HEARTBEAT.md`

放提醒规则，例如：

- 早上提醒我看今天重点
- 晚上提醒我补今天进度
- 周末提醒我做本周复盘和下周规划

## 5. 这套方案为什么可行

这套方案可行，不是靠猜，而是和 OpenClaw 当前能力对得上。

### 5.1 会话持续推进是可行的

OpenClaw 的会话历史会持久化在 Gateway 主机上。

这意味着：

- 你在飞书私聊里做的第一次周计划整理
- 后面可以在同一个 agent 的主 DM 会话里继续追问

这是可靠的。

### 5.2 工作区文件本来就是 OpenClaw 的长期资料层

官方文档明确说：

- 工作区是 agent 的 home
- `MEMORY.md`
- `memory/*.md`
- `HEARTBEAT.md`

都属于长期可用的工作区资料

这意味着：

- 把周计划和每日推进放进独立 agent 的工作区，是符合 OpenClaw 设计方式的

### 5.3 后续提醒这条线，第一阶段更适合先用 Cron

Heartbeat 当然能做提醒，但对第一次实验来说，**先用 Cron 更稳**。

原因是：

- 触发时间更精确
- 更容易验证“到底有没有发出来”
- 更容易排错
- 更适合先把固定节奏跑通

所以这套手册当前的推荐顺序是：

1. 第一阶段先用 Cron 跑通提醒
2. 第二阶段再考虑加 Heartbeat，升级成“更聪明地提醒”

### 5.4 飞书可以直接作为主要入口

官方 Feishu 插件支持：

- DM
- 配对
- 会话继续
- 绑定到同一个 agent

所以这套助手不是只能先在 Dashboard 里试，而是可以直接以飞书私聊作为主入口来验证。

## 6. 这套小助手的实现边界

为了保证这份手册是“真能跑通”的，下面几条边界必须说清楚。

### 6.1 这套方案不依赖“默认自动魔法”

它不应该依赖下面这些过度乐观的假设：

- 第二章默认就能自动写所有文件
- 普通文件下次会自动完整回读
- 默认就会主动每天追问你有没有完成任务

这些都不能作为第一版验证前提。

### 6.2 这套方案的第一版，重点是“你来触发，它能持续推进”

也就是说，第一版最稳的闭环是：

1. 你来输入
2. 它整理
3. 你再追问
4. 它继续推进
5. 后续再逐步补提醒

### 6.3 真正的“半主动提醒”要靠后面的 Heartbeat

所以这份手册里会分两档：

- 飞书里可验证的最小版本
- 加上提醒后的完整版本

## 7. 先确定结构：这个案例推荐单独建一个 agent，不建议直接塞进 `main`

这是这次最重要的设计决策。

按官方多 agent 设计，一个 agent 本来就代表一套独立的：

- 工作区
- 会话历史
- agent 身份与规则
- 可选的认证与模型配置

所以“个人周计划与每日待办助手”这种要长期滚动、长期积累上下文的案例，更适合单独建一个 agent，而不是直接混在 `main` 里。

### 为什么不建议先用 `main`

因为 `main` 往往会慢慢变成：

- 临时试命令的入口
- 以后别的案例的入口
- 泛用聊天和调试入口

如果把这个长期助手也直接塞进去，后面很容易出现：

- 上下文混杂
- 工作区文件混杂
- 提醒规则混杂
- 后续别的实战和这个案例互相串台

### 这个案例更推荐的结构

建议保留：

- `main`
  - 作为泛用、调试、临时实验入口

新增：

- `weekly-assistant`
  - 专门承载这套“个人周计划与每日待办助手”

### 真实使用里是不是也常常会有多个 agent

是的，而且这比“一个 main agent 处理所有长期任务”更合理。

比较常见的长期结构就是：

- 一个泛用主 agent
- 一个个人待办 / 复盘 agent
- 一个研究或写作 agent
- 一个编码或项目 agent

所以这次把案例做成单独 agent，不只是为了教学清晰，也更接近真实长期用法。

## 8. 先把这个独立 agent 建出来，并把飞书入口指向它

推荐 agent id：

```text
weekly-assistant
```

### 第一步：创建 agent

```bash
openclaw agents add weekly-assistant --workspace ~/.openclaw/workspace-weekly-assistant
```

执行后，它会有自己独立的：

- 工作区：`~/.openclaw/workspace-weekly-assistant`
- agentDir：`~/.openclaw/agents/weekly-assistant/agent`
- sessions：`~/.openclaw/agents/weekly-assistant/sessions`

### 第二步：把飞书消息路由给它

如果你当前这只飞书机器人，准备先只服务这个案例，最简单的做法是把这个 Feishu 账号的消息直接绑定给 `weekly-assistant`。

推荐配置思路如下：

```json5
{
  agents: {
    list: [
      { id: "main", default: true },
      {
        id: "weekly-assistant",
        workspace: "~/.openclaw/workspace-weekly-assistant",
      },
    ],
  },
  bindings: [
    {
      agentId: "weekly-assistant",
      match: { channel: "feishu", accountId: "main" },
    },
  ],
}
```

这里的意思是：

- `main` 还保留着
- 但这个飞书账号进来的消息，优先交给 `weekly-assistant`

### 这里的 `accountId: "main"` 是什么意思

它指的是 `channels.feishu.accounts.main` 这个账号。

如果你飞书账号的实际 id 不是 `main`，就把这里改成你自己的账号 id。

如果你还想系统理解：

- `session`
- `sessions`
- `MEMORY.md`
- `memory/*.md`
- `memory_search`
- `QMD`

之间的关系，建议结合阅读：

- [../../book/15-OpenClaw-记忆与检索-从工作区文件到QMD.md](../../book/15-OpenClaw-记忆与检索-从工作区文件到QMD.md)

### 以后如果还要做别的案例怎么办

后面有两种扩展方式：

- 最简单：
  - 每个长期案例一个独立渠道账号或独立机器人

- 稍复杂但更灵活：
  - 继续用同一个飞书账号
  - 再用更细的 `bindings`，按私聊对象或群组路由到不同 agent

当前这次验证，建议先走最简单的版本：**这只飞书机器人先只服务 `weekly-assistant`。**

## 9. 再把配置调到适合这个案例

如果按本地 onboarding 的默认配置，OpenClaw 很可能只拿到 `messaging` 级别的工具权限。这样不一定足够支撑这个案例里的文件读写。

### 推荐做法

先检查你的 `~/.openclaw/openclaw.json`，把工具权限至少调到能：

- 读取文件
- 写入文件
- 编辑文件

推荐起步配置：

```json5
{
  agents: {
    list: [
      {
        id: "weekly-assistant",
        tools: {
          profile: "messaging",
          alsoAllow: ["read", "write", "edit", "memory_search", "memory_get"]
        }
      }
    ]
  }
}
```

这套配置的意思是：

- 保留 `messaging` 这套默认消息能力
- 只给 `weekly-assistant` 额外放开工作区读写
- 只给 `weekly-assistant` 额外放开 Memory 检索

### 为什么不直接推荐 `full`

因为这个案例一开始不需要把所有能力都放开。

### 为什么也不直接推荐 `coding`

`coding` 当然也能做，但它会顺手打开更多运行时能力。对“个人待办助手”的第一版验证来说，没必要一上来就放那么大。

## 9.1 在飞书里原生打开调试可见性

如果你现在的目标是观察 agent 的执行过程，优先用 OpenClaw 原生聊天命令，不要再靠提示词里塞“执行简报”。

推荐在飞书私聊里这样开：

```text
/think low
/reasoning on
/verbose full
```

这 3 条分别是在做：

- `/think low`
  - 把思考强度开到较低但可见，先避免一上来太重。
- `/reasoning on`
  - 打开原生 reasoning 可见输出。
- `/verbose full`
  - 打开更完整的工具和过程可见性。

调试完以后，建议关回去：

```text
/reasoning off
/verbose off
```

再记住这几个常用命令：

- `/status`
  - 看当前会话和运行状态。
- `/new`
  - 开一个新会话，避免旧聊天历史干扰。
- `/reset`
  - 重置当前对话。
- `/compact`
  - 让长会话压缩上下文。

### 小白怎么用最稳

建议第一次就按这个顺序试：

1. `/new`
2. `/think low`
3. `/reasoning on`
4. `/verbose full`
5. 发你的测试问题
6. 看完后再 `/reasoning off` + `/verbose off`

### 为什么不用自定义“调试回执”

因为原生命令的好处更明确：

- 不需要额外改提示词去模拟
- 不会为了“解释过程”再额外浪费一轮输出
- 观察的是 Gateway/会话层原生暴露出来的过程，而不是 agent 自己编的一段总结

## 9.2 如果你想让 `memory_search` 也能搜会话

默认情况下，`memory_search` 只会搜：

- `MEMORY.md`
- `memory/**/*.md`

如果你还想让它把这个 agent 的 `sessions/*.jsonl` 也纳入检索，需要打开 `QMD` 后端，并开启 session transcript 索引。

推荐配置：

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      includeDefaultMemory: true,
      sessions: { enabled: true },
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }]
      }
    }
  }
}
```

### 打开它的好处

- 当你换了 `/new` 或 `/reset` 之后，`memory_search` 仍有机会回忆之前会话里谈过的关键内容
- 适合找“之前我们讨论过但还没整理进 `memory/*.md`”的内容
- 对排查“到底是哪轮聊天里确认过这件事”很有帮助

### 为什么默认不打开

- 这是实验性能力
- 会把 session transcript 也纳入索引，隐私面更大
- 首次可用前要准备额外环境，启动和首次查询也会更慢
- 如果你的日常习惯已经很好地把关键结论写进 `memory/*.md`，未必需要一开始就依赖它

### 打开之后还需要准备什么，才会真正可用

至少还要满足这些前提：

- WSL/Linux 里有 `qmd` 可执行文件
- WSL/Linux 里有 `bun`
- 系统里有可用的 SQLite 扩展环境
- Gateway 重启后，`qmd` 能在它的 `PATH` 里被找到

如果这些前提没满足，OpenClaw 会自动回退到 builtin memory backend。也就是说：

- 配置已经打开
- 但 `memory_search` 仍然只能搜普通 memory 文件
- 不会真的开始搜 session transcript

### 现在怎么检查它有没有真正生效

在 Ubuntu 里执行：

```bash
openclaw memory status --agent weekly-assistant
openclaw memory index --agent weekly-assistant --verbose
```

重点看：

- backend 是不是 `qmd`
- sources 里有没有 `sessions`
- 有没有关于 `qmd` 缺失或回退到 builtin 的提示

## 10. 这个 agent 的 DM 会话，推荐收敛成一个主会话

这个点在飞书场景里非常关键。

推荐配置：

```json5
{
  session: {
    dmScope: "main"
  }
}
```

### 这条配置在这里为什么合理

因为你当前要验证的是：

- 单个用户
- 单个飞书机器人
- 单个长期助手 agent

在这个前提下，你更希望：

- 周计划整理
- 日常追问
- 晚间复盘
- 后续提醒

都尽量围绕 `weekly-assistant` 的主 DM 会话来滚动。

### 什么时候不要这么配

如果这只飞书机器人后面会给多人共用，就不要为了图省事，把所有人的 DM 都并进一个主会话。

但对于这次“个人助手”验证，这个配置是合理的。

## 11. 先准备 `weekly-assistant` 的文件骨架

后面这套助手的文件，建议全部放在它自己的工作区：

```bash
mkdir -p ~/.openclaw/workspace-weekly-assistant/memory/weekly/archive
touch ~/.openclaw/workspace-weekly-assistant/MEMORY.md
touch ~/.openclaw/workspace-weekly-assistant/HEARTBEAT.md
touch ~/.openclaw/workspace-weekly-assistant/memory/weekly/current-week.md
touch ~/.openclaw/workspace-weekly-assistant/memory/$(date +%F).md
```

### `MEMORY.md` 初始模板

```md
# 长期记忆

## 个人工作方式

- 默认用中文输出。
- 优先帮我把任务整理成可以执行的清单，而不是空泛建议。
- 如果我一次说了很多事，先帮我分优先级，再给每日建议。

## 这套助手的固定规则

- 每周至少做一次周复盘。
- 周复盘之后，要生成下一周建议清单。
- 每天的推进记录放到 `memory/YYYY-MM-DD.md`。
- 当前一周的任务总表维护在 `memory/weekly/current-week.md`。
```

### `HEARTBEAT.md` 初始模板

```md
# 待办助手提醒规则

- 如果现在是白天，优先提醒我今天最应该做的 1 到 3 件事。
- 如果今天还没有推进记录，提醒我补一条今天的计划。
- 如果现在是晚上，提醒我补充今天完成了什么、卡住了什么、明天第一步做什么。
- 如果现在接近周末，提醒我回顾本周，并准备下周计划。
- 如果没有明显需要提醒的内容，保持安静，不要硬发消息。
```

### `memory/weekly/current-week.md` 初始模板

```md
# 当前周计划

## 本周目标

- 先留空，等第一次整理时生成

## 本周待办

- 先留空，等第一次整理时生成

## 当前优先级

- 先留空，等第一次整理时生成

## 延后事项

- 先留空，等第一次整理时生成
```

### `AGENTS.md` 和 `SOUL.md` 现在要不要改

第一轮搭骨架时，可以先不急着改。

但到了这一步，如果你已经开始验证：

- `/new` 之后它到底先读什么
- “今天 / 本周”类问题到底是先读文件还是先搜索
- daily 文件是不是能稳定维护

那就应该开始做**有针对性的 `AGENTS.md` 优化**了。

### 那什么时候再考虑改

真正适合改 `AGENTS.md` / `SOUL.md` 的时机，不是“还没用起来之前盲改”，而是你已经看到它有稳定偏差，例如：

- 总是不先读文件
- 总是把“今天 / 本周”问题先拿去做模糊搜索
- 总是太像客服
- 总是回答太啰嗦
- 总是不先给结论
- 总是忘了结合今天记录回答

这时再改，才是有针对性的优化。

### 这次优化后，`AGENTS.md` 保留了什么，去掉了什么

保留：

- Session 启动顺序
- 文件职责分工
- 文件优先 / 搜索优先的判断规则
- daily 文件维护规则
- 输出风格和安全边界

去掉或大幅缩短：

- 群聊发言规则
- reaction 规则
- 泛用 Heartbeat 巡检清单
- 和当前周计划助手无关的社交化说明

原因很简单：

- `AGENTS.md` 会进提示词链路
- 越短越聚焦，越容易让这个 agent 稳定地做对当前任务
- 不相关的规则越多，越容易分散注意力和浪费 token

还有一个很重要的写法原则：

- 尽量把规则写成“你现在要怎么做”
- 不要写成“AGENTS.md 负责什么”“这个文件是干嘛的”这种元说明

因为对模型来说，它看到的就是规则本身。  
直接写执行动作，比写一层“文件职责解说”更自然，也更容易稳定执行。

如果你想系统理解这组根文件的分工，而不是只看这次 `weekly-assistant` 的定制版，可以继续看：

- [../../book/16-OpenClaw-工作区根文件-AGENTS-SOUL-USER-IDENTITY-TOOLS-HEARTBEAT-BOOTSTRAP-MEMORY.md](../../book/16-OpenClaw-工作区根文件-AGENTS-SOUL-USER-IDENTITY-TOOLS-HEARTBEAT-BOOTSTRAP-MEMORY.md)

## 15. 周末怎么做“滚动到下周”

这是这套助手最关键的一步，建议固定在周末做。

### 第一步：在飞书里做周复盘

```text
现在请你和我一起做本周复盘，并准备下周计划。

请你完成下面 5 件事：
1. 读取 `memory/weekly/current-week.md`
2. 结合本周的每日记录 `memory/*.md`
3. 帮我总结本周完成了什么、没完成什么
4. 判断哪些任务应该延续到下周
5. 生成下周建议清单

输出结构：
1. 本周完成
2. 本周未完成
3. 延续到下周
4. 建议删除或暂缓
5. 下周建议清单
```

### 第二步：确认后再归档当前周

```text
请把当前周计划归档到 `memory/weekly/archive/2026-Wxx.md`，
然后把新的下周建议清单写回 `memory/weekly/current-week.md`。

归档文件里保留：
1. 本周完成
2. 本周未完成
3. 延续到下周
4. 本周复盘摘要
```

### 为什么这个动作适合单独 agent

因为它会持续写：

- 当前周计划
- 每日推进
- 历史周归档
- 提醒规则

这整套资料如果和别的实验场景混在同一个工作区里，后面会越来越乱。

## 16. 当前实验版的提醒：先用 Cron，不先上 Heartbeat

这份手册当前的推荐顺序已经调整了：

- **第一阶段先用 cron**
- **第二阶段再考虑 heartbeat**

原因很简单：

- cron 更容易理解
- cron 更容易验证
- cron 的触发时间更精确
- 现在我们最重要的是先把“提醒真的能到飞书”跑稳

### 当前实验版已经配置好的 3 个提醒

当前这套验证手册对应的实验提醒是：

- 工作日 `10:00`
  - 提醒“今天最重要的 1 到 3 件事”
- 工作日 `17:00`
  - 追问“今天完成了什么 / 当前卡点 / 明天第一步”
- 每周五 `17:30`
  - 做周复盘，并继续整理下周计划

这 3 个提醒更适合你现在的验证节奏，因为：

- 时间固定
- 触发时你大概率在电脑前
- 出问题时更容易判断到底是“没触发”还是“触发了但内容不对”

### 为什么当前阶段不先把 Heartbeat 打开

不是 Heartbeat 不好，而是它更适合第二阶段。

Heartbeat 的强项是：

- 周期性检查
- 上下文判断
- 如果今天已经处理过，就不再打扰

但它也更像“智能检查器”，对第一次实验来说不如 cron 直观。

所以这份手册当前的主线是：

1. 先用 cron 验证“能稳定提醒”
2. 再用 heartbeat 升级成“更聪明地提醒”

### 以后什么时候再加 Heartbeat

等你把下面这些都验证稳定之后，再考虑加：

- cron 能稳定发到飞书
- `weekly-assistant` 能结合当前周计划回答
- 每日记录能持续写入
- 周复盘和下周规划已经形成固定流程

到那时，Heartbeat 更适合承担：

- 今天已经更新过就别重复提醒
- 晚上如果还没写 daily 记录，再补一条提醒
- 周末如果还没做复盘，再温和催一次

## 16.5 `/new` 和 `/reset` 对这个助手有什么影响

这两个命令看起来像“小动作”，但对长期助手来说很重要。

先说结论：

- `/new`
- `/reset`

都会让**当前聊天进入一个新的 session id**，也就是把“这段聊天历史”切开。

### 会被清掉的是什么

被清掉的是：

- 当前这段聊天里的临时上下文
- 你刚刚口头说过、但还没写进文件的内容
- 这段会话累计起来的聊天历史 token

### 不会被清掉的是什么

不会被清掉的是：

- `weekly-assistant` 这个 agent 本身
- 它的工作区
- `MEMORY.md`
- `memory/weekly/current-week.md`
- `memory/YYYY-MM-DD.md`
- `HEARTBEAT.md`

所以对这个助手来说，更准确的理解是：

- 聊天历史像“短期脑子”
- 工作区文件像“长期笔记本”

`/new` 和 `/reset` 是把“短期脑子”清一下，不是把“长期笔记本”删掉。

### 这样做对 token 有帮助吗

有帮助。

因为新的会话不需要继续背着一大段旧聊天历史往前走，所以通常能减少这部分上下文消耗。

但它不是把 token 直接清零，因为每次新会话仍然会加载一些固定文件，例如：

- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `HEARTBEAT.md`
- 在主私聊里还会读 `MEMORY.md`

所以它的真实效果是：

- 清掉长聊天历史
- 保留这个 agent 的稳定规则和文件记忆

### 什么时候适合用

比较适合用 `/new` 或 `/reset` 的时候：

- 当前会话已经很长了
- 话题明显切换了
- 你怀疑它被旧聊天带偏了
- 你想降低聊天历史的 token 占用

### 什么时候不要急着用

如果你这轮对话里刚补充了很多重要信息，但这些信息还没写进文件，就不要立刻 `/new`。

更稳的顺序应该是：

1. 先让它把关键信息写进 `MEMORY.md` 或今天的 `memory/YYYY-MM-DD.md`
2. 再发 `/new` 或 `/reset`

### 对这个案例的推荐用法

最稳的习惯是：

- 一个阶段聊完
- 先写文件
- 再 `/new`

例如你可以先发：

```text
请先把我们刚才确认的关键信息写进今天的每日记录文件，然后我准备开一个新会话继续。
```

等它写完后，再单独发：

```text
/new
```

这样既能保留长期资料，又能把聊天历史负担降下来。

- 输出已经开始像可执行清单，而不是泛泛建议

### 第 3 步：确认文件能写进去

确认这些都成立：

- `MEMORY.md` 已经写好
- `memory/weekly/current-week.md` 已经生成内容
- `memory/YYYY-MM-DD.md` 已经能写入今天推进

### 第 4 步：确认日常更新能跑

确认这些都成立：

- 你能在飞书里问“我今天先做什么”

确认这些都成立：

- 它能总结本周完成 / 未完成

- 它能把上一周归档到 `archive/`

确认这些都成立：

- cron 提醒已经稳定
- 提醒内容已经足够实用
- 你开始觉得“固定提醒”还不够聪明
- 你希望它根据今天是否已经更新过来决定要不要再提醒

这时再加 Heartbeat，才是升级，而不是一开始就把排错复杂度拉高

## 17.1 当前这套 cron 的具体验证链路

这部分是给你明天和后面几天直接照着测的。

### 工作日上午 10:00：验证“今天要做什么”

你想看到的理想结果是：

- 飞书机器人准时来一条消息
- 它会结合 `memory/weekly/current-week.md`
- 如果今天已经有 daily 记录，它会优先根据今天记录提醒
- 输出不长，但能给出 1 到 3 个明确动作

如果它来得太泛，下一步就检查：

- 当前周计划是不是太空
- 今天的 `memory/YYYY-MM-DD.md` 是不是还没有内容

### 工作日下午 17:00：验证“追问进度”

你想看到的理想结果是：

- 它会明确问你 3 件事
- 今天完成了什么
- 当前卡点是什么
- 明天第一步是什么

这一步的重点不是“它讲了很多道理”，而是：

- 它有没有把你往 daily 更新流程里拉

### 周五下午 17:30：验证“周复盘 + 下周规划”

你想看到的理想结果是：

- 它会围绕本周 daily 记录和当前周计划来问
- 会把“完成 / 未完成 / 原因 / 下周重点 / 下周一第一步”串起来
- 不是只做复盘，而是真的往下周滚动

### 每次提醒触发后，建议你都做 3 个动作

#### 动作 1：先看飞书消息是不是到了

最先判断：

- 到没到
- 是否重复
- 内容是不是明显走偏

#### 动作 2：在 WSL 里看 cron 是否记录了执行

可以执行：

```bash
openclaw cron list
openclaw cron runs --id <job-id>
```

如果你只想手动立即验证，也可以执行：

```bash
openclaw cron run <job-id>
```

#### 动作 3：确认它有没有读到正确文件

如果提醒内容不对，优先检查：

- `memory/weekly/current-week.md`
- 当天的 `memory/YYYY-MM-DD.md`

因为这个案例里，大部分“提醒内容不够准”的问题，最终都不是 cron 本身坏了，而是它读取到的工作区内容还不够完整

### 17.2 这次规则优化后，接下来重点验证什么

这一步不是再泛泛地看“它会不会回消息”，而是专门验证这次**文件优先 / 搜索收敛 / daily 落盘**三件事有没有生效。

#### 验证 1：`/new` 之后，“今天 / 本周”类问题应优先读文件

在飞书里先执行：

```text
/new
/think low
/reasoning on
/verbose full
```

然后发：

```text
今天咱们都聊了啥呀，我干了啥呀，给我总结简单的列点形式，让我有个整体的认知。
```

这次你重点观察：

- 它是不是先去读今天的 `memory/YYYY-MM-DD.md`
- 它是不是会去读 `memory/weekly/current-week.md`
- 它有没有一上来就跑一条很散的 `memory_search`

理想结果是：

- 先读文件
- 再给总结
- 即使开了调试可见性，也不要刷很多没有信息量的碎消息

#### 验证 2：模糊历史类问题，才优先走 `memory_search`

你可以接着发：

```text
之前我们讨论飞书绑定的时候，最后是怎么定的？
```

这次重点观察：

- 它会不会触发 `memory_search`
- search query 是否主要来自你当前这句话
- query 是否足够短、足够聚焦

理想结果是：

- 不再用那种“今天 完成 权限 骨架 周计划”式的大串 query
- 更接近：
  - `飞书绑定`
  - `weekly-assistant 飞书绑定`
  - 或者再加一个具体日期范围

#### 验证 3：17:00 追问后，daily 文件要真正更新

到了工作日下午 17:00，或者你手动跑一次 cron：

```bash
openclaw cron run b25518f6-ab7c-4668-b2fd-992dca8d7e73
```

你回复这 3 个问题之后，再去看：

- `~/.openclaw/workspace-weekly-assistant/memory/YYYY-MM-DD.md`

这次重点不是看它有没有“安慰你”，而是看：

- 它有没有把你的回答写回今天的 daily
- 写入结构是否清楚
- 会不会只停留在聊天里，没有落盘

#### 验证 4：10:00 提醒能不能补齐空的 daily skeleton

如果你想专门验证这点，可以在非工作内容较少的一天，先把当天 daily 文件留空或删到很少内容，然后等 10:00 的提醒。

理想结果是：

- 它会先发现今天 daily 内容不足
- 会先补一个最小 skeleton，或明确把你往 skeleton 补齐流程里带
- 再根据 `current-week.md` 给出今天最重要的 1 到 3 件事

## 18. 这套方案当前最关键的现实判断

为了避免书里后面写过头，这里把最重要的判断再写一次。

### 可以确定成立的

- 用独立 agent 承载这类长期助手场景
- 用飞书私聊作为主要互动入口
- 用同一 agent 的主 DM 会话持续修订计划
- 用工作区文件保存长期资料
- 用 `MEMORY.md` + `memory/*.md` 支撑持续使用
- 用 per-agent Cron 先稳定做提醒

### 目前不应该写成默认就会自动发生的

- 不配文件工具也能自动把一切写好
- 普通文件默认每次都会自动完整回读
- 一开始就每天主动且准确地催你完成任务
- 一个 `main` agent 长期承载所有不同类型场景，仍然能始终保持清晰

这些都要在这份验证手册的基础上逐步补齐，而不是先当作既成事实。

---

## 19. 一句话总结

这套“个人周计划与每日待办助手”当前最稳的实现方式不是：

**把所有东西都塞进 `main`，再靠聊天自己记住一切。**

而是：

**给它单独建一个 agent，用飞书做主入口，用会话做整理，用独立工作区做保存，用 `MEMORY.md` 管长期规则，用 `memory/*.md` 管周计划和每日推进，第一阶段先用这个 agent 自己的 Cron 做提醒，后面再考虑加 Heartbeat。**
