测试背景：

- 测试时间：`2026-03-08`
- 测试入口：飞书私聊机器人
- 测试 agent：`weekly-assistant`
- 测试目标：验证“周末复盘 + 下周规划”这条主线能否在飞书里跑通

这份文档主要回答 4 个问题：

1. 这次测试到底哪些地方已经通过了
2. 你在飞书里看到的那些“英文思考过程”到底是什么
3. 为什么助手会发散、会额外给模板

## 2. 本次测试结论

### 2.1 后续修复更新

这份文档记录的是第一次失败测试。后续排查中，已经确认并修复了“最后一轮没写入”的根因：

- 不是飞书本身的问题
- 不是场景设计不可行
- 而是 `weekly-assistant` 当时的工具配置写错了

之前错误地用了限制性 `allow` 写法，导致模型实际没有稳定拿到 `read/write/edit` 工具。后续改成了：

```json5
tools: {
  profile: "messaging",
  alsoAllow: ["read", "write", "edit", "memory_search", "memory_get"]
}
```

修复后，本地 CLI 已经验证通过真实写入。

先说结论：

- **主会话路由是通的**
- **飞书消息确实进了 `weekly-assistant`**
- **周末复盘 + 下周规划的对话主线可以跑起来**
- **但当前“调试可见性”和“输出稳定性”还不够好**

更具体地说：

### 已通过

- 19:06 这轮消息，已经正确进入 `agent:weekly-assistant:main`
- 19:18 和 19:24 两轮追问与补充，也继续落在同一个主会话里
- 助手能根据你的补充信息继续收敛下周计划

### 未通过

- 它暴露给飞书的是**原始 reasoning / thinking 片段**
- 这些内容并不是你真正想看的“调用了哪些工具、做了什么动作”
- 它会顺手扩展、给模板、发散
- 最后一轮“请写入文件”没有完成，最终表现为**没有回复**

## 4. 你提的 4 个问题，我的判断

## 4.1 问题一：执行简报不是我想要的

这个反馈是对的。

你现在真正想看的是：

- 这轮它打算做什么
- 实际读了哪些文件
- 实际写了哪些文件
- 调用了哪些工具
- 工具有没有成功

而不是：

- 它帮你总结的周报执行简报

也就是说，你真正想看的不是任务内容摘要，而是 OpenClaw 原生的过程可见性。

### 当前为什么没做到

因为我们之前为了防止它在飞书里疯狂刷英文 reasoning，临时把“执行简报”收敛成了比较轻的尾部说明。

这解决了“刷屏”问题，但没有满足你“学习和观察工具调用过程”的目标。

### 后续更合适的方案

后续更合适的方式，不是再额外设计一套“调试回执”，而是直接使用 OpenClaw 原生聊天命令：

```text
/think low
/reasoning on
/verbose full
```

这样更适合：

- 在飞书里直接观察 agent 运行过程
- 不额外污染 `MEMORY.md`
- 不再要求 agent 自己生成一份“过程总结”

---

## 4.2 问题二：它发出来的英文内容到底是什么

你在飞书里看到的那些英文，大体上就是：

- 模型的原始 `thinking`
- 或者说 reasoning 片段
- 里面还混进了它想调用工具时的内部自言自语

比如会出现这种东西：

- `Let's use exec to inspect`
- `Need to actually call exec tool`
- `I need tool call format`

这些内容**不是正式回答**，也不是可靠的“工具调用日志”。

更准确地说，它是：

- 模型脑内草稿
- 工具尝试前的中间态
- 有时甚至是失败的思路碎片

### 这类内容的特点

- 可能是英文
- 可能不完整
- 可能前后矛盾
- 可能包含“本来不该发给用户”的中间话

所以它不适合直接当成教学材料。

### 对这类内容应该怎么理解

最实用的理解方式是：

- **它是原始 reasoning 泄露**
- **不是结构化调试日志**
- **只能拿来辅助判断模型当时大概在想什么**
- **不能把它当成真正可靠的执行记录**

---

## 4.3 问题三：它会发散，还想给模板

这个问题目前存在，但不算底层故障。

### 为什么会这样

因为当前 agent 的倾向还是“尽量帮助你补全任务”，所以它会自然做三件事：

1. 主动补结构
2. 主动给模板
3. 主动扩展成“更完整方案”

这在一般助理场景里是优点，但在你现在这种**验证型测试**里，会显得太发散。

### 这个问题后续怎么收

后面更适合加一条更硬的规则：

- 没有明确要求时，不额外给模板
- 没有明确要求时，不主动扩写成方案包
- 先完成用户指定动作，再决定是否补建议

换句话说，要把它从“热心助理”往“先精确执行，再适度补充”的方向收。

---

## 4.4 问题四：最后一次为什么没有回复

这个问题这次已经能明确判断为：

- 不是飞书断了
- 不是 Gateway 崩了
- 不是服务没起来

而是这一轮 agent 本身没有成功产出最终回复。

### 目前看到的直接现象

服务日志：

- `queuedFinal=false`
- `replies=0`

会话转录：

- 有一段 `thinking`
- 没有正式回答
- 没有文件写入结果

工作区文件：

- 没有被更新

### 我的判断

这更像是一次**模型执行中断 / 回复落空**，而不是通道故障。

从转录看，这一轮它把注意力错误地放到了：

- “先确定今天日期”
- “想调用某个工具”

但最后没有顺利走到：

- 读取现有文件
- 写入周计划文件
- 给出确认结果

也就是说，这次失败更像：

- **任务执行路径跑偏**
- **最终没有形成可发送的正式回答**

## 5. 这次测试真正验证出来了什么

即使有这些问题，这次测试仍然验证出了几个很重要的事实。

### 5.1 这条主线场景本身是成立的

“周末复盘 + 下周规划”这个场景，在飞书里是能跑起来的。

因为你已经验证到：

- 能发起真实对话
- 能在一个主会话里持续补信息
- 能逐步把问题从“散乱想法”收敛成“下周计划”

### 5.2 当前卡点不在场景本身，而在交互质量

也就是说，现在不是：

- 这个案例不可做

而是：

- 这个案例能做
- 但还要继续优化调试可见性、输出收敛和写文件稳定性

这两者差别很大。

## 6. 我建议的优化方向

这里先不直接改，只把方案列出来，方便下一轮逐项决定。

## 方案 A：改用原生 `/reasoning` + `/verbose` 调试

这是现在更推荐的方向。

目标：

- 用 OpenClaw 原生命令观察过程
- 不再让 agent 自己额外生成“执行简报”

推荐组合：

1. `/new`
2. `/think low`
3. `/reasoning on`
4. `/verbose full`

调试完后再关掉：

1. `/reasoning off`
2. `/verbose off`

## 方案 B：限制飞书里的碎片化过程输出

这一步不是不让你学习，而是把“学习素材”换成更稳的形式。

建议目标：

- 尽量减少没有信息量的碎片化过程输出
- 让飞书里保留“能看懂”的原生调试信息，而不是一堆空转碎句

因为原始 thinking：

- 不稳定
- 不一定可靠
- 很容易刷屏

## 方案 C：收紧发散行为

建议加 3 条规则：

1. 先完成用户指定动作
2. 没有明确要求，不主动给模板
3. 没有明确要求，不额外扩展成完整方案包

这样更适合你当前的验证阶段。

## 方案 D：给“写文件”动作加失败兜底

这一步很重要。

建议增加的规则是：

- 如果写文件失败，也必须回复
- 回复里明确告诉用户：
  - 想写哪个文件
  - 卡在哪一步
  - 现在文件有没有改动

这样至少不会再出现“服务正常，但用户完全收不到回复”的情况。

## 方案 E：专门为飞书场景做一套测试提示词

也就是把你今晚这类验证用语，整理成几条固定测试模板：

1. 讨论与追问测试
2. 收敛为周计划测试
3. 写入文件测试
4. `/new` 后恢复上下文测试

这样后面每次改动后，都可以快速回归验证。

---

## 7. 我建议下一轮怎么做

如果按投入产出比排序，我建议下一轮按这个顺序做：

1. 先改成原生 `/reasoning` + `/verbose` 调试
2. 再继续收紧飞书里的碎片化过程输出
3. 再加“写文件失败也必须回执”的兜底规则
4. 最后再收紧“不要乱给模板”

原因很简单：

- 你现在最在意的是“我想看它做了什么”
- 同时又不希望看到一堆不稳定的英文思考碎片

所以第一优先级现在应该是：

**把过程可见性从“乱飞的原始 reasoning”，收敛成“可控的原生命令调试模式”。**
