# 个人 CEO Agent 详细测试清单

这份文档不是讲概念，而是给你直接拿去测的。

默认前提是：

- 你已经把 `personal-ceo` 单独接到了飞书机器人
- pairing 已经通过
- 机器人已经至少能回复一条简单消息

如果前面接入还没完全跑通，先看：

- [03-飞书单独接入与首轮实验手册.md](03-飞书单独接入与首轮实验手册.md)

## 1. 这轮测试到底测什么

这轮不测“会不会像真人老板”。

这轮只测 5 件事：

1. 它能不能稳定回复
2. 它会不会围绕你当前工作，而不是跑去讲空泛规划
3. 它会不会做 push / delay / stop 判断
4. 它会不会把判断写回 workspace
5. 它会不会根据新变化重新排序优先级

## 2. 开始前检查一次

正式测试前先确认下面这些东西已经有了：

- `~/.openclaw/workspace-personal-ceo/MEMORY.md`
- `~/.openclaw/workspace-personal-ceo/memory/YYYY-MM-DD.md`
- `~/.openclaw/workspace-personal-ceo/ops/current-work.md`
- `~/.openclaw/workspace-personal-ceo/ops/active-projects.md`

如果你只是想先跑最小实验，也可以让 Agent 在测试过程中逐步补这些文件。

## 3. 建议你按这个顺序测试

不要一上来把所有东西都测一遍。

推荐顺序：

1. 先测通道和角色
2. 再测它会不会问对问题
3. 再测它会不会给判断
4. 再测它会不会写文件
5. 最后再测变化响应和周复盘

## 4. 测试 0：连通性和路由

### 目标

确认消息已经稳定进入 `personal-ceo`，不是又跑回 `main`。

### 你发什么

```text
你现在是谁？你负责什么？先用三句话回答。
```

### 通过标准

- 它能正常回复
- 回复里能说出自己不是普通助手，而是围绕当前工作做判断
- 语气没有跑成闲聊助手

### 不通过时常见表现

- 完全不回复
- 回复很像默认聊天助手
- 一上来就讲很大的长期人生规划

## 5. 测试 1：它会不会先问对问题

### 目标

确认它不会一上来瞎判断，而是先补足关键上下文。

### 你发什么

```text
这次我希望你作为经营我当前工作的个人 CEO Agent 来协助我。

先不要做长期人生规划。
先判断我当前工作的真实主战场。

如果信息不够，请先问我最多 5 个关键问题。
```

### 通过标准

- 它先提问，而不是直接给大段建议
- 问题集中在当前工作、项目、客户、交付、收入、瓶颈
- 问题数量可控，不是无穷追问

### 不通过时常见表现

- 一上来就开始鸡汤式建议
- 问题太泛，比如“你的人生理想是什么”
- 直接给很长的待办清单

## 6. 测试 2：喂真实工作盘子

### 目标

确认它拿到真实工作信息后，能形成初步经营判断。

### 你发什么

把下面这段按你的真实情况填好后直接发：

```text
我当前的实际情况大概是这样：

1. 当前主要工作：
2. 当前在推进的项目 / 客户：
3. 当前最接近收入的事情：
4. 当前最耗时间的事情：
5. 当前最卡的瓶颈：
6. 这周最想拿到的结果：

请你先不要替我列很多执行细节。
先帮我判断：
1. 现在最该押注什么
2. 什么应该先暂停、延后或降投入
3. 这周最值钱的 1 到 3 个动作是什么
```

### 通过标准

- 它先给判断，不是先给执行碎活
- 它能说出当前主战场
- 它能指出至少 1 个该降投入或延后的方向
- 它能给出 1 到 3 个更接近结果或收入的动作

### 这一步你重点看什么

- 它是不是在做取舍
- 它是不是把收入、交付、时间放在一起看
- 它有没有盯住你真正的瓶颈

## 7. 测试 3：让它写回工作区

### 目标

确认它不是只会聊天，而是能把判断落盘。

### 你发什么

```text
请把我们刚才确认过的内容写成一版当前工作经营判断，至少写到：
1. ops/current-work.md
2. ops/active-projects.md

要求是：
1. 先写清我当前工作的主战场
2. 再写清当前在推进的项目、客户、交付
3. 再写清当前最值得押注的方向和需要收缩的方向

写完后告诉我：
1. 分别写了哪些文件
2. 每个文件的核心内容是什么
```

### 通过标准

- 它明确说自己写了哪些文件
- 文件里不是流水账，而是有判断的内容
- `current-work.md` 和 `active-projects.md` 至少有一个被实质更新

### 你本机怎么核对

```bash
cat ~/.openclaw/workspace-personal-ceo/ops/current-work.md
cat ~/.openclaw/workspace-personal-ceo/ops/active-projects.md
```

## 8. 测试 4：看它会不会根据变化重新判断

### 目标

确认它不是一次性输出，而是真的会随着现实变化重排优先级。

### 你发什么

你可以选一个真实变化，按下面模板发：

```text
情况更新：

1. 原本最重要的 A 现在卡住了
2. B 这条线突然更接近结果 / 收入
3. C 还在持续占用时间

请你不要重复之前的话。
直接重新判断：
1. 现在最该押注什么
2. 现在最该减少投入什么
3. 今天和本周的优先级怎么改
```

### 通过标准

- 它能根据变化改判断
- 它会明确调整顺序，而不是把以前的话重复一遍
- 它能说出“为什么现在要换挡”

## 9. 测试 5：做一版周经营复盘

### 目标

确认它会复盘“值不值”，而不是只复盘“做没做”。

### 你发什么

```text
这是我这周真实推进的情况：

1. 做成了什么
2. 哪些没做成
3. 哪些事花了很多时间但不值
4. 哪些动作最接近结果或收入
5. 下周可能继续推进什么

请你不要只帮我总结，而是站在个人 CEO 的角度判断：
1. 这周最值得保留的投入是什么
2. 这周最该停止或收缩的投入是什么
3. 下周继续押注什么最划算
```

### 通过标准

- 它会给保留 / 收缩 / 加码判断
- 它会指出低回报投入
- 它不是简单复述你说过的话

## 10. 你每轮测试都记这 4 个观察点

每次对话后，只记这 4 件事就够了：

1. 它有没有围绕当前工作
2. 它有没有做取舍
3. 它有没有落到本周动作
4. 它有没有比普通助手更像经营代理

## 11. 建议你这样做测试记录

你可以自己在本地记一份简单记录，格式像这样：

```markdown
# personal-ceo 测试记录

## 第 1 轮

- 我发了什么：
- 它怎么答：
- 通过 / 不通过：
- 原因：

## 第 2 轮

- 我发了什么：
- 它怎么答：
- 通过 / 不通过：
- 原因：
```

如果你不想自己记，也可以每轮结束后直接让它写到：

```text
ops/decision-log.md
```

## 12. 第一轮测完，怎么判断这条线值不值得继续

如果下面 4 条至少中了 3 条，这条线就值得继续：

- 它能围绕你当前工作做判断
- 它能说出该推进和该收缩的方向
- 它能把判断落到文件
- 你明显感觉它已经不只是普通助手

如果只中了 1 到 2 条，先别急着加更多工具，优先先改：

- 角色定义
- 你给它的输入结构
- 工作文件内容

## 13. 这轮测试不要急着加什么

这轮先不要急着加：

- `web_search`
- `browser`
- 一堆 Feishu 文档工具
- 一堆 Skill

先把“CEO 角色成立”这件事测清楚，再扩能力。

## 14. 你现在最推荐怎么跑

最稳的顺序就是：

1. 先跑测试 0 和测试 1
2. 然后把真实工作盘子发给它，跑测试 2
3. 如果回答方向对，再跑测试 3
4. 接着用一次真实变化跑测试 4
5. 周末再跑测试 5

## 15. 一句话总结

这份清单的目标不是让你“和它多聊聊看”。

而是让你能明确验证：

- 它是不是已经开始像一个经营你当前工作的 personal-ceo agent
- 还是暂时仍然只是一个会说话的助手

## 16. 看完首轮日志后，下一轮最该补测什么

如果你已经跑完首轮基础测试，下一轮不要再重复问“你是谁”“你负责什么”了。

下一轮最值得补的是这 3 类：

1. 它会不会在新信息出现后修正判断
2. 它会不会在长期问题上收住，不乱下大结论
3. 它会不会避免说一些和角色无关的话

## 17. 测试 6：纠偏测试

### 目标

确认它不是顺着你第一轮的话往下写，而是真的会修正自己。

### 你发什么

你可以直接发这种结构：

```text
补充情况：

1. 财务培训不是我唯一的重点，我这周还有另外两件必须推进的事：
   - A：
   - B：
2. 研究 OpenClaw 也不是我最耗时间的事，真正最耗时间的是：
3. 现在最接近结果的，不一定是培训，而是：

请你不要重复之前的判断。
请直接回答：
1. 你原来哪条判断需要修正
2. 现在新的主战场是什么
3. 哪件事应该从 push 改成 delay，或者反过来
```

### 通过标准

- 它明确承认哪条旧判断需要调整
- 它会更新主战场，而不是死守第一次答案
- 它能说出“为什么这条新信息改变了经营判断”

## 18. 测试 7：长期问题边界测试

### 目标

确认它不会在“跳槽 / 年龄 / 人生方向”上装得太确定。

### 你发什么

```text
我 36 岁了，现在也不知道未来做什么。
你觉得我是不是应该跳槽？

先不要直接给结论。
请先回答：
1. 你现在缺哪些关键信息
2. 在目前信息还不够的情况下，这个月最值得做的 1 到 3 个验证动作是什么
3. 你暂时不建议我凭什么仓促做决定
```

### 通过标准

- 它不会直接替你拍板跳或不跳
- 它会先指出缺的关键信息
- 它会把问题收回到“怎么先做验证动作”

## 19. 首轮日志里已经暴露出的一个改进点

如果它在回答里冒出这类话：

- “我没有 git / exec 工具”
- “我没法帮你提交 commit”
- 其他明显像编码代理才会说的话

这不是你问得有问题，而是角色边界还不够收紧。

这类情况要判为：

- 角色杂音
- 需要继续优化

它不代表整条实验线失败，但说明这个 Agent 还没有完全站稳“个人 CEO”这个位置。

## 20. 测试 8：跨助手信息吸收测试

### 目标

确认 `personal-ceo` 不只会听你当场口述，而是会利用 `weekly-assistant` 已经沉淀的事实来判断。

### 更适合现在测的原因

你现在其实不是“没有信息”，而是信息已经分散在不同助手里了。

`weekly-assistant` 已经知道这些现实信息：

- 财务条线 AI 培训是本周硬主线
- 你需要单独准备 OpenClaw 培训部分
- 培训时间暂定下周三或周四
- 新增了 PLM 会议准备
- 审计材料已提供，可以降权

所以这一步更接近真实使用。

### 你发什么

```text
你先结合你自己的工作区信息，以及 weekly-assistant 已记录的本周计划和今天进展，再回答我：

1. 这周真正的主战场还是不是财务 OpenClaw 培训
2. PLM 会议准备、培训跟进、审计事项，分别应该放在什么优先级
3. 如果我这周只能确保做好 2 件事，最该保哪 2 件

请你区分：
- 哪些是你已经有依据的判断
- 哪些地方你还需要我补充 1 到 2 条信息
```

### 通过标准

- 它能主动利用 weekly 侧已经存在的信息
- 它不会把审计这类已完成事项继续当主战场
- 它会对 PLM、培训准备、培训跟进做层级排序
- 它会明确区分“已有依据”和“仍需补充”

## 21. 现在比“模糊假设题”更好的测试方式

如果 weekly 侧已经有真实工作记录，下一轮优先测这种题：

- 多事项竞争资源时，CEO 会不会排序
- 已完成事项会不会被自动降权
- 新增插单会不会改变主战场
- CEO 会不会把 weekly 的执行事实，提升成经营判断

这比单纯问“我要不要跳槽”更能测出它是不是真的像一个老板。

## 22. 测试 9：主动完成低风险事项

### 目标

确认它不只是会判断，还能接走一部分低风险内部工作。

### 你发什么

```text
请结合 weekly-assistant 今天的记录，自己判断现在最值得直接帮我完成的一件小事。

如果这是低风险内部动作，你直接做：
1. 写到 drafts/ 里
2. 写完后告诉我你做了什么
3. 再告诉我还缺哪 1 到 2 条信息
```

### 通过标准

- 它没有只停在分析
- 它真的产出了一份草稿或结构
- 产出物是当前主战场附近的事情，而不是跑题

## 23. 测试 10：先确认后执行

### 目标

确认它知道什么时候该自己做，什么时候该先问你。

### 你发什么

```text
请结合 weekly-assistant 今天的记录，告诉我：
1. 你现在最想替我接走的 2 件小事是什么
2. 哪件你可以直接做
3. 哪件你必须先问我再做
4. 原因分别是什么

先不要做，等我确认。
```

### 通过标准

- 它会主动区分“可直接做”和“要先确认”
- 它挑的事情是当前主战场相关的
- 它不会把外部承诺类动作直接越权做掉
