# 个人 CEO Agent：飞书单独接入与首轮实验手册

这份手册解决两个最实际的问题：

1. 如果你愿意给这条线单独接一个飞书机器人，应该怎么配
2. 个人 CEO Agent 第一轮到底该开哪些工具，哪些不要一上来全开

先说结论：

- **可以单独接飞书机器人，而且我建议你这么做**
- **但第一轮不要把工具和技能全开**

原因很简单：

- 单独机器人有利于观察和复盘
- 全开工具会把实验噪声放大
- 个人 CEO 这条线第一轮最重要的是“经营判断是否成立”，不是“能力是否最全”

## 1. 这轮实验推荐的结构

如果你准备给它单独一只飞书机器人，我建议这轮实验用下面这套结构：

```text
默认主线（保留）
└─ main

个人 CEO 实验线（新增）
└─ personal-ceo
   ├─ 独立飞书机器人
   ├─ 独立 workspace
   └─ 独立 sessions
```

也就是说：

- 理论上这条线是在验证“默认主 Agent 能不能升级成 CEO”
- 但为了实验干净、便于观察，**这轮先用一个单独的实验 agent 承载**

等实验稳定后，你再决定：

- 把方法迁回 `main`
- 或者继续保留成独立的 personal-ceo agent

## 2. 第一轮到底要不要把工具技能全开

不建议。

### 第一轮推荐只开这些

- `messaging` 基础消息能力
- `read`
- `write`
- `edit`

### 第二阶段再考虑加这些

- `memory_get`
- `memory_search`

### 明确不建议第一轮默认就开这些

- `web_search`
- `web_fetch`
- `browser`
- 很多 Feishu 文档 / wiki / drive 能力
- 一堆自定义 Skill
- `full` 工具档位

原因：

### 为什么第一轮不建议全开工具

因为个人 CEO 第一轮要验证的重点是：

- 它会不会围绕你当前工作做判断
- 它会不会做取舍
- 它会不会把判断写进工作区

这些事情不需要一上来就全能力拉满。

### 为什么第一轮不建议先堆 Skills

因为你现在先要验证的是：

- 这个角色是否成立
- 你的交互方式是否稳定

如果一上来就堆很多 Skill，最后你很难判断：

- 是 CEO 角色本身成立
- 还是 Skill 在硬拉流程

所以第一轮更稳的做法是：

- 先靠角色定义 + 工作文件 + 连续对话来验证

## 3. 第一步：创建实验 agent

推荐 agent id：

```text
personal-ceo
```

在 WSL / Linux shell 里执行：

```bash
openclaw agents add personal-ceo --workspace ~/.openclaw/workspace-personal-ceo
```

执行完后，它会有自己独立的：

- 工作区：`~/.openclaw/workspace-personal-ceo`
- agentDir：`~/.openclaw/agents/personal-ceo/agent`
- sessions：`~/.openclaw/agents/personal-ceo/sessions`

## 4. 第二步：先准备最小工作文件

先在这个 workspace 里准备两份文件：

```text
ops/current-work.md
ops/active-projects.md
```

你可以先这样建：

```bash
mkdir -p ~/.openclaw/workspace-personal-ceo/ops
nano ~/.openclaw/workspace-personal-ceo/ops/current-work.md
nano ~/.openclaw/workspace-personal-ceo/ops/active-projects.md
```

### `ops/current-work.md` 第一版模板

```markdown
# 当前工作盘子

## 当前主要工作

- 

## 当前最重要的收入通道

- 

## 当前最耗时间的事情

- 

## 当前最卡的瓶颈

- 

## 本周最想拿到的结果

- 
```

### `ops/active-projects.md` 第一版模板

```markdown
# 当前项目与交付

## 项目 / 客户清单

### 项目 1

- 当前状态：
- 最接近结果的动作：
- 当前风险：
- 本周是否继续加码：

### 项目 2

- 当前状态：
- 最接近结果的动作：
- 当前风险：
- 本周是否继续加码：
```

第一轮不用写得很全。  
只要你把当前工作现实情况填进去，这条线就能开始跑。

## 5. 第三步：在飞书开放平台创建第二个机器人

这一步不要复用你当前已经在用的飞书机器人，直接新建一只。

推荐命名：

- `OpenClaw-Personal-CEO`
- `个人CEO实验助手`

操作顺序：

1. 在飞书开放平台创建新的企业自建应用
2. 记录新的 `App ID` 和 `App Secret`
3. 开启机器人能力
4. 启用长连接事件订阅
5. 添加事件 `im.message.receive_v1`
6. 创建版本并发布应用

## 6. 第四步：更新 `~/.openclaw/openclaw.json`

如果你已经有一只主飞书机器人在跑，最稳的做法是：

- 保留原来的 `main`
- 给个人 CEO 这条线新增一个 `ceo` 账号
- 用 `bindings` 把它路由给 `personal-ceo`

推荐配置示意：

```json5
{
  agents: {
    list: [
      { id: "main", default: true },
      {
        id: "personal-ceo",
        workspace: "~/.openclaw/workspace-personal-ceo",
        tools: {
          profile: "messaging",
          alsoAllow: ["read", "write", "edit"]
        }
      }
    ]
  },

  session: {
    dmScope: "main"
  },

  bindings: [
    {
      agentId: "personal-ceo",
      match: { channel: "feishu", accountId: "ceo" }
    }
  ],

  channels: {
    feishu: {
      defaultAccount: "main",
      dmPolicy: "pairing",
      groupPolicy: "allowlist",
      requireMention: true,
      accounts: {
        main: {
          appId: "cli_xxx_main",
          appSecret: "xxx_main"
        },
        ceo: {
          appId: "cli_xxx_ceo",
          appSecret: "xxx_ceo",
          botName: "personal-ceo"
        }
      }
    }
  }
}
```

这份配置有 4 个关键点：

1. `personal-ceo` 有独立 workspace
2. `bindings` 里用 `accountId: "ceo"` 固定路由
3. `dmScope: "main"` 让它在这只机器人里尽量保持单一长期 DM 会话
4. 工具只开到 `read/write/edit`，不一上来全开

## 7. 第五步：重启并检查 Gateway

改完配置后执行：

```bash
openclaw gateway restart
openclaw gateway status
openclaw channels status --probe
```

至少要确认：

1. Gateway 正常
2. 原来的飞书机器人没受影响
3. 新增的 `ceo` account 已经加载成功
4. 没有明显的凭证或连接错误

## 8. 第六步：先验收路由是否正确

先给新的飞书机器人发一条：

```text
ping
```

如果触发 pairing，就执行：

```bash
openclaw pairing list feishu
openclaw pairing approve feishu <CODE>
```

然后再次发：

```text
ping
```

再在终端里检查：

```bash
openclaw sessions --agent personal-ceo
openclaw sessions --agent main
```

理想结果：

- `personal-ceo` 出现新 session
- `main` 不因为这只新机器人增加 session

只要这一步成立，后面的实验就干净了。

## 9. 第七步：第一轮开场消息直接这样发

在新的飞书机器人私聊里直接发：

```text
这次我不想把你只当普通助手，而是想让你先作为“经营我当前工作的个人 CEO Agent”来协助我。

先不要给我很大的长期规划，也不要上来讲人生目标。

你先只围绕我当前的工作来判断：
1. 我现在真正的主战场是什么
2. 哪些项目、客户、交付和收入线最值得优先投入
3. 哪些事情在占用我的时间，但未必值得继续投
4. 这周我最该推进的 1 到 3 件事是什么

如果信息不够，请先问我最多 5 个关键问题，把“当前工作盘子”先问清楚。
```

这一轮重点看：

- 它是不是先问现实情况
- 它是不是开始做取舍判断
- 它有没有跑去讲空泛长期规划

## 10. 第八步：第二轮把真实工作材料喂给它

当它第一轮发问方向对了，再发：

```text
我当前的实际情况大概是这样：

1. 当前主要工作：
2. 当前在推进的项目 / 客户：
3. 当前最接近收入的事情：
4. 当前最耗时间的事情：
5. 当前最卡的瓶颈：
6. 这周最想拿到的结果：

请你先不要替我列很多执行细节。
先帮我判断：
1. 现在最该押注什么
2. 什么应该先暂停、延后或降投入
3. 这周最值钱的 1 到 3 个动作是什么
```

## 11. 第九步：第三轮要求它写回工作区

如果前两轮已经像样，就发：

```text
请把我们刚才确认过的内容写成一版当前工作经营判断，至少写到：
1. ops/current-work.md
2. ops/active-projects.md

要求是：
1. 先写清我当前工作的主战场
2. 再写清当前在推进的项目、客户、交付
3. 再写清当前最值得押注的方向和需要收缩的方向

写完后告诉我：
1. 分别写了哪些文件
2. 每个文件的核心内容是什么
```

然后去本机确认：

```bash
ls ~/.openclaw/workspace-personal-ceo/ops
```

如果你想继续核对内容：

```bash
nano ~/.openclaw/workspace-personal-ceo/ops/current-work.md
nano ~/.openclaw/workspace-personal-ceo/ops/active-projects.md
```

## 12. 第十步：第一周怎么用最稳

### 每天早上

直接问：

```text
基于我们已经确认过的当前工作盘子，今天最该押注的 1 到 3 件事是什么？
```

### 每天中途

如果现实发生变化，直接补：

```text
情况更新：现在 A 卡住了，B 有机会提前成交，C 还要不要继续投？
请你从个人 CEO 的角度重新排一下优先级。
```

### 每周末

直接问：

```text
这是我这周真实推进的情况：

1. 做成了什么
2. 哪些没做成
3. 哪些事花了很多时间但不值
4. 哪些动作最接近结果或收入
5. 下周可能继续推进什么

请你不要只帮我总结，而是站在个人 CEO 的角度判断：
1. 这周最值得保留的投入是什么
2. 这周最该停止或收缩的投入是什么
3. 下周继续押注什么最划算
```

## 13. 什么时候再加更多工具

下面这个顺序更稳：

### 第一个阶段

只用：

- `messaging`
- `read`
- `write`
- `edit`

### 第二个阶段

如果你发现它开始需要跨天回忆历史，再加：

- `memory_get`
- `memory_search`

### 第三个阶段

如果你想让它开始：

- 看市场机会
- 比较行业变化
- 看岗位 / 定价 / 商机

再加：

- `web_search`
- `web_fetch`
- 必要时 `browser`

### 第四个阶段

如果你已经把这条线跑稳了，再考虑补 Skill。

Skill 最适合固定的是：

- 周复盘框架
- 当前工作盘子更新规则
- 收入机会判断模板

## 14. 第一轮成功标准

如果下面这些事都发生了，说明这条线已经有戏：

- 新飞书机器人已经稳定路由到 `personal-ceo`
- 它没有滑到空泛人生规划
- 它开始围绕你当前工作做判断
- 它能给出明确的押注 / 收缩建议
- 它已经把经营判断真正写回工作区

## 15. 一句话总结

这条实验第一轮真正想验证的不是：

**工具是不是开得最全。**

而是：

**在单独飞书机器人 + 独立实验 agent 的前提下，个人 CEO 这个角色能不能先围绕你当前工作做出真实、有用、可落盘的经营判断。**
