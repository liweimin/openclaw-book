# 行业研究助手：检索与配置说明

## 1. 先回答一个关键问题

不是只写一个 `SKILL.md` 就够了。

`SKILL.md` 解决的是：

- 什么时候触发这个流程
- 信息不全时先问什么
- 信息补全后怎么组织工作

但真正把行业研究场景跑起来，还至少依赖下面 4 类 OpenClaw 自带能力：

1. 渠道能力：通过飞书接收需求和回传结果
2. 浏览器能力：打开网站、阅读网页、点击、截图、提取页面信息
3. 文件与记忆能力：保存研究偏好、来源清单、阶段性结论
4. 定时能力：按天、按周、按月持续执行

所以最小闭环不是“只有 Skill”，而是：

`飞书 + Skill + browser + memory + cron`

## 2. 这个场景优先用哪些 OpenClaw 内置能力

## 2.1 飞书渠道

用途：

- 用户在飞书里发起行业研究
- Agent 在飞书里追问补充信息
- Agent 在飞书里返回研究结果

建议配置：

```json5
{
  channels: {
    feishu: {
      enabled: true,
      connectionMode: "websocket",
      dmPolicy: "pairing"
    }
  }
}
```

## 2.2 browser 能力

用途：

- 打开用户指定的网站、行业媒体、统计局页面、公司官网
- 读取网页内容
- 在需要时点击分页、打开详情页、截图留证

建议配置：

```json5
{
  browser: {
    enabled: true,
    defaultProfile: "openclaw"
  }
}
```

说明：

- `browser.enabled` 必须为 `true`
- 建议 `defaultProfile: "openclaw"`，这样用的是 OpenClaw 管理的独立浏览器

行业研究里，`browser` 是最关键的内置能力之一。  
没有它，Skill 只能规定流程，不能稳定地读网页来源。

## 2.3 检索链路的兜底顺序

行业研究默认还是先走 OpenClaw 内置链路：

`web_search -> web_fetch -> browser`

但这套链路并不总是因为“模型不会用”而失败，很多时候是环境本身没打通。  
在当前这类 Windows + WSL + Clash TUN 的机器上，最常见的两类问题是：

1. 目标站点被解析到 `198.18.x.x`，导致内置 `web_fetch` 被 SSRF 规则拦住
2. WSL 访问不到 Windows Chrome 的 CDP 端口，导致内置 `browser` 起不来

所以这类场景建议预先定义好兜底顺序，而不是临时碰运气：

1. 搜索层：Tavily Search
2. 抽取层：Firecrawl
3. 摘要层：Summarize
4. 浏览层：agent browser

使用原则：

- 内置 `web_search` 正常时，可以继续保留
- 内置 `web_fetch` 一旦连续出现 SSRF / special-use IP 错误，就改走 Firecrawl
- 内置 `browser` 一旦起不来，就不要再做 fetch vs browser 结论，直接改用 agent browser
- 长页面、批量网页、报告整理时，用 Summarize 做压缩，不要把原始长文整段塞回对话

如果这些 skill 没装上，就应该把它记录成“环境未就绪”，而不是写成“该来源没有内容”。

## 2.4 文件与记忆能力

用途：

- 保存用户的研究偏好
- 保存来源清单
- 记录阶段结论
- 对历史研究结果做持续检索

建议配置：

```json5
{
  agents: {
    defaults: {
      workspace: "D:/openclaw-workspace",
      memorySearch: {
        provider: "openai",
        model: "text-embedding-3-small",
        remote: {
          apiKey: "YOUR_OPENAI_API_KEY"
        }
      }
    }
  }
}
```

再建议打开：

```bash
openclaw hooks enable session-memory
```

## 2.5 cron 定时能力

用途：

- 每天跑行业日报
- 每周跑行业周报
- 每月跑行业复盘

建议用法：

- 首版先人工确认
- 通过后再建 `isolated` cron

## 3. 这个场景应该提取哪些“配置字段”

这里说的“配置”，不一定都写进 `openclaw.json`。  
更合理的做法是把它们拆成两层：

## 3.1 系统配置

这部分放在 `openclaw.json`：

- 飞书接入
- browser 开关
- memorySearch 开关
- workspace 路径
- 是否允许 doc / wiki / drive

## 3.2 研究简报配置

这部分更适合沉淀到记忆或工作区文档，而不是硬写到系统配置里。

建议提取下面这些字段：

1. `industry`
2. `segment`
3. `geography`
4. `focus_dimensions`
5. `priority_sources`
6. `excluded_sources`
7. `priority_urls`
8. `tracked_companies`
9. `tracked_brands`
10. `tracked_channels`
11. `tracked_keywords`
12. `cadence`
13. `output_format`
14. `delivery_target`

你可以把这理解成一份“研究简报对象”。

## 4. 用户给了网站、域名或具体链接时，应该怎么处理

这是这个场景里最重要的细节之一。

用户补充：

- 网站名
- 域名
- 公众号名
- 报告链接
- 公司官网链接
- 招股书链接
- 具体文章 URL

这些都应该被当成高优先级输入，不应该被忽略。

处理规则建议固定成这样：

1. 用户给了具体 URL：优先读这个 URL
2. 用户给了网站或域名：优先在这个站点内检索
3. 用户给了一组来源：按用户顺序作为优先级
4. 用户明确说“不看某类来源”：加入排除来源

额外补一条：

5. 如果这一轮为了取证改用了 Tavily、Firecrawl 或 agent browser，要在输出里写清楚切换点和原因

## 5. 推荐的研究简报模板

如果用户信息不全，你可以按这个模板追问或脑内补齐：

```text
研究对象：
细分赛道：
地域范围：
关注维度：
重点公司/品牌/渠道：
优先来源：
禁用来源：
指定链接：
输出频率：
输出形式：
投递位置：
```

## 6. 推荐的持久化位置

如果要让这个场景长期可用，建议在工作区保留一份研究画像文件，例如：

```text
<workspace>/memory/industry-research-profile.md
```

或者把长期稳定偏好写入：

```text
<workspace>/MEMORY.md
```

推荐写入的内容：

- 我长期关注哪个行业和赛道
- 我优先相信哪些来源
- 我习惯的日报或周报结构
- 我希望投递到哪个飞书会话或文档

## 7. 结果里至少要保留哪些证据

行业研究不是只交结论，还要能回看“这结论怎么来的”。

每次输出至少保留下面这些信息：

1. 实际使用的搜索词
2. 实际打开的来源链接
3. 实际使用的工具或 skill
4. 关键错误原文
5. 哪些来源是一手，哪些是二手

只有这样，后续复盘时才能区分：

- 是搜索词不对
- 是来源选错了
- 是 `web_fetch` / `browser` 环境没通
- 还是模型真的不会总结

## 7. 推荐的最小操作顺序

1. 接通飞书私聊
2. 打开 browser
3. 打开 memorySearch
4. 安装并启用本 Skill
5. 在飞书里先跑一版研究
6. 确认口径后再建立 cron

## 8. 这个 Skill 实际依赖的能力清单

建议在书里直接写成下面这张表：

| 能力 | 是否必需 | 用途 |
| --- | --- | --- |
| 飞书渠道 | 必需 | 接收需求、追问、回传结果 |
| Skill | 必需 | 约束先追问再研究的流程 |
| browser | 必需 | 打开并读取网站、文章、报告页面 |
| read/write/edit | 建议 | 保存研究过程和结果到工作区 |
| memorySearch | 建议 | 记住研究偏好并检索历史结论 |
| session-memory hook | 建议 | 把阶段性会话沉淀为记忆 |
| cron | 持续跟踪必需 | 每天/每周自动运行 |
| feishu_doc | 可选 | 把周报写成飞书文档 |

## 9. 一句最重要的结论

行业研究助手不是“给一个行业名，然后直接搜”。  
更稳的做法是：

先通过 Skill 把研究简报问清楚，再用 OpenClaw 内置的 browser、memory、cron 和飞书渠道把整个流程真正跑起来。
