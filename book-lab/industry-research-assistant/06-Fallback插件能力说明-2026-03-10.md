# Fallback 插件能力说明（2026-03-10）

这份说明只回答 3 个问题：

1. 这些插件分别替代哪一层
2. 现在这台机器上哪些已经可用
3. 还缺什么前置条件

---

## 1. 结论先看

当前最实用的替代关系：

- `web_fetch` 的主要替代：`Firecrawl`
- 内置 `browser` 的主要替代：`agent-browser`
- 搜索补充层：`Tavily`
- 摘要压缩层：`Summarize`

当前机器上的状态：

- `Firecrawl`：已安装、已写入 `FIRECRAWL_API_KEY`、`firecrawl_search` / `firecrawl_scrape` 已验证可用
- `agent-browser`：已安装、运行时已安装、已验证 `open / get title / snapshot / close` 可用
- `Tavily`：已安装、已写入 `TAVILY_API_KEY`、搜索与部分链接抽取可用
- `Summarize`：已安装并可调用，但是否稳定可用仍取决于你给它的模型 API key 和目标页面

---

## 2. Firecrawl

Skill：

- `firecrawl-search`

本机已接好的命令：

- `firecrawl_search`
- `firecrawl_scrape`
- `firecrawl_crawl`

你这次给的 `FIRECRAWL_API_KEY`，在当前 skill 里直接对应：

- 搜索：`firecrawl_search`
- 单页正文抓取：`firecrawl_scrape`
- 站点级 crawl：`firecrawl_crawl`

本机已验证：

- `firecrawl_scrape https://example.com`：成功
- `firecrawl_search "OpenAI API platform" --limit 3`：成功
- `firecrawl_scrape https://openai.com/api/`：成功

本机已看到的限制：

- `firecrawl_scrape https://www.stats.gov.cn/...`：返回 `SCRAPE_TIMEOUT`
- 说明它能绕过本机 Clash fake-ip 对内置 `web_fetch` 的 SSRF 拦截，但不代表所有站点都能稳定抓取

官方能力认知：

- 官方文档入口：[Firecrawl Docs](https://docs.firecrawl.dev/introduction?view=api)
- 官方搜索接口说明：[Search | Firecrawl Docs](https://docs.firecrawl.dev/features/search)
- 从官方入口和当前 skill 看，`Firecrawl` 至少覆盖了：
  - 搜索
  - 单页抓取 / scrape
  - 站点 crawl
- 官方文档首页说明同一套 API key 可用于其 API 体系；当前这个 skill 只封装了 `search / scrape / crawl`，不是把 Firecrawl 全部能力都暴露出来了

对实验的意义：

- 它是当前最像 `web_fetch` 替代层的插件
- 默认优先拿它做“读链接正文”

---

## 3. agent-browser

Skill：

- `agent-browser`

本机已接好的命令：

- `agent-browser`

本机已验证：

- `agent-browser open https://example.com`
- `agent-browser get title`
- `agent-browser snapshot -i`
- `agent-browser close`

验证结果：

- `example.com` 可以正常打开并返回标题与交互元素
- `github.com/googleworkspace/cli` 可以正常打开并返回页面标题
- `pcworld` 某篇文章曾出现 `page.goto timeout 25000ms exceeded`

官方能力认知：

- 官方 skill 页面：`agent-browser`
- 官方 CLI 仓库：[vercel-labs/agent-browser](https://github.com/vercel-labs/agent-browser)
- 当前 skill 和 CLI 能力覆盖：
  - 打开页面
  - 快照交互元素
  - click / fill / press / scroll
  - get title / url / text / html / attr
  - 截图 / PDF / 录屏
  - session / cookies / storage / network / trace

对实验的意义：

- 它是当前最像内置 `browser` 的替代层
- 当内置 `browser` 报 `Failed to start Chrome CDP on port 18800` 时，直接切它，不必继续空转

## 4. Tavily

Skill：

- `tavily-search`
- `openclaw-tavily-search`

本机已接好的命令：

- `tavily_search`
- `tavily_extract`

本机已验证：

- `tavily_search "OpenAI latest API docs" -n 3`：成功
- `tavily_search "site:stats.gov.cn 工业增加值" -n 3`：成功
- `tavily_extract https://openai.com/api/`：成功
- `tavily_extract https://example.com`：失败

官方能力认知：

- 官方文档入口：[Tavily Docs](https://docs.tavily.com/documentation/quickstart)
- 官方概览页说明 Tavily API 不止有搜索，还包括：
  - Search
  - Extract
  - Crawl
  - Map
  - Research
- 当前安装的两个 skill 里：
  - `tavily-search` 更直接，带 `search` 和 `extract`
  - `openclaw-tavily-search` 更偏兼容 `web_search` 风格输出

对实验的意义：

- 它不是 `web_fetch` 的首选替代
- 更适合作为发现层补充，或在某些链接上做轻量正文提取

## 5. Summarize

Skill：

- `summarize`

本机已接好的命令：

- `summarize`

本机已验证：

- `summarize --help`：可调用
- `summarize https://example.com --extract ...`：本机这次返回 `fetch failed`

官方能力认知：

- 官网：[summarize.sh](https://summarize.sh/)
- 当前 CLI 帮助和官网说明显示它可处理：
  - URL
  - 本地文件
  - PDF
  - 图片
  - 音频
  - YouTube
- 它还支持：
  - `--extract`
  - `--firecrawl auto|always`
  - `--model ...`
  - 多家模型提供商 key

对实验的意义：

- 它不是发现层，也不是主抓取层
- 更适合放在“抓到正文之后”的压缩总结步骤
- 如果后面要把 Firecrawl 抽到的长正文压短，它很有用

---

## 6. 当前推荐顺序

如果只看这台机器今天的可用性，建议按这个顺序走：

1. 发现层：优先 `Brave / Kimi / Tavily`
2. 读正文：优先 `Firecrawl`
3. 页面交互 / 动态站点：`agent-browser`
4. 长文压缩：`Summarize`

不要再把下面两件事当成今天的默认链路：

- 内置 `web_fetch`
- 内置 `browser`

因为这两层在本机仍然分别受：

- Clash fake-ip / SSRF block
- Chrome CDP 启动失败

---

## 7. 一句话判断

今天这台机器上，真正已经进入“可拿来做实验”的 fallback 组合是：

`Tavily / Brave / Kimi 负责发现 -> Firecrawl 负责读链接 -> agent-browser 负责复杂页面 -> Summarize 负责压缩整理`
