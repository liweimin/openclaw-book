# 搜索能力测评执行手册

这份手册是“搜索能力测评实验”的第一轮执行说明。

目标不是一下测完所有 provider，而是先建立一套统一评测方法，然后用 3 到 4 组组合跑出第一轮结论。

## 1. 第一轮建议测哪些组合

建议先测这 4 组：

1. Brave + web_fetch
2. Perplexity Search API + web_fetch
3. Gemini grounding + web_fetch
4. web_fetch vs browser（同一来源对比）

如果第一轮顺利，再进第二轮：

- Tavily
- Exa
- Firecrawl extract
- blogwatcher
- coding agent 深研究协同

## 2. 第一轮统一测试题材

为了避免“题材不同导致结果不可比”，第一轮先固定两个测试题材：

### 2.1 行业题材

- 调味品行业最近 7 天的重要变化

### 2.2 技术题材

- OpenClaw 最近 7 天的重要动态

这样可以同时看：

- 中文消费行业资料
- 英文/技术产品资料

## 3. 每次测试都记录什么

每跑一组，都按同样格式记录：

1. 检索组合
2. 输入问题
3. 实际返回形态
4. 来源质量
5. 结果质量
6. 可解释性
7. 速度感受
8. 成本感受
9. 最适合的使用场景
10. 是否适合默认启用

## 4. 统一评分维度

建议每项按 1 到 5 分打分：

- 来源发现能力
- 结果结构化程度
- 来源透明度
- 中文资料友好度
- 时间新鲜度控制
- 域名 / 来源控制
- 可解释性
- 成本效率
- 适合作为默认模式的程度

## 5. 第一轮怎么提问

### 5.1 发现层测试问题

对 Brave / Perplexity / Gemini 统一用类似问题：

```text
请查找最近 7 天中国调味品行业最值得关注的变化，优先给出来源链接，不要直接长篇总结。
```

对 OpenClaw 统一用：

```text
请查找最近 7 天 OpenClaw 最值得关注的产品、文档或社区动态，优先给出来源链接，不要直接长篇总结。
```

重点看：

- 返回的是链接清单，还是综合答案
- 是否带引用
- 结果是否足够新

### 5.2 取证层测试问题

针对同一条来源链接，分别用：

- web_fetch
- browser

去读取，然后比较：

- 谁读得更完整
- 谁更稳定
- 哪类页面 fetch 不够，必须上 browser

## 6. 第一轮结论应该怎么写

每一组最后都写成这 4 段：

1. 这组组合最适合什么任务
2. 这组组合最大的问题是什么
3. 这组组合是否适合默认启用
4. 如果让一把手看研究结果，这组组合够不够可信

## 7. 第一轮的预期结论方向

这只是预期，不是最终结论。

第一轮大概率会回答下面这些问题：

- Brave 更适不适合做“发现层默认模式”
- Perplexity 更适合快速综合，还是更适合直接做主搜索
- Gemini grounding 对行业研究是不是太偏答案型
- fetch 在哪些页面已经够用
- browser 在哪些来源上是必须的

## 8. 测完之后要产出什么

第一轮测完，至少应该产出：

1. 一张能力对照表
2. 一版默认检索模式建议
3. 一版“研究结果里的来源说明模板”
4. 一版“哪些场景要升级到外部工具或 coding agent”的建议

## 9. 第二轮再考虑什么

只有第一轮结论稳定后，再进入第二轮去看：

- Tavily 是否值得接入
- Exa 是否值得接入
- Firecrawl 是否值得接入
- blogwatcher 是否值得用于持续跟踪
- coding agent 是否值得作为深研究增强层
