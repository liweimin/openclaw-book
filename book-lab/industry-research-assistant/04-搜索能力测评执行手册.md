# 搜索能力测评执行手册

这份手册不是“概念说明”，而是第一轮可以直接照着做的操作手册。

它的目标很明确：

- 不先讨论哪个行业研究结论更准
- 先把 OpenClaw 的搜索栈怎么搭配更合适，做出第一轮对照结论

如果你是第一次做这条实验，建议先把它理解成：

**同一个问题，换不同搜索组合，看看谁更适合做行业研究助手的默认检索模式。**

---

## 1. 这份手册最终要帮你得到什么

你做完第一轮，至少应该得到下面 4 个结果：

1. 哪个组合更适合做“来源发现层默认模式”
2. 哪个组合更像“快速综合答案模式”
3. `web_fetch` 什么时候已经够用，什么时候必须上 `browser`
4. 行业研究助手第一版默认应该先启用哪一套搜索栈

这轮实验不是为了“把所有 provider 都测完”，而是为了先选出：

- 第一版默认值
- 第一版不该默认打开的能力

---

## 2. 先记住这次实验的基本原则

### 原则 1：同一轮只测一个变量

不要一轮里同时改：

- provider
- 提示词结构
- 题材
- 输出要求

否则最后你不知道差异是怎么来的。

### 原则 2：先看来源，再看总结

行业研究最怕的不是“它不会总结”，而是：

- 找错来源
- 没读来源
- 黑盒综合

所以第一轮重点不是文风，而是：

- 找到什么
- 读到了什么
- 能不能解释来源为什么可信

### 原则 3：每组尽量开新会话

建议每一组测试都：

- 在飞书里先 `/new`

这样更干净，避免上一组上下文污染下一组。

---

## 3. 开始前先准备什么

## 3.1 先决定：这轮实验单独用一个飞书机器人 + 一个实验 agent

这次手册的唯一推荐方案是：

- `weekly-assistant` 继续使用你现在正在测试的飞书机器人（例如 `accountId: "main"`）
- 搜索测评单独新建一个飞书机器人（例如 `accountId: "eval"`）
- OpenClaw 里新增一个独立 agent：`industry-search-eval`

这样做的好处非常直接：

- 不影响 `weekly-assistant` 的正常测试
- 路由最简单：按 `accountId` 直接分给不同 agent
- 可复现：实验对话、工作区、配置都独立
- 后续最好迁移：测评结论可以直接沉淀给正式的“调研助手”

### 推荐结构

- 飞书机器人 A：`main`
  - 继续服务 `weekly-assistant`
- 飞书机器人 B：`eval`
  - 只服务 `industry-search-eval`

### 第一步：创建实验 agent（WSL）

```bash
openclaw agents add industry-search-eval --workspace ~/.openclaw/workspace-industry-search-eval
```

### 第二步：在飞书开放平台创建第二个机器人

这一步不要复用 `weekly-assistant` 当前那只机器人，直接新建一只专门给搜索测评用。

按官方飞书接入步骤做：

1. 在飞书开放平台创建一个新的企业自建应用
2. 记录新的 `App ID` 和 `App Secret`
3. 在应用能力里开启机器人能力
4. 在事件订阅里启用长连接，并添加 `im.message.receive_v1`
5. 创建版本并发布应用

建议把这只机器人命名成容易识别的名字，例如：

- `OpenClaw-Search-Eval`
- `行业搜索测评助手`

### 第三步：把第二只机器人接到 OpenClaw

第二只机器人这一步，**最稳的做法是直接改** `~/.openclaw/openclaw.json`。

如果你确认本机的 `openclaw channels add` 已支持继续追加第二个 Feishu account，也可以走交互式方式；但这份手册以下都按“直接改配置文件”来写。

推荐配置示例：

```json5
{
  agents: {
    list: [
      { id: "main", default: true },
      {
        id: "weekly-assistant",
        workspace: "~/.openclaw/workspace-weekly-assistant",
      },
      {
        id: "industry-search-eval",
        workspace: "~/.openclaw/workspace-industry-search-eval",
      },
    ],
  },

  bindings: [
    {
      agentId: "weekly-assistant",
      match: { channel: "feishu", accountId: "main" },
    },
    {
      agentId: "industry-search-eval",
      match: { channel: "feishu", accountId: "eval" },
    },
  ],

  channels: {
    feishu: {
      defaultAccount: "main",
      dmPolicy: "pairing",
      accounts: {
        main: {
          appId: "cli_xxx_main",
          appSecret: "xxx_main",
          botName: "weekly-assistant",
        },
        eval: {
          appId: "cli_xxx_eval",
          appSecret: "xxx_eval",
          botName: "industry-search-eval",
        },
      },
    },
  },
}
```

这里最关键的是 3 个点：

- `channels.feishu.accounts.eval` 是第二只机器人
- `bindings` 里用 `accountId: "eval"` 把它固定路由到 `industry-search-eval`
- `defaultAccount` 继续保留 `main`，这样原来的 `weekly-assistant` 不会被扰动

### 第四步：给实验 agent 打开最小可用工具集

这份实验至少要用到：

- `web_search`：发现来源
- `web_fetch`：读取原文
- `browser`：只在对比 `web_fetch vs browser` 或遇到复杂页面时使用

推荐起步（示意）：

```json5
{
  agents: {
    list: [
      {
        id: "industry-search-eval",
        tools: { profile: "messaging", alsoAllow: ["web_search", "web_fetch", "browser"] }
      }
    ]
  }
}
```

### 第五步：重启 Gateway，让新账号和新路由生效

```bash
openclaw gateway restart
openclaw gateway status
openclaw channels status --probe
```

你至少要确认：

1. Gateway 正常
2. 两个 Feishu account 都加载成功
3. 没有明显的凭证或连接错误

### 第六步：分别验收两只机器人，确认没有串路由

先测原来的 `weekly-assistant` 机器人：

1. 在飞书里私聊原机器人，发 `ping`
2. 终端检查：

```bash
openclaw sessions --agent weekly-assistant
openclaw sessions --agent industry-search-eval
```

验收标准：

- `weekly-assistant` 出现新 session
- `industry-search-eval` 不因为这条消息新增 session

再测新的 `industry-search-eval` 机器人：

1. 在飞书里私聊新的测评机器人，发 `ping`
2. 如果触发 pairing，就执行：

```bash
openclaw pairing list feishu
openclaw pairing approve feishu <CODE>
```

3. 再次发送 `ping`
4. 终端检查：

```bash
openclaw sessions --agent industry-search-eval
openclaw sessions --agent weekly-assistant
```

验收标准：

- `industry-search-eval` 出现新 session
- `weekly-assistant` 不因为这条消息新增 session

只要这一步通过，后面的搜索测评就能放心在新机器人里做，`weekly-assistant` 可以继续正常使用。

### 第七步：固定实验入口，不再混用 `main`

从这里开始，这份手册后面所有测试动作都默认在：

- 飞书里的第二只机器人 `eval`
- OpenClaw 里的 `industry-search-eval`

不要再用 `main` 跑这份实验，不然后面的记录会失去可复现性。

## 3.2 先确认你到底要测哪几组

第一轮只建议测这 4 组：

1. Brave + `web_fetch`
2. Kimi + `web_fetch`
3. Gemini grounding + `web_fetch`
4. `web_fetch` vs `browser`（同一来源对比）

如果你是第一天开始测，建议不要一口气做完 4 组。  
最稳的顺序是：

- 第一天：先做 1 和 2
- 第二天：再做 3 和 4

## 3.3 先确认当前 OpenClaw 至少满足这些条件（在新机器人已接通的前提下）

建议在 WSL 里先确认：

```bash
openclaw gateway status
openclaw channels status --probe
```

你至少要满足：

1. Gateway 正常
2. 飞书渠道正常
3. 你已经能和自己的实验 agent 稳定对话

## 3.4 先确认搜索 provider 是否已经配好

这一步非常重要。  
如果 provider 没配好，你后面测到的不是“搜索能力差异”，而只是“哪家没配成功”。

建议先确认：

- Brave 对应 `BRAVE_API_KEY`
- Kimi 对应 `KIMI_API_KEY` 或 `MOONSHOT_API_KEY`
- Gemini 对应 `GEMINI_API_KEY`

如果你还没配，先不要急着测。  
先把最小搜索配置补顺。

最简单的配置入口通常是：

```bash
openclaw configure --section web
```

如果你当前是在 Windows + WSL 里跑 OpenClaw，而 `openclaw` 不在 PATH，也可以直接用：

```bash
~/.npm-global/bin/openclaw configure --section web
```

### 3.4.1 切换 provider 最省事的方式

这轮实验里，最怕的是“你以为自己切到了 Kimi / Gemini，实际上还在用 Brave”。

如果你已经按本手册前面的步骤把实验目录建在当前仓库里，可以直接用这个脚本切换：

`D:\00容器\openclaw\book-lab\industry-research-assistant\switch-web-search-provider.ps1`

示例：

```powershell
powershell -File D:\00容器\openclaw\book-lab\industry-research-assistant\switch-web-search-provider.ps1 brave
powershell -File D:\00容器\openclaw\book-lab\industry-research-assistant\switch-web-search-provider.ps1 kimi -KimiBaseUrl https://api.moonshot.cn/v1
powershell -File D:\00容器\openclaw\book-lab\industry-research-assistant\switch-web-search-provider.ps1 gemini
```

这个脚本会做两件事：

- 修改 `~/.openclaw/openclaw.json` 里的 `tools.web.search.provider`
- 自动重启 Gateway

每次切完 provider，不要直接开始正式实验，先跑下面的 smoke test。

### 3.4.2 Kimi 的 baseUrl 不要想当然

Kimi 在 OpenClaw 里虽然叫 `provider: "kimi"`，但底层还是走 Moonshot 接口。

至少要分清两个端点：

- 国际端点：`https://api.moonshot.ai/v1`
- 中国端点：`https://api.moonshot.cn/v1`

**2026-03-10 本机实测：**

- `https://api.moonshot.cn/v1/chat/completions` 可返回 `200`，并能触发 `$web_search` 的 `tool_calls`
- `https://api.moonshot.ai/v1/chat/completions` 在当前网络下报 `SSL EOF`

这意味着：

- 不能看到 `.ai` 报 TLS，就直接下结论说“Kimi 不可用”
- 如果你当前机器是中国网络、或者用的是 Moonshot 中国侧 key，先优先确认 `.cn`
- 记录时必须把 **Kimi 的 `baseUrl`** 一并写进 smoke test 证据，不要只写“Kimi 失败”

### 3.4.3 smoke test 前先跑本目录诊断脚本

如果你当前是 Windows + WSL，或者本机开了 Clash / TUN，建议先跑：

```powershell
powershell -File D:\00容器\openclaw\book-lab\industry-research-assistant\diagnose-search-stack.ps1
```

它至少会帮你看到 4 件事：

1. 当前 `web_search provider`
2. 当前 `kimi.baseUrl`
3. WSL 里目标域名是不是被解析成 `198.18.x.x`
4. 最近的 `browser` / `web_fetch` 关键错误

建议这次在飞书里：

说明：

- `/new`：保证每组尽量干净


## 3.6 正式 A/B 前必须先跑 10 分钟 smoke test

这一步一定不要省。

如果 smoke test 没过，你后面记录出来的大部分“优劣”都不是真正的能力差异，而是：

- 路由没切对
- provider 根本没连通
- `web_fetch` 被安全策略拦住
- `browser` 根本起不来

### 3.6.1 先测路由有没有打到实验 agent

在飞书里给新的 `eval` 机器人发一条：

```text
ping
```

然后在 WSL 里看：

```bash
~/.npm-global/bin/openclaw sessions --agent industry-search-eval
```

如果你想看得更死一点，就打开：

```bash
cat ~/.openclaw/agents/industry-search-eval/sessions/sessions.json
```

你至少要确认：

- 有新的 session
- `deliveryContext.accountId` 是 `eval`
- 没有串到 `weekly-assistant`

### 3.6.2 再测 provider 锁定是否生效

在飞书里先发一句最短确认：

```text
只回答一句：你当前会把 web_search 默认 provider 当成什么？如果不确定就直说。
```

通过标准：

- 它明确回答当前 provider
- 回答和你刚才切换的一致

如果这里答不出来，后面整轮实验都先不要跑。

### 3.6.3 再测 provider 本身能不能通

每切一组 provider，建议先做一次最短查询：

```text
【实验控制项】本轮 web_search provider = Brave（或 Kimi / Gemini）。
如果你无法确保 web_search 使用该 provider，请直接说明并停止本轮。

请只做来源发现，列出 3 个最近 7 天值得读的来源，给标题和链接即可。
```

这一步你只看 3 件事：

1. 有没有真正返回结果
2. 返回的是“来源列表”还是“答案型综合”
3. 有没有明显报错

判定规则：

- 有结果，可以进入正式实验
- API/TLS/认证失败，记为“环境失败”，这一组不纳入能力优劣
- 没报错但结果明显跑偏，才记为“能力问题”

这里再补一条硬要求：

- 不要只听 agent 自己说“我现在用的是 Kimi / Brave”
- 要尽量在 `verbose` 输出或 `session jsonl` 里看到真正的 provider 证据
- 如果本轮测的是 Kimi，再把 `kimi.baseUrl` 一并记下

### 3.6.4 再测 `web_fetch` 有没有最小可用性

先不要挑复杂站点，先挑一个普通文章页，发：

```text
请不要搜索，只直接读取这个页面：

<这里贴链接>

优先用轻量方式读取内容，然后告诉我：
1. 你有没有成功读到正文
2. 你读到的核心信息是什么
3. 哪些信息可能没有取到
```

通过标准：

- 它明确说出有没有读到正文
- 如果失败，能把失败原因说出来
- 不能把“没读到正文”硬说成已经读到了

如果这一关连续两次都失败，这一轮就不要急着做 `web_fetch` vs `browser` 对比，先把失败原因单独记下来。

如果你本机用了 **Clash + TUN**，这一步还要额外看一件事：

- 目标域名是不是被解析成了 `198.18.0.0/15` 里的 fake-ip

只要解析结果落到这类地址，OpenClaw 的 `web_fetch` 很容易返回：

`Blocked: resolves to private/internal/special-use IP address`

注意：

- 这是 **站点 / 安全策略限制**
- 不是“这个网页真的在内网”
- 也不是只补 `FIRECRAWL_API_KEY` 就能解决

原因很简单：

- 内置 `web_fetch` 遇到这类 SSRF block 会直接退出
- 不会自动继续走内置 Firecrawl fallback

这不是拍脑袋判断，源码里能直接对上两层机制：

- `198.18.0.0/15` 被当成 RFC2544 benchmark / special-use 网段处理
- `web_fetch` 捕获到 `SsrFBlockedError` 时会直接抛出，不会进入后面的 Firecrawl fallback

所以只要你继续保留 Clash fake-ip，这个问题通常不是“重试几次”能解决的。

所以这时你只有两条路：

1. 先修 Clash 的 fake-ip / DNS 策略
2. 或者把取证层改成独立的 Firecrawl skill，不再依赖内置 `web_fetch`

### 3.6.5 最后测 `browser` 环境是否可用

先看状态：

```bash
~/.npm-global/bin/openclaw browser status --json
```

再尝试启动：

```bash
~/.npm-global/bin/openclaw browser start
```

通过标准：

- `running = true`
- `cdpReady = true`

如果 `browser` 起不来，就先不要做 `fetch vs browser` 的优劣比较，只记录：

- 这台机器当前 browser 没打通
- 这轮只能先做 provider 层面对比

如果你当前是 **Gateway 在 WSL、Chrome 在 Windows**，还要补一条显式检查：

```bash
curl http://WINDOWS_HOST_OR_IP:9222/json/version
```

只有当 WSL 能访问到 Windows Chrome 的 CDP 端口，`browser` 才值得继续测。  
如果这一步不通，就不要再把问题归因为：

- `browser.noSandbox`
- 提示词
- 页面太复杂

而应该直接记成：

- **WSL -> Windows CDP 链路未打通**

再补一个很容易忽略的点：

- OpenClaw 在 WSL 里跑 `openclaw` managed profile 时，会给浏览器带上 Linux 侧的 `--user-data-dir`
- 如果你实际用的是 Windows 的 `chrome.exe` / `msedge.exe`，这条链路经常不如“Windows 侧独立开 CDP，再让 WSL 远程挂过去”稳定
- 这时更推荐两种模式：
  - **remote CDP profile**：Windows 侧手动启动 Chrome / Edge 的独立调试实例，WSL 用 `cdpUrl` 连接
  - **chrome extension relay**：直接改走内置 `chrome` profile，并按需设置 `browser.relayBindHost`

换句话说，**WSL 调 Windows 浏览器** 这件事不是完全不能用，但比“Linux 网关 + Linux 浏览器”或者“WSL 网关 + Windows 远程 CDP”更容易踩坑。  
如果 Chrome 老是有问题，换 Edge 也可以，但重点不是品牌，而是：

- 是否能稳定暴露 CDP 端口
- WSL 是否真的能访问到那个端口
- 你是不是在用独立调试实例，而不是把参数丢给已有日常浏览器会话

### 3.6.6 失败归因一定要分清

你后面记录时，必须把失败归因写成下面 4 类之一：

1. **能力问题**
   - 例如：有结果，但来源明显很差，或结构化很弱
2. **环境失败**
   - 例如：TLS、认证、网络不可达、Gateway 未生效
3. **站点 / 安全策略限制**
   - 例如：`web_fetch` 返回 `Blocked: resolves to private/internal/special-use IP address`
4. **实验未完成**
   - 例如：`browser` 起不来，因此 `fetch vs browser` 暂时无法比较

只有第 1 类，才能拿来讨论“哪家更适合做默认模式”。

这样不会一下子信息太多。

## 5. 每一组测试都按同样的 3 步走

每组都不要自由发挥，统一按下面 3 步：

1. 发现层测试
2. 取证层测试
3. 记录与评分

只有这样，后面横向对比才有意义。

---

## 6. 第一步：发现层测试怎么做

这一轮的目标不是让它长篇总结，而是看它：

- 找到了哪些来源
- 来源结构是不是清楚
- 来源是否足够新

## 6.1 给 agent 的统一提示词

以“调味品行业”为例，先发下面这段：

```text
【实验控制项】本轮 web_search provider = Brave（或 Kimi / Gemini grounding；每组只改这一行）。
如果你无法确保 web_search 使用该 provider，请直接说明并停止本轮。

请查找最近 7 天中国调味品行业最值得关注的变化。

这次先不要长篇总结，也不要直接给行业判断。
请先只做“来源发现”这一步，要求如下：

1. 给我 5 到 8 个你认为最值得读的来源
2. 每个来源都给出：
   - 标题
   - 链接
   - 来源类型
   - 为什么值得读
3. 来源类型只能用下面几类：
   - 官方 / 协会 / 监管
   - 公司官网 / IR / 财报
   - 行业媒体
   - 综合媒体
4. 优先最近 7 天
5. 不要把搜索结果伪装成研究结论
6. 最后补一句：你觉得这组来源像“发现层”还是已经接近“结论层”
```

如果你要测 OpenClaw，把提示词里这一行改成：

```text
请查找最近 7 天 OpenClaw 最值得关注的产品、文档或社区动态。
```

其他保持不变。

## 6.2 这一步你重点看什么

你不用急着判定“答案好不好看”，先只看下面 6 件事：

1. 返回的是来源清单，还是一坨综合答案
2. 有没有给出链接
3. 来源类型分得清不清楚
4. 是否能说明“为什么这个来源值得读”
5. 结果是否足够新
6. 有没有明显垃圾来源

## 6.3 这一轮常见好表现

好表现通常像这样：

- 结果列表清楚
- 大多数来源都能解释为什么值得读
- 来源类型比较合理
- 没有太多低质量聚合站

## 6.4 这一轮常见坏表现

坏表现通常像这样：

- 一上来就写综合结论
- 不给链接
- 来源很旧
- 看不出为什么选它
- 很多结果其实不是这个题材的核心来源

---

## 7. 第二步：取证层测试怎么做

这一轮的目标是看：

- 找到来源以后，它能不能把来源真正读下来
- `web_fetch` 和 `browser` 哪个更适合当前页面

## 7.1 先从刚才的来源里选 2 到 3 个

不要多。  
第一轮建议只选：

- 1 个行业媒体链接
- 1 个公司官网或官方来源链接

如果页面复杂，再加第 3 个。

## 7.2 给 agent 的统一提示词

先发：

```text
现在请读取你刚才列出的前 2 个来源，基于原文做一个小结。

要求：

1. 先明确写出你实际读取了哪些链接
2. 每条结论后尽量标出对应来源
3. 区分“确定事实”和“趋势判断”
4. 不要空泛总结
5. 如果某个来源页面你没有真正读到，请直接说明
```

## 7.3 如果你要专门测 `web_fetch` vs `browser`

找一个你怀疑 `web_fetch` 不够稳的页面，然后分两轮发。

### 先测 `web_fetch`

```text
请不要搜索，只直接读取这个页面：

<这里贴链接>

优先用轻量方式读取内容，然后告诉我：
1. 你有没有成功读到正文
2. 你读到的正文是否完整
3. 哪些信息可能没有取到
```

### 再测 `browser`

重新 `/new`，再发：

```text
请不要搜索，只直接打开这个页面并读取正文：

<这里贴同一个链接>

如果轻量抓取不够，请改用 browser。
然后告诉我：
1. 你最后是怎么读到内容的
2. 内容是否比轻量抓取更完整
3. 如果仍然不完整，缺在哪里
```

## 7.4 这一轮你重点看什么

重点只看下面 5 件事：

1. 它有没有真的读页面，而不是只复述搜索摘要
2. 能不能明确说出“我读了哪些链接”
3. 事实和判断有没有分开
4. `web_fetch` 是否已经够用
5. 页面是否真的需要上 `browser`

## 9. 第一天最推荐的实操顺序

下面这组动作，默认都在飞书里的第二只机器人 `eval` 中进行。

如果你今天就要开始，建议按这个顺序，不要跳。

## 9.1 先测 Brave + `web_fetch`

原因：

- 更像基础发现层
- 结果通常更结构化
- 适合先建立比较基线

步骤：

1. `/new`
2. 发“发现层提示词”
3. 选 2 个链接
4. 发“取证层提示词”
5. 记录评分

## 9.2 再测 Kimi + `web_fetch`

原因：

- 它更适合测试中文场景下“答案型搜索”的表现
- 最容易看出和 Brave 的风格差异

步骤和上一组完全相同，不要改提示词结构。

## 9.3 第一天做到这里就可以停

不要第一天就把 Gemini、browser 对比、Tavily、Exa、Perplexity 全测了。  
先把前两组做扎实，结论反而更干净。

## 9.4 2026-03-10 本机第一轮实测样例

下面这段不是理论步骤，而是本机已经跑过的一轮样例，目的是给你一个“什么叫有效测试记录”的参照。

### 9.4.1 本轮测试目的

- 确认第二个飞书机器人 `eval` 是否稳定路由到 `industry-search-eval`
- 确认 Brave / Kimi 在当前机器上到底哪些能用
- 确认 `web_fetch` 和 `browser` 当前能不能进入可比较状态

### 9.4.2 本轮测试方法

按下面顺序做的：

1. 检查 Gateway 和 Feishu channel 状态
2. 检查 `industry-search-eval` 的 session 归属
3. 确认默认 `web_search provider = brave`
4. 读取 `industry-search-eval` 的 session jsonl，看 Brave 这一轮实际搜到了什么
5. 按 OpenClaw 的 Kimi 请求结构，分别直连 `api.moonshot.cn` 和 `api.moonshot.ai`
6. 用两个真实 URL 做 `web_fetch` 取证 smoke test
7. 检查 WSL / Windows 下目标域名的 DNS 解析
8. 运行 `openclaw browser open` 并查看 Gateway 日志
9. 手动拉起 Windows Chrome 远程调试，再看 WSL 能不能访问 CDP

### 9.4.3 本轮测试结果

#### A. 路由与入口

- `Feishu eval` 和 `Feishu main` 都正常
- `industry-search-eval` 的 session 已写入 `~/.openclaw/agents/industry-search-eval/sessions/sessions.json`
- 该 session 的 `deliveryContext.accountId = "eval"`，说明新机器人确实打到了实验 agent

#### B. Provider 层

- **Brave：可用**
  - `industry-search-eval` 的 session jsonl 里能看到 `provider: "brave"`
  - 通过 agent 跑“来源发现”时，实际搜到过国家统计局、36Kr 等候选来源
  - 当前更适合作为“发现层基线”
- **Kimi：当前可用，但端点不能配错**
  - `https://api.moonshot.cn/v1/chat/completions` 返回 `200`
  - 同样的请求打到 `https://api.moonshot.ai/v1/chat/completions` 在当前网络下报 `SSL EOF`
  - 所以这台机器上，Kimi 不能简单记成“不可用”，而应该记成：
    - `.cn` 可用
    - `.ai` 当前网络不通
    - 需要把 `baseUrl` 一起纳入 smoke test 记录

#### C. `web_fetch` 层

- 用国家统计局页面做直读测试，失败
- 用 36Kr 页面做直读测试，仍失败
- 失败信息一致：`Blocked: resolves to private/internal/special-use IP address`
- WSL / Windows 下这些域名都被解析成了 `198.18.0.x`
- 这与 Clash TUN 的 fake-ip 行为一致

这说明当前机器上的 `web_fetch` 至少在本轮测试 URL 上还不能作为稳定取证基线。  
而且这里要特别注意：

- 失败根因不是网页正文本身有问题
- 根因是 **fake-ip -> SSRF block**
- 这类错误不会自动落到内置 Firecrawl fallback

#### D. `browser` 层

- `openclaw browser open https://example.com` 返回 `Failed to start Chrome CDP on port 18800 for profile "openclaw"`
- 手动拉起 Windows Chrome 并打开 `--remote-debugging-port=9222` 后，Chrome 自己会打印 `DevTools listening`
- 但 WSL 里仍然无法访问 `127.0.0.1:9222`，访问 Windows 默认网关地址也超时

所以当前这台机器还不能做有效的 `web_fetch vs browser` 能力比较。  
这时真正的归因应当写成：

- **WSL -> Windows CDP 链路未打通**

### 9.4.4 本轮结论怎么用

这轮真正有用的结论不是“谁赢了”，而是先把可测边界定清楚：

1. **现在可以做的有效比较**
   - Brave vs Kimi 在 provider 层的发现风格差异
   - 前提是你明确记录了 provider 证据和 Kimi 的 `baseUrl`
2. **现在不该做的比较**
   - `web_fetch` vs `browser` 的优劣
3. **给后续调研助手的初步建议**
   - 默认 discovery 候选先保留 Brave
   - Kimi 可以保留为中文“答案型搜索”候选，但这台机器先固定 `tools.web.search.kimi.baseUrl = https://api.moonshot.cn/v1`
   - `web_fetch` / `browser` 要先把环境打通，再决定默认升级策略
   - 如果内置 `web_fetch` / `browser` 继续被环境卡住，就用独立技能兜底，而不是把这轮失败误记成能力差

## 10. 这次实验最容易犯的错误

### 错误 1：一开始就看“总结写得漂不漂亮”

这轮重点不是文风，是：

- 来源
- 取证
- 可解释性

### 错误 2：上一组刚聊完，不 `/new` 就直接测下一组

这样容易把上一组结果带进下一组。

### 错误 3：每组提示词都不一样

提示词一变，后面很难横向比较。

### 错误 4：只看搜索结果，不继续看它有没有真正读原文

行业研究最怕这一点。

### 错误 5：不做记录，最后只剩感觉

第一轮对照实验，记录比发挥更重要。

## 12. 第一轮测完之后，下一步再做什么

### 12.1 如果你要做后续的“调研助手”，建议先把实验与长期使用拆开

建议你保留两只 agent：

- `industry-search-eval`：只负责 A/B 测评、对照、记录与复盘
- `industry-research-assistant`（未来）：只负责长期行业调研与输出

这样做的好处是：

- 实验不会污染长期调研的会话与偏好
- 你可以在实验里大胆改提示词、换 provider，而不影响日常
- 当实验结论稳定后，只需要把“默认检索模式”迁移给调研助手

这次测评完成后，你至少应该固化 3 件事（后面写进调研助手的 `AGENTS.md` / Skill 配置里）：

1. 默认检索模式（默认 provider + `web_search -> web_fetch`，以及何时升级 `browser`）
2. 输出必须透明：每次都列出实际读取的 URL 清单，并区分事实与判断
3. 用户可控入口：支持用户补充“特别关注的网站/链接/域名”，并在输出里说明是否覆盖到了这些来源

第一轮结论稳定后，再进入第二轮看：

- Tavily 是否值得接入
- Exa 是否值得接入
- Firecrawl 是否值得接入
- blogwatcher 是否适合持续跟踪
- coding agent 是否值得作为深研究增强层

### 12.2 如果内置栈持续不通，先用 ClawHub 做环境兜底

这不是第一轮默认栈。  
但如果你当前机器已经明确碰到了：

- Clash TUN fake-ip，导致内置 `web_fetch` 被 SSRF block
- WSL -> Windows CDP 不通，导致内置 `browser` 起不来

那就不要继续硬耗内置链路，先用一套独立技能把实验跑起来：

- **发现层**：Tavily Search
- **取证层**：Firecrawl
- **总结层**：Summarize
- **浏览层**：agent browser

2026-03-10 本机在 ClawHub 搜索时，能搜到的候选 slug 包括：

- `firecrawl-search`
- `openclaw-tavily-search`
- `summarize`
- `agent-browser`

注意：

- 这里先把它们当成 **候选 slug**
- 这一轮不要把“内置栈”和“ClawHub fallback 栈”混在同一组里比
- 如果 ClawHub 接口报 `Rate limit exceeded`，就先记成“环境限制”，稍后重试

最小动作可以先记住这 4 条命令：

```bash
~/.npm-global/bin/clawhub search firecrawl --limit 10
~/.npm-global/bin/clawhub search tavily --limit 10
~/.npm-global/bin/clawhub search summarize --limit 10
~/.npm-global/bin/clawhub search "agent browser" --limit 10
```

换句话说：

**第一轮先选默认栈，第二轮再谈增强层。**

## 13. 一句话总结

这份手册真正想帮你做到的，不是“测很多”，而是“测得能比较、能复用、能做决定”。

最稳的做法永远是：

1. 固定题目
2. 固定步骤
3. 固定评分
4. 一次只改一个变量

这样你最后拿到的，就不是一堆零散体验，而是一套能真正指导行业研究助手默认配置的实验结论。
