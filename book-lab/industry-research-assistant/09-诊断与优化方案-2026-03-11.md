## 三、诊断发现的 8 个核心问题

### 问题 1：Fallback Skill 从未被实际触发 🔴

> [!CAUTION]
> SKILL.md 花了大量篇幅定义了兜底顺序（Tavily → Firecrawl → agent-browser → Summarize），但今日session中**没有一次fallback skill调用**。

**原因分析：**
- `industry-research-assistant` 没有 `skills/` 目录 — 意味着 Tavily/Firecrawl/agent-browser 这些 skill **根本没有挂载到这个 agent 上**
- 即使 SKILL.md 里写了兜底策略，agent 在运行时也无法调用未安装的 skill
- 今日 web_fetch 恰好全部成功（0 error），所以没暴露问题，但**一旦遇到 SSRF block 或站点超时，就会直接失败而无法兜底**

**建议修复：**
```bash
# 把 fallback skill 挂载到 industry-research-assistant
openclaw skills install tavily-search --agent industry-research-assistant
openclaw skills install firecrawl-search --agent industry-research-assistant
openclaw skills install agent-browser --agent industry-research-assistant
openclaw skills install summarize --agent industry-research-assistant
```

### 问题 2：Agent 缺少 persona.md 和独立配置 🔴

**当前状态：**  
`~/.openclaw/agents/industry-research-assistant/` 目录下**没有 `persona.md`**、**没有 `config.json`**、**没有 `skills/`** 目录。

**影响：**
- Agent 没有身份定义，使用全局默认人格，无法保证"先追问再研究"的行为一致性
- 没有 agent 级别的工具白名单，可能调用到不需要的工具
- 用户实际上是把 SKILL.md 设计好了，但没有真正安装到 agent 运行时

**建议修复：**
- 从 [SKILL.md](file:///d:/00容器/openclaw/book-lab/industry-research-assistant/skill/SKILL.md) 中提取核心指令写入 `persona.md`
- 为 agent 设置 `tools.profile: messaging` + `alsoAllow: [web_search, web_fetch, browser]`
- 把本项目的 `skill/` 目录正式安装为 agent skill

### 问题 3：内置 browser 持续不可用 🟡

**当前状态：**
- `browser.enabled = true`，但 `cdpPort: 18800` 始终无法启动
- `remote-edge` profile 配置指向 `172.18.240.1:9224`，但无法确认 Windows 侧是否已启动独立 Edge 调试实例
- 第一轮测试(03-10)记录：`running=false, cdpReady=false, Failed to start Chrome CDP on port 18800`

**影响：**
- 需要浏览器交互的复杂页面（动态渲染、分页、登录态）无法处理
- SKILL.md 中 `browser` 被定义为 `web_search → web_fetch → browser` 的第三层升级路径，但这条路径断了

**建议修复（选一）：**
1. **WSL 本地 Chrome**：确认 Playwright Chromium 是否已安装成功 + `--no-sandbox` 是否生效
2. **Remote Edge**：Windows 侧手动启动 `msedge.exe --remote-debugging-port=9224 --user-data-dir=C:\temp\edge-openclaw`，WSL 侧确认 `curl http://172.18.240.1:9224/json/version` 可通
3. **改用 agent-browser**：放弃内置 browser，默认改走 `agent-browser` skill（前提是先解决问题1）

---

### 问题 4：Session 复用导致上下文污染 🟡

**当前状态：**
- 今日 session `b8e342bb` 被复用处理了多个不同课题（调味品、财务培训、海天差异化…）
- 65 行记录里包含了多轮不同方向的研究
- 第一轮测试结论(08文件)已明确指出：*"主 session 会复用历史上下文，做 provider 对比时容易被旧记忆污染"*

**影响：**
- 模型在后续轮次中可能受到前几轮结论的隐性影响
- 无法准确归因"这次研究结论好不好"是模型能力还是上下文残留

**建议修复：**
- 每次新课题在飞书里先发 `/new` 开新 session
- 考虑在 SKILL.md 中加一条："如果是全新研究课题，建议先 /new"
- 或者在 persona.md 中写入"每次收到全新行业/赛道请求时，主动建议用户开新会话"

---

### 问题 5：memory_search 使用频率过低 🟡

**当前状态：**
- 今日 65 行 session 中，`memory_search` 仅被调用 **1 次**
- 工作区已有 `research-profile.md`、`research-history.md`、`priority-sources.md` 等文件

**影响：**
- Agent 每次研究时不会主动检索历史研究结论和偏好
- 重复课题不会自动利用之前已验证的来源和结论
- 持续跟踪场景下无法做"与上期相比的新变化"

**建议修复：**
- 在 persona.md 或 SKILL.md 中增加明确指令："在开始任何研究前，先 memory_search 检索是否有该行业/赛道的历史研究画像和来源清单"
- 检查 `memorySearch` 配置是否完整（当前 openclaw.json 中**未看到 `memorySearch` 配置**）
- 建议补上：
```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",
        model: "text-embedding-3-small"
      }
    }
  }
}
```

### 问题 6：web_search provider 切换缺乏运行时证据验证 🟡

**当前状态：**
- 当前 provider 为 `brave`，今日运行正常
- 但 SKILL.md / 执行手册中定义了还需要支持 Kimi / Gemini 测评对比
- 第一轮测试中提到"不要只听 agent 自己说用的是什么 provider，要在 session jsonl 中看到真正的 provider 证据"

**影响：**
- 如果后续切换 provider 做对比，没有自动化验证机制确认是否真正生效

**建议修复：**
- 在 smoke test 流程中加入脚本化验证，自动从 session jsonl 提取 `provider` 字段
- 建议在 `diagnose-search-stack.ps1` 中增加一步：读取最新 session 的 provider 证据

---

### 问题 7：Kimi/Gemini 搜索参数兼容性问题 🟢

**已知状态（来自08测试结论）：**
- Kimi：`country / language / freshness` 参数均报 `unsupported_*`
- Gemini：`country / language` 报 `unsupported_*`
- Kimi 更像"答案型搜索"，不适合做发现层

**建议处理：**
- SKILL.md 的搜索执行规则中，增加 provider 特性说明
- 在选择 provider 时，直接告诉模型各 provider 的参数支持范围
- Kimi 定位为"中文答案型补充"，Gemini 定位为"带 citations 的综述补充"

---

### 问题 8：输出结构与来源透明度在实际执行中不够稳定 🟢

**今日观察：**
- 部分研究结果（如财务培训材料）更像"综合分析报告"而非"来源发现 + 取证 + 结论"的标准格式
- 12个输出文件中，并非所有都包含了 SKILL.md 要求的"参考来源"段

**建议修复：**
- 在 persona.md 中强化：每次输出必须包含"实际搜索词 + 实际来源链接 + 使用工具"
- 考虑创建一个输出模板文件，放在 workspace 中让 agent 每次参考

## 六、关键风险提示

> [!WARNING]
> **最大的隐患不是"今天测试出了bug"，而是"今天测试全部成功，但实际上兜底机制根本没有被验证"。**

今日 web_fetch 19次调用全部成功、0个错误，这让整个 fallback 链路从未被触发。一旦遇到：
- Clash TUN 下某站点被解析到 `198.18.x.x` → web_fetch SSRF block
- 某站点反爬/JS 渲染 → web_fetch 拿不到正文
- 需要浏览器操作 → browser 仍然 down

Agent 将**没有任何 fallback 能力**，只能原地失败。

## 七、一句话总结

**这个 agent 的设计文档（SKILL.md + 检索与配置说明）做得非常扎实，但从"设计"到"运行时"之间还差了一小步关键的安装和配置动作。今天最重要的事，是把 4 个 fallback skill 挂载上去、补一份 persona.md、配好 memorySearch。做完这三件事，这个 agent 的实际可靠性会从"天气好的时候能跑"升级到"大部分情况都能跑"。**
