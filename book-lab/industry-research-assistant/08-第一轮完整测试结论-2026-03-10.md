# 第一轮完整测试结论（2026-03-10）

---

## 1. 这轮到底测了什么

```text
时间：2026-03-10
机器：Windows + WSL + Clash TUN
实验对象：
- 内置 web_search provider：Brave / Kimi / Gemini
- 内置取证工具：web_fetch / browser
- fallback skill：Tavily / Firecrawl / agent-browser / summarize

说明：
- 用户口中的 “traive” 在当前手册、配置、已装 skill 里都没有对应对象。
- 本结论按 “Tavily” 纳入 fallback 搜索对比。
```

---

## 2. 一句话总结

```text
这台机器当前最稳的默认栈应当是：

Brave（发现层） -> web_fetch（默认取证） -> agent-browser（需要浏览器时升级）

同时保留：
- Firecrawl：web_fetch 的备用直读器
- Tavily：Brave 的备用搜索
- summarize：取证后的快速摘要器

不建议把 Kimi / Gemini 设成第一默认 discovery provider。
不建议继续依赖内置 browser。
```

---

## 3. 分项结论

### 3.1 内置搜索 provider

| 组别 | 当前状态 | 这轮实际表现 | 结论 |
|---|---|---|---|
| Brave | 可用 | 能稳定给出来源列表；对 `country / language / freshness` 这类控制项支持最好；结果更像“发现层” | **默认 discovery 首选** |
| Kimi | 可用 | 在 clean agent 的 session jsonl 中已看到 `provider: "kimi"`；但 `country / language / freshness` 都报 `unsupported_*`；返回内容更像无明确引用的答案块 | **不适合默认发现层，更适合中文答案型补充** |
| Gemini | 可用 | 在 clean agent 的 session jsonl 中已看到 `provider: "gemini"`；`country / language` 也报 `unsupported_*`；会返回带 citations 的长答案，但更像“资料综述”而不是“最近该读哪几条” | **可做补充，不建议压过 Brave 成为默认 discovery** |

### 3.2 内置取证工具

| 工具 | 当前状态 | 这轮实际表现 | 结论 |
|---|---|---|---|
| web_fetch | 可用 | 复测已成功读取国家统计局和 36Kr 正文，返回 `status: 200` | **恢复为可用，默认先用它取证** |
| browser | 不可用 | `browser status --json` 显示 `running=false`、`cdpReady=false`；`openclaw browser open https://example.com` 返回 `HTTP 404` | **当前不适合作为默认升级路径** |

### 3.3 fallback 技能

| 工具 | 当前状态 | 这轮实际表现 | 结论 |
|---|---|---|---|
| Tavily Search | 可用 | `tavily_search 'OpenAI latest API docs' -n 3` 成功 | **Brave 的备用搜索** |
| Tavily Extract | 可用 | `tavily_extract https://openai.com/api/` 成功 | **可做轻量直读备份** |
| Firecrawl | 可用 | `firecrawl_scrape https://openai.com/api/` 成功 | **web_fetch 的备用直读器** |
| agent-browser | 可用 | 初次冷启动报 daemon socket 错误，重试后 `open -> get title -> close` 成功 | **内置 browser 的实际替代品** |
| summarize | 可用 | `summarize https://openai.com/api/ --extract` 成功 | **适合取证后的快速摘要，不适合代替原文取证** |

## 4. 这轮最关键的修正

```text
1. 旧样例里“web_fetch 因 fake-ip / SSRF 全面不可用”的结论，已不再适用于当前状态。
   - 当前复测时，虽然 WSL 里 stats.gov.cn / 36kr.com 仍解析到 198.18.x.x，
     但 web_fetch 仍然成功返回了正文。
   - 这说明：之前那条失败结论现在不能继续当成默认前提。

2. 内置 browser 仍然没有恢复。
   - 所以这轮真正需要 fallback 的重点，已经从 web_fetch 转成了 browser。

3. Kimi / Gemini 不是“不能用”，而是“不适合当第一默认 discovery”。
   - Kimi：答案味重，缺少稳定可核验来源。
   - Gemini：有 citations，但更像综合回答，不像筛来源。
```

## 5. 默认配置建议

```text
默认发现层：
- Brave

默认取证层：
- web_fetch

浏览器升级路径：
- agent-browser

备用搜索：
- Tavily

备用直读：
- Firecrawl

快速摘要：
- summarize

Kimi 的定位：
- 只保留为中文“答案型搜索/补问”候选

Gemini 的定位：
- 只保留为“带 citations 的补充综述”候选
```

## 6. 对调研助手的最终结论

```text
如果今天就要给调研助手定默认栈，我会这样定：

1. 默认搜索：Brave
2. 默认读链接：web_fetch
3. 遇到强交互 / 需要真实浏览器渲染：直接走 agent-browser
4. 如果 web_fetch 某站读不稳：切 Firecrawl
5. 如果 Brave 某轮搜得不理想：切 Tavily 做补搜
6. summarize 只放在最后做压缩总结，不参与原始证据判定
```

## 7. 还剩下的风险

```text
- `industry-search-eval` 主 session 会复用历史上下文，做 provider 对比时容易被旧记忆污染。
- 因此本轮 Kimi / Gemini 的 provider 证据，是用临时 clean agent 拿到的。
- 内置 browser 还没真正修好，所以“browser 对 web_fetch 的能力优势”这题暂时仍不建议继续深挖。
```
