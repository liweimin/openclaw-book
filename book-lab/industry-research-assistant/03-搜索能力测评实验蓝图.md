# 搜索能力测评实验蓝图

这份蓝图属于“行业研究助手”的子实验。

它不直接解决“研究哪个行业”，而是先回答下面这个更基础的问题：

`OpenClaw 在做行业研究时，到底该用什么搜索栈，怎样搭配才更可信、更可解释、更省资源。`

## 1. 为什么先单独做这个子实验

行业研究助手的核心不是“会搜”，而是：

- 能不能找到对的来源
- 能不能把来源读下来
- 能不能说明这次结论是怎么来的
- 能不能让用户知道自己现在用的是哪一套搜索能力

如果不把搜索能力拆出来单独验证，后面会把这些问题混在一起：

- 是研究简报没问清楚？
- 是搜索 provider 不行？
- 是 fetch / browser 读源方式不对？
- 是 skill 触发规则不对？
- 是模型总结质量不稳定？

所以这条子实验的价值是：

- 给行业研究主线打底
- 建立一套可解释的检索模式
- 后面让用户自己选，而不是黑盒

## 2. 这条子实验到底要测什么

先不测“某个行业结论准不准”，而是先测下面 6 件事：

1. 发现来源的能力
2. 读取原文的能力
3. 来源可解释性
4. 结果结构化程度
5. 时间新鲜度与过滤能力
6. 成本 / 速度 / 资源消耗

## 3. OpenClaw 当前内置的搜索相关能力

基于官方文档，这条子实验第一批优先测这些内置能力：

### 3.1 web_search

官方当前支持的 provider：

- Brave
- Gemini
- Grok
- Kimi
- Perplexity

其中可以先重点测：

- Brave
- Kimi
- Gemini grounding

原因：

- Brave 更像结构化搜索发现层
- Kimi 更适合测试中文场景下的“答案型搜索”路线
- Gemini 适合测试“答案型搜索”路线

### 3.2 web_fetch

作用：

- 把 URL 真正抓下来
- 做可读内容提取
- 作为搜索结果之后的原文读取层

### 3.3 browser

作用：

- 处理 JS 重、动态页、需要真实页面阅读的场景
- 用于验证“只靠 fetch 不够”的页面

### 3.4 Firecrawl fallback

官方 `web_fetch` 支持 Firecrawl 作为提取后备层。

这更适合归类为：

- 抽取增强
- 不是主发现层

### 3.5 ClawHub / skills

`ClawHub` 不是搜索引擎，而是 OpenClaw 的技能注册表。

这条子实验里，它的作用应该是：

- 找可安装的研究类 skill
- 看哪些 skill 值得接入研究工作流

而不是把它当成搜索 provider 本身。

## 4. 候选外部增强能力

这些不是第一轮就全接，而是进入候选池：

- Tavily
- Exa
- Firecrawl（作为 crawl / extract 增强）
- blogwatcher 这类持续跟踪 skill
- coding agent（Codex / Claude / Gemini）

这里要分清：

- Tavily / Exa：更像搜索与研究 API
- Firecrawl：更像抽取与 crawl 层
- blogwatcher：更像持续跟踪层
- coding agent：更像深研究执行层

## 5. 这条子实验的评测对象怎么分层

建议把研究搜索栈拆成 4 层来测：

### 5.1 发现层

找候选来源：

- Brave
- Kimi
- Gemini grounding
- Tavily
- Exa

### 5.2 取证层

把来源真正读下来：

- web_fetch
- browser
- Firecrawl extract

### 5.3 编排层

决定何时用什么：

- 行业研究助手 Skill
- AGENTS / MEMORY 规则
- 研究简报字段

### 5.4 增强层

用于深研究或持续跟踪：

- blogwatcher
- coding agent
- cron

## 6. 每个能力要怎么测

建议每个候选能力都按同一张表来测：

1. 适合发现还是适合取证
2. 返回结果是结构化链接，还是直接综合答案
3. 能否限制时间范围
4. 能否限制域名 / 来源
5. 能否给出清晰引用
6. 对中文行业资料是否友好
7. 速度如何
8. 成本如何
9. 是否容易向用户解释
10. 是否适合默认启用

## 7. 这条子实验的核心产出

这条子实验最终应该产出 3 个东西：

### 7.1 检索模式表

例如：

- 标准模式
- 综合答案模式
- 深研究模式
- 重度自动化模式

### 7.2 来源说明模板

每次研究输出时，说明：

- 本次用了哪套搜索栈
- 哪些来源类型是优先级最高的
- 这次实际用了哪些来源
- 哪些结论是事实，哪些是趋势判断

### 7.3 默认推荐策略

不是所有工具全开，而是：

- 默认开哪些
- 哪些按需开
- 哪些只在重研究时再上

## 8. 第一轮最值得先测什么

为了省资源，第一轮不要全测。

建议先测这 4 组：

1. Brave + web_fetch
2. Kimi + web_fetch
3. Gemini grounding + web_fetch
4. web_fetch vs browser（同一来源对比）

这 4 组已经足够回答很多关键问题：

- 搜索发现层谁更稳
- 中文答案型 provider 会不会太黑盒
- fetch 什么时候够用，什么时候必须上 browser

如果你后面补上了 Perplexity Search API，再把它作为第二轮补测项加进来，会更合适。

## 9. 这条子实验怎么服务主实验

主线行业研究助手后面不要直接硬编码某个 provider。

更好的做法是：

- 先做完这条子实验
- 再让主线研究助手支持“检索模式选择”
- 并在输出时说明“本次使用的检索模式与来源”

这样后面用户才会信，也更容易调优。
