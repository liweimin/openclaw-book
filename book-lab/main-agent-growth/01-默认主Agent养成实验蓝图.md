# 默认主 Agent 养成实验蓝图

这份蓝图不是正式书稿章节，而是“默认主 Agent 能不能先养起来”这条实验线的设计稿。

它要回答的是：

- 大多数人是不是应该先从 `main` 开始，而不是先建很多 agent
- 默认主 Agent 能通过哪些日常互动慢慢适应你
- 哪些东西只靠聊天不够，必须显式写进工作区
- 什么时候 `main` 应该继续养，什么时候应该拆分

## 1. 这条实验要做成什么

这条实验想做的不是：

- 一个复杂的新 agent
- 一套高度定制的规则系统

而是：

**验证默认主 Agent 在“尽量少改配置”的前提下，能不能先成为一个日常可用的个人助手。**

这条线真正想证明的是：

1. 很多高频小任务，本来就可以先交给 `main`
2. 日常连续对话本身就是一种“养成”
3. 不是所有偏好都要一开始写成规则文件
4. 只有当上下文开始变脏、任务开始分化时，拆 agent 才真正有价值

## 2. 为什么这条线很重要

现在很多 OpenClaw 资料天然会滑向一种写法：

- 先建 agent
- 先写 `AGENTS.md`
- 先拆工作区
- 先做 bindings

这对实验复现很友好，但不是很多人第一次真实使用的起点。

更真实的起点往往是：

1. 先把 OpenClaw 装好
2. 先在默认主 Agent 里连续对话
3. 先拿几个高频任务反复磨
4. 发现边界后，再决定要不要拆专用助手

所以这条实验线的价值，不是反对多 agent，而是把顺序放回真实使用方式。

## 3. 这条实验默认遵守什么原则

第一版默认遵守下面 6 条原则：

1. 先用默认主 Agent，不先新建 agent
2. 尽量不改复杂配置
3. 先靠日常任务和纠偏建立合作方式
4. 只有长期稳定偏好，才考虑写进 `MEMORY.md`
5. 先不引入 Skill、cron、复杂渠道编排
6. 先观察 `main` 的真实边界，再谈拆分

## 4. 这条线要验证哪些问题

第一版先验证 5 件事：

1. `main` 能不能稳定处理高频轻任务
2. 连续对话能不能让它更贴近你的表达和输出习惯
3. `/new` 之后，哪些东西还能保留，哪些不会
4. 补一版最小 `MEMORY.md` 之后，体验会不会明显变稳
5. 到什么程度，`main` 会开始变成“什么都沾一点，但不够清晰”

## 5. 适合拿来养 `main` 的任务类型

这条线适合优先用下面这些任务来验证：

- 把零散记录整理成清晰清单
- 会议纪要整理与行动项抽取
- 帮你起草飞书/邮件/私聊回复
- 让它总结今天做了什么
- 让它根据本地文件做轻量摘要
- 把一个模糊想法整理成可执行的下一步

这些任务有一个共同点：

- 高频
- 低风险
- 容易反复纠偏
- 很适合通过连续对话慢慢磨合

## 6. 不适合一开始就塞进 `main` 的任务

第一版不建议直接拿来养 `main` 的，是这些任务：

- 多人共享场景
- 高敏感财务资料
- 明确需要长期独立工作区的项目
- 很重的网页研究
- 很多 cron 与提醒并存的长期自动化

这些不是完全不能用 `main` 做，而是更容易把边界搅乱。

## 7. 推荐的实验路线

### 阶段 1：先验证“连续对话能不能形成配合”

这一阶段先不追求“记住很多事”，只看：

- 它会不会越来越懂你的表达
- 你纠偏两三次后，它会不会收敛

### 阶段 2：再验证“跨天之后还剩下什么”

这一阶段重点看：

- 跨天继续聊时，哪些习惯会自然延续
- 哪些内容不写下来就会越来越飘

### 阶段 3：补最小显式记忆

这时候只补：

- 输出语言
- 风格偏好
- 长期固定背景
- 一两条稳定工作方式

不要一上来把所有日记、项目、短期任务都塞进去。

### 阶段 4：判断是否该拆分

这一步不再问“能不能继续用”，而是问：

- `main` 是不是已经同时承载太多异质任务
- 某些任务是不是已经值得单独 agent + 单独 workspace

## 8. 推荐的最小产物

做完这条实验，至少应该得到下面 3 个产物：

1. 一份 7 天使用记录
2. 一份“适合留在 `main` 的任务类型清单”
3. 一份“什么时候该拆专用 agent”的判断规则

## 9. 第一轮成功标准

第一轮实验通过，至少要满足：

- 你已经连续几天把高频小任务交给 `main`
- 你能明确感受到纠偏后的输出在收敛
- 你能区分哪些内容靠连续对话就够，哪些必须显式沉淀
- 你能说清楚：为什么某些任务还不应该急着拆 agent
- 你也能说清楚：什么时候继续把东西都塞进 `main` 已经不划算

## 10. 这条线后面怎么服务书稿

这条实验线最适合沉淀成书里的一个独立主张：

> OpenClaw 的起步不一定是“先做 agent 工程”，更常见也更自然的起点是：先把默认主 Agent 养顺手，再根据任务边界逐步拆出专用助手。

后面如果这条线验证稳定，它很适合写成：

- 跑通之后的第一条实战线
- `weekly-assistant` 之前的过渡章
- “单 Agent 养成”与“多 Agent 分工”的分界章
