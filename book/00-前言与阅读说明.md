# 前言与阅读说明

## 为什么会有这本书

OpenClaw 本身并不简单。

它不是一个单纯的网页聊天工具，也不是一个只会写代码的 Agent 壳子。  
它同时涉及：

- Gateway
- Agent
- Tools
- Skills
- Memory
- Channels
- Nodes
- 配置、配对、安全与长期运行

这意味着它天然比“打开网页直接聊天”的产品复杂。  
真正的问题从来不是“OpenClaw 为什么这么复杂”，而是“学习顺序如果不对，就会显得特别乱”。

原来这份资料库里有很多有价值的内容，但它们分散在几十份文档中：

- 有些偏安装
- 有些偏 Windows
- 有些偏飞书
- 有些偏源码
- 有些偏技能和工具
- 有些偏最佳实践

单篇看都不一定错，但放在一起就会让初学者非常容易失去方向。  
这本书的目标，就是把那些散落材料重组为一条可以顺着走完的学习路径。

## 这本书想解决什么问题

它主要解决 5 个层次的问题：

### 第 1 层：认知问题

OpenClaw 到底是什么？  
它和普通聊天工具、代码 Agent、消息机器人平台分别是什么关系？

### 第 2 层：上手问题

怎么装？  
Windows 到底该不该原生跑？  
第一次成功的标准到底是什么？

### 第 3 层：日常使用问题

配置文件怎么理解？  
常用命令怎么记？  
出问题先看什么？

### 第 4 层：系统理解问题

Gateway 为什么是中枢？  
Agent 到底是怎么运转的？  
为什么 Tools、Skills、Memory 会一起出现？

### 第 5 层：长期使用问题

怎么把 OpenClaw 从“装起来了”变成“长期稳定可用”？  
怎么走向多渠道、飞书、自动化、持续使用和后续扩展？

## 这本书适合谁

### 适合

- 第一次接触 OpenClaw 的中文用户
- 已经装好了，但被分散文档搞乱的人

## 本书的写法原则

### 原则 1：先建立正确认知，再进入真实场景

很多人不是不会用，而是对它的定位理解错了。

### 原则 2：先让系统跑通、接上入口、完成场景，再谈复杂原理

如果连 Dashboard、Gateway、基础对话都没通，后面很多高级能力都是噪声。

### 原则 2.3：第一次安装尽量自己走完，后续再让 Agent 辅助

首次安装阶段最重要的目标，是建立对自己环境的基本判断能力。

因此本书建议：

- 第一次安装 WSL2、Ubuntu、Node、OpenClaw 时，尽量自己按步骤完成
- 不要一上来就把整个安装过程外包给 coding agent
- 等系统已经装好后，再让 Agent 帮你做配置、排错、写 Skill、整理输出和跑场景

这样做的好处是很直接的：

- 你会知道问题到底出在 Windows、WSL2、Node 还是 OpenClaw
- 后面让 Agent 介入时，你更容易判断它给出的建议是否合理
- 长期使用时，你不会把最基础的环境控制权完全交出去

### 原则 2.5：每章尽量都要留下一个可见产物

这本书不是按“知识点罗列”写的，而是按“读完这一章你手里会多出什么”来写的。

所以你会在主章节里反复看到几类固定东西：

- 本章实操场景
- 本章产物
- 本章验收标准

这样做的目的，是让整本书更像教材，而不是资料堆。

### 原则 3：关键事实尽量以官方源码和官网为准

尤其是这些最容易过时的内容：

- Node 版本要求
- Windows 支持策略
- onboarding / dashboard / gateway 的推荐入口
- 默认端口与默认运行方式

### 原则 4：原始资料保留，主线重新组织

这本书不是把旧文档删掉，而是把它们收成一条主线。  
所以你后面做 PDF、EPUB 或商业版时，仍然有完整素材库可回溯。

## 本书的术语约定

为了保证整本书的口径一致，先约定几个最常用词：

### Gateway

指 OpenClaw 的长期运行中枢和控制面服务。

### Agent

指真正执行推理、决策和工具调用流程的运行实体。

### Tools

指 Agent 可以调用的执行能力。

### Skills

指给 Agent 的可复用任务套路和行为说明。

### Memory

指长期记忆层，不等于单轮会话上下文。

### Channels

指 WhatsApp、Telegram、Discord、Feishu 等消息入口。

### Nodes

指额外接入的设备能力或节点端。

如果你后面需要更正式的术语整理，可以继续看：
