# 第一章：OpenClaw 到底是什么，以及你应该怎么学它

## 1. 为什么要先写这一章

很多人第一次接触 OpenClaw 时，最大的困难不是“不会安装”，而是“根本没搞清楚自己在装什么”。

你可能会看到这些互相冲突的说法：

- 它是个人助手。
- 它是代码 Agent。
- 它是多渠道聊天机器人。
- 它是一个本地 AI 中枢。
- 它像 Claude Code，但又不完全像。

这些说法都不算完全错，但如果没有一个统一框架，你会越看越乱。  
所以这本书的第一章，不直接讲命令，而是先解决认知问题。

## 2. 一句话定义 OpenClaw

OpenClaw 是一个运行在你自己设备上的个人 AI 助手框架。  
它通过一个长期运行的 Gateway，把模型、Agent、Tools、Skills、Memory、Channels、Nodes 和控制界面组织成一个整体。

如果你想用更通俗的话理解它，可以把它看成：

- 前台像聊天助手
- 中台像控制总线
- 后台像能调工具、能连设备、能接消息渠道的 AI 工作系统

## 3. 它不是什么

### 它不是单独一个模型

OpenClaw 自己不是 GPT、Claude 或 Gemini 这种模型。  
它是运行框架，模型只是它接入的大脑来源之一。

### 它不是只有网页聊天页

Dashboard 很重要，但它只是控制界面。  
真正的核心是 Gateway，而不是网页本身。

### 它不是只给程序员用的代码壳

虽然它有很强的终端、代码和工具能力，但官方定位明显更宽。  
它可以是：

- 聊天入口
- 消息中枢
- 工具执行器
- 自动化协调器
- 个人助手底座

### 它也不是“装了就自动变聪明”的魔法箱

OpenClaw 的能力上限很高，但使用效果高度依赖：

- 模型和认证方式
- 配置方式
- 工具和技能
- 工作区内容
- 渠道路由
- 你的使用习惯

## 4. 为什么它看起来比普通 AI 工具复杂

因为 OpenClaw 同时在做很多系统级事情。

普通网页聊天通常只做：

1. 接收文本
2. 调模型
3. 输出文本

而 OpenClaw 还要处理：

1. 网关连接
2. 会话管理
3. 设备与节点
4. 工具调用
5. 多渠道接入
6. 权限与配对
7. 状态、日志、健康检查
8. 配置和自动化

所以它的复杂度不是“文档写复杂了”，而是产品本身就是一个更完整的系统。

## 5. 小白应该如何建立正确心智模型

最容易理解的方式，是把 OpenClaw 想成四层结构。

### 第一层：入口层

这是你最先接触到的部分：

- `openclaw` CLI
- `openclaw dashboard`
- Web 控制台
- WhatsApp / Telegram / Discord / Feishu 等聊天入口

这一层解决的是“你从哪里跟它说话”。

### 第二层：Gateway 层

Gateway 是 OpenClaw 的中枢。

它负责：

- 接收连接
- 管理 WebSocket / HTTP 控制面
- 维护会话和状态
- 路由请求
- 统一安全与认证

这一层解决的是“所有东西是怎么串起来的”。

### 第三层：Agent Runtime 层

当你下达任务时，真正负责思考和执行流程的是 Agent Runtime。

它会：

- 读取上下文
- 选模型
- 拼提示词
- 决定是否调用工具
- 处理工具结果
- 输出最终回复

这一层解决的是“AI 到底是怎么工作的”。

### 第四层：能力层

这一层包括：

- Tools
- Skills
- Memory
- Browser
- Nodes
- Channels

这一层解决的是“它除了说话，还能做什么”。

## 6. 为什么 OpenClaw 容易被和 Claude Code / OpenCode 之类工具混淆

因为它们有明显重叠：

- 都会接模型
- 都会调用工具
- 都会做多轮任务执行
- 都会强调工作区和上下文

但差别也很明显：

### Claude Code / OpenCode / Kode 一类工具更偏什么

- 编程和终端任务
- 项目工作区内的局部闭环
- 面向开发者的交互体验

### OpenClaw 更偏什么

- 多入口、多渠道、多设备
- 长期运行的 Gateway
- 消息、会话和身份边界
- 个人助手式的持续使用场景

如果你把 OpenClaw 简单理解为“另一个代码 Agent”，你会低估它的渠道、网关、配对和长期运行特性。  
如果你把它简单理解为“聊天机器人平台”，你又会低估它的工具与执行能力。

## 7. 谁适合使用 OpenClaw

### 适合的人

- 想要一个长期运行的个人 AI 助手，而不是一次性网页聊天的人。
- 想把 AI 接到自己的聊天渠道、工作流或设备里的人。
- 接受“先有一点系统复杂度，后换来更大能力边界”的人。
- 愿意逐步配置和打磨自己使用环境的人。

### 不太适合的人

- 只想立即开箱得到一个零配置网页聊天页的人。
- 不愿意接触终端、配置文件和健康检查的人。
- 希望所有功能都通过单一按钮自动完成的人。

## 8. 小白最容易踩的认知误区

### 误区 1：先研究所有高级能力

一上来就研究：

- 多 Agent
- 多渠道
- Browser
- 自动化
- Nodes

往往只会更乱。

### 误区 2：把 Dashboard 当成本体

Dashboard 是很好用的入口，但本体是 Gateway。

### 误区 3：把“装上命令”当成“已经会用”

真正的入门成功标准应该是：

1. Gateway 正常
2. Dashboard 能打开
3. 能完成一次本地可用对话

### 误区 4：先信散落经验，再信官方源码

安装方式、版本要求、Windows 路线、命令参数这些内容最容易过时。  
这本书里凡是关键结论，都尽量以官方源码和官网文档为准。

## 11. 你现在要先记住的关键事实

这几个事实对后面整本书都非常重要：

- OpenClaw 官方定位是个人 AI 助手，不是单纯的代码工具。
- Windows 官方推荐通过 WSL2 使用 OpenClaw。
- 当前官方源码要求 Node `>=22.12.0`。
- 第一次最关键的命令是 `openclaw onboard --install-daemon`。
- 第一次最推荐先跑 `openclaw dashboard`，而不是先接一堆渠道。
- Gateway 默认本地入口是 `127.0.0.1:18789`。
