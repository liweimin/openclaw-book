# 第六章：稳定使用、排错与安全

## 1. 为什么把这几部分放在一起

因为当你真正开始长期使用 OpenClaw 后，最关心的问题往往不再是“它有没有这个功能”，而是：

- 出问题时怎么查
- 怎样用得更稳
- 哪些安全边界不能踩
- 它到底适合用在什么场景

这四件事看起来分散，实际上最后都指向同一个目标：让 OpenClaw 从“好像很厉害的项目”变成“你愿意长期保留的个人系统”。

所以这一章不是零碎附录，而是整本书主线的收束。

### 1.1 本章实操场景

这一章你应该完成的真实场景是：

想象一个真实情况：你已经在飞书里开始用 OpenClaw 整理计划或日报，结果某天早上它突然不回了。

这一章的实操目标，就是让你在这种情况下能在几分钟内判断问题落在哪一层，并留下自己的“自救基线文件”，以后排错不再靠猜。

### 1.2 本章产物

读完这一章，你手里至少应该有这两样东西：

- 一套固定的排错顺序
- 一份 `ch06-ops/ops-baseline.txt`

### 1.3 开始前先准备什么

这一章默认你已经至少跑通过一个真实入口或真实任务，例如：

- 在 Dashboard 里整理过文件
- 在飞书里拿到过一次真实回复

## 2. 调试的第一原则：先看可见性，再做猜测

很多人调 AI 系统，第一反应是猜：

- 是不是模型问题
- 是不是网络问题
- 是不是插件问题

但在 OpenClaw 里，更可靠的方式永远是先看可见性。你至少应该先掌握这些入口：

- `openclaw gateway status`
- `openclaw status`
- `openclaw health`
- `openclaw doctor`
- `openclaw logs`

如果这些入口你都不会用，再高阶的调试技巧意义都不大。因为你连系统现在处于什么状态都还没看清。

## 3. 新手最该先掌握的调试顺序

对大多数日常问题来说，最实用的顺序通常就是：

1. `openclaw gateway status`
2. `openclaw status`
3. `openclaw health`
4. `openclaw doctor`
5. `openclaw logs`

这套顺序看起来朴素，但非常有效。

### `gateway status`

先回答一个最根本的问题：中枢是不是活着。

### `status`

看整体运行状态，确认系统是不是处于可工作的基本姿态。

### `health`

看健康项和工作前提是否满足。

### `doctor`

在配置写错、环境缺失、依赖异常时，往往能更快地给出方向。

### `logs`

当你已经知道问题大致落在哪一层时，日志才是最有价值的细节来源。

很多人上来就追日志，结果会被海量信息淹没。先看状态，再看日志，效率高得多。

### 3.1 一个可以直接照着做的 60 秒排错模板

如果你现在就遇到“它不工作了”，不要慌，也不要立刻乱改配置。直接按下面这组命令走：

```bash
openclaw gateway status
openclaw status
openclaw health
openclaw doctor
openclaw logs --follow
```

你可以这样理解这 60 秒：

1. 前两条先判断系统是不是还活着
2. 第三条看工作前提是否满足
3. 第四条看有没有明显配置或环境错误
4. 最后一条才进入日志细节

如果你接了消息渠道，再额外补一条：

```bash
openclaw channels status --probe
```

这样你基本就能把问题先归到环境、Gateway、配置、渠道、Agent 这五层中的某一层。

### 3.2 现在就给自己做一份“自救基线文件”

这一步非常有用，因为你以后真的出故障时，可以直接拿新的状态去和今天的基线比较。

在 Ubuntu 里执行：

```bash
mkdir -p ~/.openclaw/workspace/ch06-ops
{
  echo "# OpenClaw 基线检查"
  date
  echo
  echo "## gateway status"
  openclaw gateway status
  echo
  echo "## status"
  openclaw status
  echo
  echo "## health"
  openclaw health
  echo
  echo "## doctor"
  openclaw doctor
} > ~/.openclaw/workspace/ch06-ops/ops-baseline.txt
```

然后再检查文件是否存在：

```bash
ls ~/.openclaw/workspace/ch06-ops
```

如果你看到了 `ops-baseline.txt`，这份文件以后就能拿来做两件事：

1. 当飞书、Dashboard 或工具调用异常时，和当前状态做对比
2. 当你改了配置、装了插件、升级了版本后，确认系统有没有被改坏

## 4. OpenClaw 的调试必须分层

OpenClaw 不是单点系统，所以调试也必须分层。只要一把所有问题混在一起，排错就会迅速失控。

### 第 1 层：环境层

检查：

- Node 版本对不对
- 安装是否完整
- PATH 是否正常
- CLI 是否能正确启动

如果这一层没过，后面的 Gateway、渠道和 Agent 问题都不必谈。

### 第 2 层：Gateway 层

检查：

- 端口是否监听
- Gateway 是否运行
- token / auth 是否正常
- 本地或远程连接是否可达

这一层决定了系统的“中枢”是否存在。

### 第 3 层：配置层

检查：

- `openclaw.json` 是否合法
- 键名、类型、层级是否正确
- 多个策略之间有没有冲突

很多问题表面像渠道不工作，实际是配置写错。

### 第 4 层：渠道层

检查：

- token / App Secret 是否正确
- 事件订阅是否生效
- pairing、allowlist、groupPolicy 是否挡住了消息

这一层最容易把“收不到消息”和“收到了但不处理”混为一谈。

### 第 5 层：Agent / Tool 层

检查：

- 模型是否真的可用
- 工具调用有没有被策略挡住
- 会话上下文是否正确
- Skill、Memory 或 Hook 是否影响了结果

只有这样分层，你调试时才不会越查越乱。

## 5. 四类最常见问题，应该怎么拆

### 问题一：Dashboard 打不开

优先检查：

- Gateway 是否启动
- 端口 `18789` 是否被占用
- 本机浏览器访问的是不是正确地址
- 有没有认证或 token 问题

这类问题通常还停留在 Gateway 层。

### 问题二：渠道收不到消息

优先检查：

- 平台凭证
- 插件是否安装
- 事件订阅是否生效
- Gateway 日志里有没有连接或认证错误

这类问题多数属于渠道层，而不是模型层。

### 问题三：能聊天，但不能执行任务

优先检查：

- 工具权限是不是过于保守
- 是否被 Hook、审批、allowlist 或 sandbox 挡住
- 模型是否真的支持你当前的工作方式

这类问题通常落在 Agent / Tool 层。

### 问题四：私聊正常，群里表现异常

优先检查：

- `groupPolicy`
- 群组 allowlist
- `requireMention`
- 当前发送者是否具备触发资格

这类问题最适合按“群是否允许”“消息是否满足触发条件”两步去拆。

## 6. 安全为什么必须单独强调

OpenClaw 和普通聊天网页不同，它能：

- 接消息
- 调工具
- 跑命令
- 读写文件
- 连设备

这意味着它的危险半径天然更大。只要进入真实使用场景，安全就不再是附加话题，而是系统设计的一部分。

你可以把安全理解成一句最朴素的话：任何能替你做事的 AI，也同样可能替别人做你不希望发生的事。

## 7. 小白最该先守住的安全边界

### 第一，不要轻易全开工具

尤其在你还没理解 Tool Access、sandbox、Hook、审批机制之前。

### 第二，保留 pairing 和 allowlist 思维

不要一上来就让任何人都能给你的助手发私信，或者在任何群里都能触发它。

### 第三，渠道接入时先最小权限

尤其是飞书、Discord、Slack 这类平台接入，不要第一次就把所有权限、所有群组、所有路由都放开。

### 第四，把配置错误也视为安全问题

错误配置不只是“不工作”，也可能让边界意外变松。能否解释清楚“为什么这条消息会被处理、为什么这个工具会被调用”，本身就是安全成熟度的一部分。

### 7.1 新手立刻就能做的四个安全动作

如果你现在还不知道怎么谈高级安全策略，那也没关系。先把下面四个动作做了，安全水平就会明显比“默认乱试”高很多。

第一，只先接一个入口。

不要同一天同时接飞书、Telegram、WhatsApp。入口越多，暴露面越大，排错也越乱。

第二，私聊先保留 pairing。

只要你还没完全想清楚谁可以直接触发你的助手，就先不要把私聊策略放得太松。

第三，群聊先从一个群开始。

而且第一次最好要求显式触发，例如 @ 提及，而不是让它在所有群里自由响应。

第四，每次改完配置都做一次最小检查。

```bash
openclaw doctor
openclaw status
```

如果这次改动涉及 Gateway、渠道或认证，再补：

```bash
openclaw gateway restart
```

这四个动作看起来朴素，但对长期使用来说非常关键。

## 8. 运行习惯决定你能不能长期用下去

很多人第一次装好 OpenClaw 后，会马上冲向各种功能演示。但真正能长期用下来的用户，往往都形成了一些很朴素的运行习惯。

### 先固定一个 workspace

别今天一个目录，明天一个目录。工作区一旦稳定，配置、技能、记忆和文件路径都会稳很多。

### 先固定一个主渠道或主入口

先让一个入口真正稳定，再考虑扩展更多平台。

### 每次只改一类东西

不要一次同时改模型、端口、工具权限、渠道凭证、群策略。那会让你根本无法定位问题。

### 每次改完都做最小检查

至少跑一遍：

- `openclaw doctor`
- `openclaw status`
- 必要时 `openclaw logs`

这样你会很快养成“改动后先验证”的肌肉记忆。

## 9. 怎样把“能玩”变成“稳定可用”

真正长期可用的 OpenClaw，通常具备这些特征：

1. 平台路线清晰
2. Gateway 稳定
3. 配置简单且能解释
4. 工具权限有边界
5. 状态检查和日志查看已形成习惯
6. 渠道只按需要扩展

换句话说，一个成熟的 OpenClaw 使用者，追求的不是“能不能让它做更多”，而是“我能不能始终知道它现在在做什么、为什么这么做、出了问题应该从哪一层查起”。

## 10. OpenClaw 最适合投入的几类场景

归档里的 Showcase 和玩法文档给了很多灵感，但如果要写成一本循序渐进的书，最值得推荐给新手的场景其实只有几类。

### 10.1 本地执行型助手

适合：

- 改代码
- 看日志
- 分析项目
- 跑命令

这是最容易建立成就感的一类场景，因为反馈快、闭环短。

### 10.2 消息入口型助手

适合：

- 在飞书、Telegram、WhatsApp 等渠道里完成日常交互
- 把提醒、查询、问答搬到你原本就常用的入口

这类场景的关键不是“炫技”，而是让 OpenClaw 真正进入日常使用。

### 10.3 资料处理型助手

适合：

- 读长文档
- 整理笔记
- 做资料归纳和检索

如果你愿意认真维护工作区和 Memory，这一类场景的长期收益很高。

### 10.4 长期个人系统型助手

适合：

- 维护长期偏好
- 连接多个入口
- 做长期持续的个人工作流

这一类场景最强大，但也最不适合第一天就上。它应该建立在前面三类已经跑稳的基础上。

## 11. 为什么 Showcase 值得看，但不该当主线

社区 Showcase 能告诉你上限，但不适合做入门主线。原因很简单：

- 它展示的是别人已经跑顺后的结果
- 它默认你已经具备安装、配置、调试和安全意识

所以更合理的用法是：

- 先用这本书建立主线
- 再去看 Showcase 找方向

这样你看到的是“我以后可以做到什么”，而不是“为什么别人能做到，我却根本不知道从哪开始”。

## 12. 本章小结

当你开始真正使用 OpenClaw，最重要的能力已经不是“会装”，而是：

- 会看状态
- 会分层调试
- 知道安全边界
- 知道哪些玩法值得投入，哪些只是短期热闹

做到这一点，OpenClaw 才会从“一个很酷的 AI 项目”变成“一个你愿意长期保留的个人系统”。

### 12.1 本章验收标准

做到下面这 5 条，这一章才算真正完成：

1. 你能按固定顺序跑 `gateway status -> status -> health -> doctor -> logs`
2. 你知道自己的核心配置和工作区该怎么备份
3. 你知道私聊、群聊、工具权限至少有哪些边界不能乱开
4. 你遇到问题时，已经不再只靠猜，而是先看状态
5. 工作区里已经留下一份 `ch06-ops/ops-baseline.txt`
