# OpenClaw 小白最佳实践：第一次安装后的配置、使用与长期迭代指南

本文面向这样一种场景：

- 你刚安装好 OpenClaw
- 还没怎么聊天
- 想把它从“能跑”变成“真的能帮你干活”
- 但又不想一上来把配置搞得很复杂

目标是给你一条清晰路线：

1. **第一次安装后先配什么**
2. **工作区里的关键 `.md` 文件该怎么组织**
3. **Tools、Skills、Memory 应该怎么用**
4. **平时怎么和 Agent 协作**
5. **每周/每月怎么迭代，让它越来越像你的助手**

这份文档是基于当前本地仓库 `D:\code\anzhuang\openclaw` 的文档和源码整理出来的最佳实践版，不是单纯翻译官方文档。

---

## 1. 先说最重要的原则

如果你是第一次安装 OpenClaw，请先记住下面 4 句话：

1. **先让它稳定可用，再慢慢调教，不要一开始就全开。**
2. **调教的核心不在仓库源码里，而在你的 workspace 和 config。**
3. **长期生效的东西要写到文件里，不要只靠聊天说一次。**
4. **先把 Agent 变成“可靠助理”，再把它变成“复杂自动化系统”。**

也就是说，小白第一次上手，不要先追求：

- 多 agent
- 复杂 hooks
- 一大堆 skills
- 很多渠道同时接入

先把一个主 Agent 养好，收益最大。

## 2. 第一次安装后，先完成什么

## 第一阶段目标：先跑通最小闭环

你第一次安装后，最小目标不是“什么都配完”，而是：

- Gateway 正常运行
- 你能在 Control UI 或一个渠道里稳定聊天
- workspace 已创建
- 你知道自己的配置文件在哪里
- 你知道哪些文件是以后要长期维护的

## 推荐步骤

### 第 1 步：完成安装和 onboarding

官方推荐流程：

```bash
openclaw onboard --install-daemon
```

如果已经安装完成，下一步确认：

```bash
openclaw gateway status
openclaw dashboard
```

你应该至少能做到：

- Gateway 正常启动
- 打开浏览器控制台聊天

### 第 2 步：确认 OpenClaw 的两个“用户地盘”

你以后最常用的不是源码仓库，而是这两个位置：

- 配置：`~/.openclaw/openclaw.json`
- 工作区：`~/.openclaw/workspace`

从官方 `docs/start/setup.md` 来看，真正属于“你自己的定制”的东西，应该尽量放在这里，而不是改源码仓库本身。

### 第 3 步：跑一次 workspace 初始化

```bash
openclaw setup
```

这一步通常会帮你把基础工作区文件准备好。

## 3. 第一次安装后，记忆需要先放内容吗

答案是：**需要，但不要一开始塞很多。**

你刚装好 OpenClaw 时，并不需要立刻写满 `MEMORY.md`。  
更好的做法是：

- 先写少量高价值、长期稳定的信息
- 不要把所有背景一次性灌进去

## 建议：初始记忆只放 3 类内容

### 3.1 你的长期偏好

例如：

- 我希望默认用中文交流
- 我喜欢先结论后细节
- 默认不要啰嗦
- 如果是代码问题，优先给我可执行方案

### 3.2 你的长期工作环境信息

例如：

- 我的主要项目目录在 `D:\code`
- 我常处理的是源码分析、配置整理、文档编写
- 我希望输出文档时优先中文

### 3.3 长期有效的边界和习惯

例如：

- 做文件修改前先核对，不要直接覆盖我的现有文档
- 危险操作先提示我
- 输出文档时尽量结构清晰，适合小白阅读

## 不建议一开始就放进去的内容

- 你全部人生背景
- 大量历史聊天记录
- 过于细碎的临时任务
- 会很快过期的短期安排
- 敏感 secrets

## 一个初始 `MEMORY.md` 的推荐样子

```md
# Long-Term Memory

## User Preferences
- 默认使用中文沟通。
- 回答时优先给结论，再给必要细节。
- 文档尽量写得适合小白理解。

## Work Style
- 用户主要使用 OpenClaw 做源码分析、文档整理和 agent 调教。
- 做修改前先核对，不要直接覆盖现有文档。

## Stable Environment Notes
- 主要工作目录在 `D:\code`。
```

这已经足够了。  
一开始宁可少，也不要乱。

## 4. 第一次安装后，其他关键 `.md` 文件应该怎么组织

这是最值得你一开始理清的部分。

## 4.1 `AGENTS.md`：工作规则文件

这是最重要的文件之一。  
它适合写“Agent 以后做事的规则”。

适合放：

- 做事顺序
- 风险边界
- 何时先确认你
- 记忆写入原则
- 文档输出习惯
- 代码修改习惯

### 推荐初始结构

```md
# AGENTS.md

## Working Rules
- 先理解任务，再动手。
- 修改文件前，先说明将修改什么。
- 不要覆盖用户已有文档，除非明确要求。
- 对危险操作先确认。

## Documentation Rules
- 输出文档优先中文。
- 对小白说明时，先讲结论，再讲细节。
- 优先给出可操作步骤。

## Memory Rules
- 长期偏好写入 `MEMORY.md`。
- 当天决定和临时上下文写入 `memory/YYYY-MM-DD.md`。
- 写入记忆后，优先告诉用户写到了哪个文件。
```

## 4.2 `SOUL.md`：风格和人格文件

这不是工作规则，而是“你想让它怎么说话”。

适合放：

- 简洁还是详细
- 直接还是温和
- 是否避免套话
- 是否更像务实工程师

### 推荐初始结构

```md
# SOUL.md

- 说话直接、清楚、少废话。
- 不要使用夸张鼓励和客套开场。
- 优先像一个靠谱工程助手，而不是营销型聊天机器人。
- 需要详细时再展开，不默认长篇大论。
```

## 4.3 `USER.md`：用户画像文件

这个文件不是让你写规则，而是写“你是谁”。

适合放：

- 你的名字或称呼
- 你偏好的称呼方式
- 时区
- 你常做的工作
- 你的沟通偏好

### 推荐初始结构

```md
# USER.md

- Name: 你的名字或昵称
- Preferred language: 中文
- Timezone: Asia/Shanghai
- Main use cases:
  - 源码分析
  - 文档整理
  - 配置管理
  - 调教 Agent
```

## 4.4 `TOOLS.md`：环境说明文件

这个文件非常容易被忽略，但实际很有用。

适合放：

- 你的目录约定
- 常用机器/盘符/路径别名
- 常见命令约定
- 本地环境特征
- 项目目录说明

### 推荐初始结构

```md
# TOOLS.md

## Paths
- 主要项目目录：`D:\code`
- 当前 OpenClaw 相关目录：`D:\code\anzhuang`

## Notes
- 文档产出默认放在当前工作目录。
- 如果有多个已有文档，新增优先新建文件，不覆盖旧文件。
```

## 4.5 `HEARTBEAT.md`：可选

小白刚开始不用急着配。  
等你真的需要定期检查任务、提醒、轮询时再加。

## 4.6 `MEMORY.md` 和 `memory/`

初始只放少量关键内容即可。  
先养成结构，再慢慢积累。

## 5. 初始配置文件 `openclaw.json` 应该怎么配

小白第一次配置，不要追求大而全。  
建议采用“最小可用配置”。

## 推荐最小配置思路

### 5.1 明确 workspace

```json5
{
  agents: {
    defaults: {
      workspace: "~/.openclaw/workspace"
    }
  }
}
```

### 5.2 如果接入外部聊天渠道，先配最基本安全策略

比如：

- `dmPolicy: "pairing"`
- 或 allowlist

不要一开始就开完全开放。

### 5.3 暂时不要急着配复杂 sandbox / cron / hooks

第一次安装后的建议是：

- sandbox：先不折腾复杂隔离
- cron：先不用
- hooks：先不用
- 多 agent：先不用

### 5.4 Skills 保持默认即可，后面按需加

官方技能系统很强，但第一次安装时别急着开太多。  
最好的策略是：

- 先用默认 bundled skills
- 后续按需求加
- 不要一次性装 20 个你根本用不到的 skill

## 6. 自带的工具、Skills，要一开始就配吗

答案是：**不用全配，按场景来。**

## 6.1 Tools 的原则

OpenClaw 的工具很多，但你不需要一开始把所有能力都当成“必学项”。

第一次上手，先把它当成这几类工具：

- 读写文件
- 搜索和检索
- 基本命令行执行
- memory 工具
- 可能的 web / browser 能力

对小白来说，先掌握这件事就够了：

**Tools 是 Agent 干活的手，不是你一开始就要全部手动配置的东西。**

你真正需要做的是：

- 在 `AGENTS.md` 里写清楚这些手怎么用
- 在 `TOOLS.md` 里写清楚你的环境特点

## 6.2 Skills 的原则

小白第一次安装后的技能策略建议是：

### 第一阶段：先不用自己写 skill

只先观察：

- 这个 Agent 哪些任务老是做不好
- 哪些任务你总是重复教它

### 第二阶段：重复 3 次以上的流程，再技能化

例如：

- 固定格式的源码分析
- 固定格式的日报/周报
- 固定的网站查询流程
- 固定的发布前检查流程

### 第三阶段：技能要少而精

技能应该：

- 名字明确
- description 明确
- 只做一件事

不要一开始就造很多大而泛的技能。

## 7. 第一次安装后，推荐你怎么开始使用

## 日常使用的推荐起点：先用 Control UI 聊天

小白最好的起点不是先接 5 个渠道，而是：

- 先在 dashboard / control UI 里用
- 先调试出一个顺手的主 Agent
- 再考虑接 Telegram/Discord/WhatsApp 等

这样做的好处：

- 排查问题更简单
- 少了渠道噪音
- 可以先把工作流养顺

## 推荐的前 7 天使用方式

### 第 1 天：只做基础对话和文件调教

任务目标：

- 让 Agent 读懂你的 workspace 文件
- 根据你的反馈改 `AGENTS.md`、`SOUL.md`、`USER.md`、`TOOLS.md`

你可以这样做：

- 让它解释它现在理解到的工作规则
- 让它根据你的偏好更新 `SOUL.md`
- 让它把你对文档风格的要求写进 `AGENTS.md`

### 第 2-3 天：开始让它做小任务

例如：

- 总结文件
- 分析源码目录
- 比较文档差异
- 写一份新文档
- 帮你整理清单

这时重点观察：

- 它哪些地方总是理解不稳定
- 哪些规则你重复说了不止一次

### 第 4-5 天：开始沉淀 memory

当你发现某些偏好和上下文已经稳定时：

- 写入 `MEMORY.md`
- 当天事项写到 `memory/YYYY-MM-DD.md`

### 第 6-7 天：开始考虑 skill 化

如果某类任务你已经反复教了几次，就准备做 skill。

## 8. 日常怎么用，才会越用越顺手

## 建议你采用“三层指令法”

以后和 Agent 协作时，把需求分成三层：

### 第一层：这次要干什么

例如：

- “写一份源码结构说明”
- “分析这个目录”
- “帮我整理配置问题”

### 第二层：以后都怎么干

例如：

- “以后遇到这类文档，先给结论，再分模块讲”
- “以后不要改原文档，新建一份”

这种内容应该落到 `AGENTS.md`

### 第三层：什么需要记住

例如：

- “这个风格偏好记到长期记忆”
- “今天这个决定记到 short-term memory”

这样你和 Agent 的协作会很稳定。

## 推荐你常用的几类话术

### 调规则

- “把这条规则写进 `AGENTS.md`。”
- “把这个风格要求更新到 `SOUL.md`。”
- “把这个环境说明写进 `TOOLS.md`。”

### 调记忆

- “把这个长期偏好写进 `MEMORY.md`。”
- “把今天这个决定记到当天 memory 文件里。”
- “写完之后回读给我确认。”

### 调技能

- “这个流程我已经重复让你做三次了，帮我整理成 skill 草案。”

## 9. 定期怎么迭代更新，才能越来越像“能干活的 Agent”

这是最关键的长期部分。

## 推荐节奏：每周一次小迭代

### 每周做这 5 件事

#### 1. 清理和更新 `AGENTS.md`

问自己：

- 最近我有没有重复提醒过它什么规则？
- 有没有新形成的工作习惯？

把真正稳定的规则补进去。

#### 2. 清理 `MEMORY.md`

问自己：

- 哪些长期记忆已经过时？
- 哪些 daily memory 应该提炼进长期记忆？

#### 3. 看最近的 `memory/*.md`

问自己：

- 有没有重要事项没有沉淀？
- 有没有纯流水信息可以忽略？

#### 4. 检查技能是否要新增

问自己：

- 我有没有某个任务已经教了太多次？
- 它是否值得变成 skill？

#### 5. 检查 config 是否还是“够用但不臃肿”

问自己：

- 有没有开了却不用的功能？
- 有没有该加上的安全限制？

## 每月做一次“大整理”

建议做：

- 备份 workspace
- 检查 `openclaw.json`
- 更新 OpenClaw 版本
- 看 `CHANGELOG`
- 看哪些 skills 已经没价值
- 看是否要拆分成多个 agent

## 10. 什么时候考虑升级成更复杂的配置

只有当你满足下面这些条件，再考虑进入下一阶段：

### 升级到更多 skills

当你已经明确发现：

- 某类任务高频
- 靠聊天反复教很低效

### 升级到多个 agent

当你已经明确发现：

- 一个 agent 同时承担多种完全不同的角色
- 私聊、群聊、工作、研究混在一起很乱

### 升级到 subagents

当你已经明确发现：

- 有些任务很慢
- 有些任务适合并行
- 主 Agent 被阻塞明显

### 升级到 hooks / automation

当你已经明确发现：

- 你有稳定重复的自动化需求
- 单纯靠手工聊天已经不够了

## 11. 一个适合小白的“第一版配置与文件策略”

如果你让我给一个最保守、最实用的初始方案，我建议是：

## 初始文件策略

- `AGENTS.md`：只写 5-10 条关键工作规则
- `SOUL.md`：只写 3-5 条风格要求
- `USER.md`：只写最基本的用户画像
- `TOOLS.md`：只写核心环境说明
- `MEMORY.md`：只写少量高价值长期记忆
- `memory/`：从今天开始按天记录

## 初始配置策略

- 一个主 agent
- 一个 workspace
- 先使用 Control UI
- 先不开复杂 sandbox / hooks / cron
- 渠道先接一个最需要的
- DM 策略保持安全默认
- skills 先不大量新增

## 初始使用策略

- 先用 1 周观察
- 不急着技能化
- 重复的规则先写 `AGENTS.md`
- 重要偏好先写 `MEMORY.md`
- 日常变化先写 `memory/YYYY-MM-DD.md`

## 12. 你可以直接照着执行的路线图

## Day 0：装好当天

1. 完成 `openclaw onboard`
2. 确认 `openclaw dashboard` 能打开
3. 运行 `openclaw setup`
4. 打开 workspace 看一遍关键文件
5. 补齐最基础的 `AGENTS.md` / `SOUL.md` / `USER.md` / `TOOLS.md`
6. 新建一个简短的 `MEMORY.md`

## Day 1-3：试运行期

1. 先在 Control UI 聊天
2. 让它做小任务
3. 把你重复说过 2 次以上的规则写进 `AGENTS.md`
4. 把稳定偏好写进 `MEMORY.md`
5. 把当天进展写进 `memory/当天日期.md`

## Day 4-7：稳定期

1. 看哪些任务最常见
2. 看哪些错误最常重复
3. 改 `AGENTS.md`、`TOOLS.md`
4. 开始考虑少量 skill 化

## 每周：小迭代

1. 清理长期记忆
2. 提炼 daily memory
3. 补规则
4. 只在必要时新增 skill

## 每月：大维护

1. 备份 workspace
2. 更新 OpenClaw
3. 看 changelog
4. 看是否需要多 agent / subagents / automation

## 13. 最后的建议

如果你现在刚装好 OpenClaw，我建议你别把自己当成“系统管理员”，而是当成“在培养一个数字助理的人”。

培养顺序应该是：

1. 先让它知道你是谁
2. 再让它知道你想怎么做事
3. 再让它记住重要偏好
4. 再让它学会重复流程
5. 最后才进入复杂自动化

所以真正的最佳实践不是“第一次就配置满”，而是：

**先最小可用，后逐步沉淀；先稳定规则，后扩展能力；先养一个好用的主 Agent，再做复杂体系。**

如果你后面愿意，我可以继续直接给你补三份可落地模板：

- 一份适合你的 `AGENTS.md` 初稿
- 一份适合你的 `SOUL.md` 初稿
- 一份适合你的 `MEMORY.md` 初稿
